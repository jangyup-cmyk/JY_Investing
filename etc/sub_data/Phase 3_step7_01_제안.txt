좋습니다. 지금 단계에서는 **설계 검토**보다 **실제로 깨지는 지점을 빨리 드러내는 통합 테스트 초안**이 더 중요합니다.

흐름 기준은 말씀하신 그대로 잡으면 됩니다.

`persist_message() → preprocess_worker.run_once() → inference_worker.run_once()`

아래처럼 정리하면 바로 테스트 코드로 옮기기 좋습니다.

---

# 1. 통합 테스트의 목적

이 테스트에서 확인해야 할 핵심은 4가지입니다.

1. **메시지 저장이 되면 후속 job 이 정확히 생성되는가**
2. **PreprocessWorker가 NORMALIZE job 을 정확히 소비하는가**
3. **InferenceWorker가 SINGLE_MESSAGE_SUMMARY job 을 정확히 소비하는가**
4. **각 단계가 끝난 뒤 DB 상태가 기대한 형태로 남는가**

즉, 단순히 “에러 없이 돈다”가 아니라
**메시지/잡/결과 테이블의 최종 상태가 설계와 일치하는지**를 검증해야 합니다.

---

# 2. 가장 먼저 잡아야 할 통합 테스트 시나리오 1

## 시나리오 1: 정상 플로우 end-to-end

가장 기본이 되는 happy path 입니다.

### 입력

* 새 텔레그램 메시지 1건 저장
* `persist_message()` 호출
* 이어서 `preprocess_worker.run_once()`
* 이어서 `inference_worker.run_once()`

### 기대 결과

최종적으로 아래가 성립해야 합니다.

* 원문 메시지가 저장되어 있다.
* message version 또는 raw payload 가 저장되어 있다.
* `NORMALIZE` job 이 생성되었다가 `DONE` 처리된다.
* preprocess 결과(정규화 텍스트, 메타 등)가 저장된다.
* `SINGLE_MESSAGE_SUMMARY` job 이 생성되었다가 `DONE` 처리된다.
* inference 결과(summary/result record)가 저장된다.
* 동일 message_id 기준으로 중복 job 이 생기지 않는다.

---

## 권장 검증 포인트

테스트에서는 최소한 아래를 assert 하는 것이 좋습니다.

### 메시지 관련

* `messages` 테이블에 해당 telegram message 존재
* channel_id / external_message_id / text 원문 일치
* 버전 테이블이 있으면 version row 1건 존재

### job 관련

* `NORMALIZE` job 상태가 `DONE`
* `SINGLE_MESSAGE_SUMMARY` job 상태가 `DONE`
* 각 job 의 `attempt_count`, `started_at`, `finished_at` 값이 정상 기록됨

### preprocess 결과 관련

* normalized_text 가 비어 있지 않음
* preprocess output row 가 정확히 1건
* input message 와 FK 연결 정상

### inference 결과 관련

* summary/result row 가 정확히 1건
* summary 텍스트가 비어 있지 않음
* 해당 결과가 원본 message 와 연결됨

# 3. 이 시나리오의 pytest 초안 구조

아래는 **구조 초안**입니다.
필드명은 실제 프로젝트 모델명에 맞게 치환하면 됩니다.

```python
def test_run_once_end_to_end_happy_path(
    db_session_factory,
    message_service,
    preprocess_worker,
    inference_worker,
):
    # 1) 입력 메시지 저장
    persisted = message_service.persist_message(
        channel_id=1,
        telegram_message_id=1001,
        text="삼성전자 관련 공시 속보",
        sent_at=datetime(2026, 3, 22, 10, 0, 0),
        raw_payload={"id": 1001, "text": "삼성전자 관련 공시 속보"},
    )

    # 2) 저장 직후 상태 확인
    with db_session_factory() as session:
        msg = session.query(Message).filter_by(id=persisted.id).one()
        assert msg.text == "삼성전자 관련 공시 속보"

        normalize_jobs = (
            session.query(AnalysisJob)
            .filter_by(message_id=msg.id, job_type="NORMALIZE")
            .all()
        )
        assert len(normalize_jobs) == 1
        assert normalize_jobs[0].status in ("PENDING", "QUEUED")

    # 3) preprocess 1회 실행
    processed_pre = preprocess_worker.run_once()
    assert processed_pre == 1

    # 4) preprocess 이후 상태 확인
    with db_session_factory() as session:
        normalize_job = (
            session.query(AnalysisJob)
            .filter_by(message_id=persisted.id, job_type="NORMALIZE")
            .one()
        )
        assert normalize_job.status == "DONE"
        assert normalize_job.finished_at is not None

        preprocessed = (
            session.query(PreprocessedMessage)
            .filter_by(message_id=persisted.id)
            .one()
        )
        assert preprocessed.normalized_text
        assert len(preprocessed.normalized_text) > 0

        summary_jobs = (
            session.query(AnalysisJob)
            .filter_by(message_id=persisted.id, job_type="SINGLE_MESSAGE_SUMMARY")
            .all()
        )
        assert len(summary_jobs) == 1
        assert summary_jobs[0].status in ("PENDING", "QUEUED")

    # 5) inference 1회 실행
    processed_inf = inference_worker.run_once()
    assert processed_inf == 1

    # 6) 최종 상태 확인
    with db_session_factory() as session:
        summary_job = (
            session.query(AnalysisJob)
            .filter_by(message_id=persisted.id, job_type="SINGLE_MESSAGE_SUMMARY")
            .one()
        )
        assert summary_job.status == "DONE"
        assert summary_job.finished_at is not None

        result = (
            session.query(MessageSummary)
            .filter_by(message_id=persisted.id)
            .one()
        )
        assert result.summary_text
        assert len(result.summary_text) > 0
```

두 번째는 단순 정상 플로우보다 더 실전적인 것을 잡아야 합니다.

## 시나리오 2: inference 실패 후 재시도 가능 상태 검증

이 테스트가 중요한 이유는 실제 운영에서 제일 자주 깨지는 곳이
**외부 모델 호출 / 파싱 / 저장 직전 예외**이기 때문입니다.

### 흐름

1. 메시지 저장
2. preprocess 는 정상 처리
3. inference 단계에서 의도적으로 예외 발생
4. job 상태가 `FAILED` 또는 `RETRY_PENDING` 으로 남는지 확인
5. attempt_count 증가 확인
6. 재실행 가능하도록 stuck/failure 상태가 설계대로 남는지 확인

## 기대 결과

* `NORMALIZE` job 은 `DONE`
* `SINGLE_MESSAGE_SUMMARY` job 은 실패 상태
* error_message / last_error 기록
* attempt_count 1 증가
* 결과 테이블에는 summary row 가 생기지 않음
* 다음 복구 루프나 scheduler 가 재처리할 수 있는 상태 유지

## 이 테스트의 핵심 포인트

이 시나리오는 보통 **inference client mock** 으로 처리합니다.

예:

* 첫 호출 시 `RuntimeError("mock llm timeout")`
* 두 번째 호출 시 정상 응답

이렇게 하면 나중에 아래까지 확장 가능합니다.

* 첫 run_once() → 실패
* scheduler_worker.requeue_stuck_jobs() 또는 retry 로직 수행
* 두 번째 run_once() → 성공
* 최종적으로 결과 row 1건만 존재

## pytest 초안

```python
def test_inference_run_once_failure_sets_retryable_state(
    db_session_factory,
    message_service,
    preprocess_worker,
    inference_worker,
    mocker,
):
    mocker.patch(
        "src.workers.inference_worker.generate_summary",
        side_effect=RuntimeError("mock llm timeout"),
    )

    persisted = message_service.persist_message(
        channel_id=1,
        telegram_message_id=1002,
        text="현대차 신규 투자 관련 기사",
        sent_at=datetime(2026, 3, 22, 10, 5, 0),
        raw_payload={"id": 1002, "text": "현대차 신규 투자 관련 기사"},
    )

    assert preprocess_worker.run_once() == 1

    processed = inference_worker.run_once()
    assert processed == 0  # 또는 구현에 따라 1, but failed
                         # 이 부분은 run_once 계약을 명확히 정의해야 함

    with db_session_factory() as session:
        job = (
            session.query(AnalysisJob)
            .filter_by(message_id=persisted.id, job_type="SINGLE_MESSAGE_SUMMARY")
            .one()
        )

        assert job.status in ("FAILED", "RETRY_PENDING")
        assert job.attempt_count == 1
        assert job.last_error is not None
        assert "mock llm timeout" in job.last_error

        result_count = (
            session.query(MessageSummary)
            .filter_by(message_id=persisted.id)
            .count()
        )
        assert result_count == 0
```

# 5. 여기서 반드시 먼저 고정해야 할 run_once() 계약

통합 테스트를 쓰기 전에 `run_once()` 의 반환 계약을 먼저 정해야 테스트가 흔들리지 않습니다.

권장 방식은 아래 셋 중 하나입니다.

## 권장안 A

`run_once()` 는 **성공적으로 DONE 처리한 job 수**를 반환

* 처리 성공 시 `1`
* 처리할 job 없으면 `0`
* job 잡았지만 실패하면 `0`

이 방식이 가장 직관적입니다.

---

## 권장안 B

상세 결과 객체 반환

예:

```python
RunOnceResult(
    claimed=1,
    succeeded=0,
    failed=1,
    skipped=0,
)
```

운영성은 가장 좋지만 초반엔 조금 무겁습니다.

---

## 비추천안

성공/실패/없음을 모두 bool 하나로 처리

예:

* `True`: 뭔가 했다
* `False`: 안 했다 또는 실패했다

이 경우 테스트에서 **“job이 없어서 False”** 와 **“실패해서 False”** 가 구분되지 않아 운영 로그와 테스트 모두 애매해집니다.

---

# 6. worker / run loop 연결 시 실제로 자주 터지는 오류

이제 가장 중요한 부분입니다.
실전에서는 테스트보다 **run loop 연결부**에서 많이 깨집니다.

아래 항목들이 핵심 방어 포인트입니다.

---

## 6-1. 같은 job 을 두 worker 가 동시에 집는 문제

### 발생 상황

* worker A, B 가 거의 동시에 polling
* 둘 다 같은 `PENDING` job 을 조회
* 둘 다 처리 시작

### 방어 포인트

* 단순 select 후 update 하지 말고
  **claim 단계 자체를 원자적으로 처리**
* 예: `SELECT ... FOR UPDATE SKIP LOCKED`
* 혹은 `UPDATE ... WHERE status='PENDING' ... RETURNING id`

즉, `claim_next_job()` 이 반드시 **트랜잭션 단위 원자 연산**이어야 합니다.

---

## 6-2. 긴 트랜잭션으로 DB 세션이 꼬이는 문제

### 발생 상황

* job claim
* 같은 세션으로 외부 inference 호출
* 호출이 오래 걸림
* connection 점유, row lock 유지, timeout 발생

### 방어 포인트

claim 과 실제 작업을 분리해야 합니다.

권장 흐름:

1. 짧은 세션으로 job claim + `RUNNING` 상태 반영 후 commit
2. 세션 종료
3. 외부 작업 수행
4. 새 세션 열어 결과 저장 + `DONE/FAILED` 처리

즉, **외부 호출 중 DB 트랜잭션을 절대 오래 물고 있지 않기**가 핵심입니다.

---

## 6-3. 부분 저장 후 예외 발생으로 상태가 꼬이는 문제

### 발생 상황

* preprocess 결과는 저장됨
* 그런데 summary job 생성 직전에 예외
* 결과적으로 preprocess 는 끝났지만 다음 job 이 없음

### 방어 포인트

한 단계의 “의미상 원자적 작업”은 한 트랜잭션으로 묶어야 합니다.

예를 들어 preprocess 단계에서는 아래를 같이 묶는 것이 좋습니다.

* preprocess output 저장
* 현재 NORMALIZE job 을 DONE 처리
* 다음 SINGLE_MESSAGE_SUMMARY job 생성

이 셋은 **같은 commit** 으로 끝나는 것이 안전합니다.

---

## 6-4. 중복 job 생성 문제

### 발생 상황

* `persist_message()` 가 같은 메시지를 두 번 받음
* 재시도 로직이나 네트워크 이슈로 동일 메시지 재삽입
* NORMALIZE job 이 중복 생성됨

### 방어 포인트

두 겹으로 막는 것이 좋습니다.

1. **DB unique constraint**

   * 예: `(channel_id, telegram_message_id)` unique
   * job 테이블도 `(message_id, job_type)` unique 검토

2. **서비스 레이어 idempotency**

   * 이미 메시지가 있으면 새로 만들지 않음
   * 이미 후속 job 이 있으면 재생성하지 않음

실전에서는 **애플리케이션 체크만으로는 부족**합니다.
최종 방어선은 unique constraint 여야 합니다.

---

## 6-5. poison message 문제

### 발생 상황

* 특정 메시지 하나가 파싱 실패를 계속 일으킴
* worker 가 그 job 만 계속 잡고 재시도
* 전체 큐가 사실상 막힘

### 방어 포인트

* `attempt_count`
* `max_attempts`
* 초과 시 `DEAD` 또는 `PERMANENT_FAILED`
* scheduler 는 이 상태는 재큐잉하지 않음
* 운영자가 별도 점검

즉, **영원히 재시도하는 구조는 반드시 피해야 합니다.**

---

## 6-6. stuck job 복구가 과하게 동작하는 문제

### 발생 상황

* 정상적으로 오래 걸리는 inference 작업도
* scheduler 가 “stuck” 으로 오판하고 재큐잉

### 방어 포인트

job type 별 timeout 을 다르게 둬야 합니다.

예:

* `NORMALIZE`: 1~3분
* `SINGLE_MESSAGE_SUMMARY`: 5~15분

그리고 `started_at` 기준뿐 아니라
가능하면 `heartbeat_at` 이 있으면 더 좋습니다.

---

## 6-7. run loop 가 빈 큐에서 과도하게 polling 하는 문제

### 발생 상황

* 처리할 job 이 없는데 while True 로 계속 DB 조회
* CPU/DB 낭비

### 방어 포인트

* `run_once()` 가 0 반환 시 짧게 sleep
* 연속 empty count 누적 시 sleep 증가(backoff)
* 로그도 매 루프마다 남기지 말고 주기적으로만 남기기

예:

* empty 1~5회: 1초
* empty 6~20회: 3초
* 그 이상: 5초

---

## 6-8. 프로세스 종료 시 RUNNING job 이 남는 문제

### 발생 상황

* 배포/재시작/SIGTERM 도중 worker 종료
* job 은 RUNNING 상태로 고정

### 방어 포인트

* 종료 시 graceful shutdown
* 다만 완전 보장은 어렵기 때문에
  최종 복구는 `SchedulerWorker` 가 담당
* 따라서 RUNNING job 은 항상 `started_at` 기준 복구 가능해야 함

즉, graceful shutdown 은 “좋으면 좋은 것”이고
**진짜 복구 책임은 scheduler 설계가 져야 합니다.**

---

# 7. 지금 당장 테스트 코드에 같이 넣어두면 좋은 보조 assert

통합 테스트는 결과만 보면 부족하고,
아래 항목을 같이 보면 나중에 디버깅이 쉬워집니다.

## 공통 assert 권장

* `created_at <= started_at <= finished_at`
* `attempt_count` 초기값/증가값 검증
* `last_error` 는 성공 시 비어 있어야 함
* 결과 row count 는 항상 1건 이하인지 확인
* 다음 job 생성 시 부모 message_id 연결 정상 확인

## 있으면 좋은 것

* worker 가 처리한 job id 반환
* 로그에 message_id / job_id 포함
* test failure 시 어떤 단계에서 멈췄는지 바로 보이게 상태 조회 helper 제공

# 8. 가장 실무적인 권장 순서

지금 바로 진행 순서는 이렇게 가는 것이 가장 자연스럽습니다.

### 1단계

먼저 **시나리오 1 happy path 통합 테스트** 작성

### 2단계

그 다음 **시나리오 2 inference 실패/재시도 테스트** 작성

### 3단계

이 두 테스트를 통과시키기 위해

* `run_once()` 반환 계약 고정
* `claim_next_job()` 원자화
* job 상태 전이 정리

### 4단계

마지막으로 scheduler 테스트 2~3개 추가

* stuck RUNNING → requeue
* max_attempts 초과 → dead
* 정상 RUNNING 은 재큐잉 금지
