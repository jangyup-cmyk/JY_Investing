좋습니다. 제안하신 방향은 **현재 문서 구조와 가장 잘 맞는 리팩토링 축**입니다.

현재 프로젝트는 이미 `src/services`, `src/workers`, `run_worker.py` 확장 지점을 갖고 있고, `run_collector.py`는 실시간 수집, `run_worker.py`는 기타 스케줄링/분석용 복합 워커 진입점으로 정의돼 있습니다. 또 Phase 3가 전처리, Phase 4가 LLM 분석으로 이어지므로, `AnalysisJobService` 도입 → `PreprocessWorker` 추가 → `InferenceWorker` 표준화 → `run_worker.py` 단일 진입점화는 로드맵과 정확히 맞물립니다.  

제가 보기엔 세 가지 모두 타당하지만, **그대로 밀어붙이면 좋은 점**과 **조심할 점**이 분명합니다.

첫째, `AnalysisJobService`를 `src/services`로 올리는 것은 맞습니다. 현재 문서에서 API는 `Depends(get_db)`로 세션 생명주기를 안정화했고, 워커 쪽은 `startup_tasks`가 `persist_func`를 주입받도록 설계해 결합도를 낮췄습니다. 이런 맥락에서는 job enqueue, claim, heartbeat, done/fail 같은 상태 전이 로직도 서비스 레이어로 모으는 것이 계층 일관성에 맞습니다. 특히 지금 프로젝트가 배치 커밋, savepoint, resume, 편집 메시지 반영 같은 정합성 규칙을 중요하게 다루고 있으므로, 이 규칙을 워커마다 복붙하는 것보다 서비스 한곳에서 관리하는 편이 훨씬 안전합니다.  

둘째, `InferenceWorker`를 `claim_next_job` 방식으로 리팩토링하는 것도 맞습니다. 현재 구조는 Telethon 이벤트가 들어오면 곧바로 DB에 upsert 하는 수집 중심 구조이고, 다음 단계는 전처리와 LLM 분석으로 확장될 예정입니다. 그러면 수집 워커와 분석 워커는 반드시 분리된 상태 머신을 가져야 하고, `analysis_jobs` 기반의 claim/lease/retry 패턴이 그 경계가 됩니다. 특히 현재 Phase 2가 “워커 재시작 후 이어받기”, “중복 적재 0건”, “편집 메시지 실시간 반영”을 검증 항목으로 이미 통과한 상태라서, 분석 워커도 같은 수준의 idempotency와 retry semantics를 가져야 일관성이 맞습니다.  

셋째, `PreprocessWorker`를 별도로 두는 것도 매우 좋습니다. 문서상 Phase 3 목표가 “원문 메시지의 노이즈 제거와 티커/키워드 추출”이고, Phase 4가 “LLM 기반 요약과 테마 연관성 분석”이기 때문에, 전처리를 inference 안에 끼워 넣는 것보다 워커를 분리하는 편이 훨씬 낫습니다. 그래야 전처리 규칙만 바뀌었을 때 재실행 범위를 명확히 할 수 있고, LLM 비용이 들기 전 단계에서 최대한 메시지를 정리할 수 있습니다. 이건 토큰 비용과 운영 안정성 측면에서도 유리합니다.  

다만, 세 번째인 `run_worker.py` 단일 실행 구조는 **좋지만 선을 잘 그어야 합니다.** `run_worker.py`를 워커 타입 선택형 엔트리포인트로 만드는 것 자체는 README 구조와 맞습니다. 하지만 여기서 `collector`까지 흡수해서 “모든 워커를 하나의 실행기”로 합치는 것은 권하지 않습니다. 문서가 현재 구조를 `run_web.py`와 `run_collector.py`의 역할 분리 원칙 위에서 설명하고 있고, `run_collector.py`는 Telethon 실시간 리스너와 백필이라는 별도 장애 특성을 가지기 때문입니다. 따라서 `run_worker.py`는 **preprocess / inference / scheduler 계열**만 다루고, `run_collector.py`는 계속 분리하는 편이 좋습니다.  

제 추천 구조는 이렇습니다.

`run_collector.py`

* 텔레그램 수집 전용
* startup backfill + realtime listener
* 메시지 적재 + normalize job enqueue까지만 담당

`run_worker.py --type preprocess`

* `analysis_jobs`에서 `normalize`, `extract_features` 처리

`run_worker.py --type inference`

* `single_message_summary`, `window_theme_analysis` 처리

`run_worker.py --type scheduler`

* stuck job recovery, dead-letter 점검, window aggregation 같은 보조 작업

이렇게 가면 현재 문서의 역할 분리 원칙을 해치지 않으면서도, 실행 UX는 단순해집니다.  

추가로, 현재 DB가 `aiosqlite`인 점을 감안하면 `claim_next_job` 방식은 맞지만 **처음에는 preprocess/inference 각각 1 worker 프로세스**로 시작하는 것이 좋습니다. 문서상 현재 강점은 savepoint, batch commit, resume 정합성이고, 아직 Phase 3/4는 이제 붙는 단계입니다. 이 시점에서 과도한 병렬 claim 경쟁까지 동시에 여는 것보다, 우선 직렬에 가까운 안전한 처리로 먼저 데이터 계약을 굳히는 편이 낫습니다.  

실행 인자 설계는 아래 정도가 가장 무난합니다.

```bash
python run_worker.py --type preprocess
python run_worker.py --type inference
python run_worker.py --type scheduler
```

그리고 옵션은 최소한

* `--poll-interval`
* `--worker-name`
* `--job-types`
  정도만 두면 됩니다. 너무 많은 실행 모드를 한 번에 넣으면 오히려 운영 복잡성이 커집니다.

정리하면, 사용자가 제안한 세 가지 방향은 전부 적절합니다. 다만 구조적으로는 다음처럼 가져가면 가장 안정적입니다.

`AnalysisJobService`는 바로 도입해도 좋습니다.
`PreprocessWorker`는 `InferenceWorker`보다 먼저 붙이는 것이 맞습니다.
`InferenceWorker`는 claim/lease/retry 기반으로 리팩토링하는 것이 맞습니다.
`run_worker.py`는 분석 계열 워커의 공용 엔트리포인트로 쓰되, `run_collector.py`까지 흡수하지는 않는 편이 좋습니다.

실행 순서까지 포함한 추천 우선순위는 이렇습니다.

1. `AnalysisJobService` 추가
2. `PreprocessWorker` 구현
3. `run_worker.py --type preprocess` 연결
4. 기존 `InferenceWorker`를 `analysis_jobs` 소비형으로 리팩토링
5. `run_worker.py --type inference` 연결
6. 마지막에 scheduler/stuck-job recovery 추가

이 순서가 현재 Phase 2의 안정성을 가장 덜 건드리면서 Phase 3로 넘어가는 최적 경로입니다.  

다음으로는 이 기준에 맞춰 **`run_worker.py` CLI 설계안과 폴더별 파일 책임 분담표**를 바로 잡는 것이 가장 자연스럽습니다.

좋습니다. 그럼 바로 **`run_worker.py` CLI 설계안**과 **파일별 책임 분담표**, 그리고 **적용 순서**까지 현재 구조 기준으로 정리하겠습니다.

문서 기준으로 지금 실행 체계는 `run_web.py`와 `run_collector.py`가 역할 분리된 2-프로세스 구조이고, `run_worker.py`는 “기타 스케줄링/분석 등 복합 워커 구동 진입점”으로 이미 예약되어 있습니다. 또 `collector_worker.py`와 `startup_tasks.py`가 수집/백필 책임을 가지고 있고, 다음 단계가 Phase 3 전처리, Phase 4 LLM 분석으로 명시되어 있으므로, `run_worker.py`를 **분석 계열 워커의 공용 진입점**으로 쓰는 방향이 가장 자연스럽습니다.   

## 1) `run_worker.py` CLI 설계안

제 추천은 단순합니다.
`run_worker.py`는 collector를 흡수하지 말고, **preprocess / inference / scheduler** 세 계열만 담당하게 두는 편이 맞습니다. 현재 문서가 `run_collector.py`를 별도 구동 진입점으로 두고 있고, Telethon 실시간 리스너와 백필은 장애 특성과 운영 방식이 다르기 때문입니다.  

권장 실행 형태:

```bash
python run_worker.py --type preprocess
python run_worker.py --type inference
python run_worker.py --type scheduler
```

추가 옵션은 최소한 아래 정도면 충분합니다.

```bash
python run_worker.py --type preprocess --worker-name preprocess-1 --poll-interval 2.0
python run_worker.py --type inference --worker-name inference-1 --poll-interval 3.0
python run_worker.py --type scheduler --worker-name scheduler-1 --poll-interval 10.0
```

그리고 나중에 확장 여지를 남기려면 `--job-types` 옵션도 받을 수 있습니다.

```bash
python run_worker.py --type inference --job-types single_message_summary window_theme_analysis
```

### 권장 argparse 골격

```python
import argparse
import asyncio

from src.workers.preprocess_worker import PreprocessWorker
from src.workers.inference_worker import InferenceWorker
from src.workers.scheduler_worker import SchedulerWorker


def build_parser() -> argparse.ArgumentParser:
    parser = argparse.ArgumentParser(description="Background worker runner")
    parser.add_argument(
        "--type",
        required=True,
        choices=["preprocess", "inference", "scheduler"],
        help="Worker type to run",
    )
    parser.add_argument(
        "--worker-name",
        default=None,
        help="Logical worker name",
    )
    parser.add_argument(
        "--poll-interval",
        type=float,
        default=2.0,
        help="Polling interval seconds",
    )
    parser.add_argument(
        "--job-types",
        nargs="*",
        default=None,
        help="Optional job type filter",
    )
    return parser


async def async_main(args):
    worker_name = args.worker_name or f"{args.type}-worker-1"

    if args.type == "preprocess":
        worker = PreprocessWorker(
            worker_name=worker_name,
            poll_interval=args.poll_interval,
        )
    elif args.type == "inference":
        worker = InferenceWorker(
            worker_name=worker_name,
            poll_interval=args.poll_interval,
            job_types=args.job_types,
        )
    elif args.type == "scheduler":
        worker = SchedulerWorker(
            worker_name=worker_name,
            poll_interval=args.poll_interval,
        )
    else:
        raise ValueError(f"Unsupported worker type: {args.type}")

    await worker.run_forever()


def main():
    parser = build_parser()
    args = parser.parse_args()
    asyncio.run(async_main(args))


if __name__ == "__main__":
    main()
```

이 정도면 지금 구조에 무리 없이 붙고, 나중에 worker 종류가 늘어나도 엔트리포인트를 다시 갈아엎지 않아도 됩니다.

## 2) 파일별 책임 분담표

지금 단계에서는 역할 경계를 분명히 잡는 것이 중요합니다. 문서상 이미 `collector_worker.py`와 `startup_tasks.py`가 수집/백필을 담당하고, `src/services`를 둘 수 있는 구조가 있으므로 아래처럼 나누는 것이 가장 깔끔합니다.  

### `run_collector.py`

역할:

* Telethon 수집 전용 프로세스 시작
* 실시간 리스너 등록
* startup backfill orchestration 호출
* 메시지 적재 + normalize job enqueue까지만 담당

여기에는 LLM 분석 로직을 넣지 않는 편이 좋습니다.

### `run_worker.py`

역할:

* preprocess / inference / scheduler 워커 실행기
* 워커 타입 선택
* 공통 CLI 옵션 처리
* collector는 제외

### `src/services/analysis_job_service.py`

역할:

* `enqueue_job`
* `claim_next_job`
* `heartbeat`
* `mark_done`
* `mark_failed`
* stuck lease recovery용 보조 메서드

즉, **작업 상태 전이 로직의 단일 진실 공급원**이 됩니다.

### `src/workers/preprocess_worker.py`

역할:

* `normalize`
* `extract_features`
* `normalized_messages`, `message_features` 생성
* 다음 단계 job enqueue

### `src/workers/inference_worker.py`

역할:

* `single_message_summary`
* `window_theme_analysis`
* 향후 `agent_debate`, `consensus`까지 확장 가능
* 외부 LLM 호출 전후 세션 분리

### `src/workers/scheduler_worker.py`

역할:

* stuck job 복구
* dead_letter 재처리 정책
* window aggregation 트리거
* 향후 cron성 maintenance 작업

### `src/workers/collector_worker.py`

역할:

* 신규/편집 메시지 수신
* 공통 persist 함수 제공
* `message upsert → version 생성 → normalize job enqueue`
* 실시간/백필 경로 공통 영속화 지점 유지

### `src/workers/startup_tasks.py`

역할:

* 백필 오케스트레이션
* `persist_func` 주입 기반 orchestration 유지
* collector 내부 구현 세부사항을 몰라도 되게 유지

현재 문서에서 `startup_tasks`가 collector와 persistence 함수를 인자로 받아 실행되도록 설계했다고 명시돼 있으므로, 이 DI 방향은 그대로 유지하는 편이 맞습니다.

## 4) 워커 공통 인터페이스 권장안

향후 유지보수를 생각하면 세 워커 모두 공통 형태를 맞추는 게 좋습니다.

```python
class BaseWorker:
    def __init__(self, worker_name: str, poll_interval: float = 2.0):
        self.worker_name = worker_name
        self.poll_interval = poll_interval
        self._running = True

    async def run_forever(self) -> None:
        raise NotImplementedError
```

그리고 각 워커는 최소한 아래 흐름을 공유하면 됩니다.

* job claim
* 입력 로드
* 처리
* 결과 저장
* 다음 job enqueue
* done/fail 마크

이렇게 맞춰두면 `run_worker.py`도 단순해지고, 테스트도 쉬워집니다.

## 5) `InferenceWorker` 리팩토링 방향

사용자 제안대로 `InferenceWorker`를 `claim_next_job` 중심으로 바꾸는 것은 맞습니다. 다만 LLM 호출이 붙는 순간부터는 아래 규칙을 반드시 지키는 것이 좋습니다.

1. job claim은 짧은 세션에서 처리
2. 세션 종료
3. normalized/features 로드용 짧은 세션
4. 세션 종료
5. LLM 호출
6. 새 세션 열어 결과 저장
7. `mark_done` 또는 `mark_failed`

현재 문서가 API 세션 생명주기 분리를 중시하고 있고, 워커 확장성 검토를 요청하고 있는 만큼, inference 쪽에서도 “세션 오래 물고 있지 않기”가 중요합니다.  

---

## 6) scheduler worker는 왜 분리하는가

이건 나중으로 미뤄도 되지만, 장기적으로는 분리하는 편이 좋습니다.

이유는:

* preprocess/inference는 처리량 중심
* scheduler는 관리 작업 중심

예를 들면 scheduler는 이런 일을 맡으면 됩니다.

* `RUNNING`인데 `lease_expires_at` 지난 job을 `PENDING`으로 복구
* `DEAD_LETTER` job 통계 집계
* 5분/15분 단위 `window_theme_analysis` job 생성
* 오래된 `normalized_messages` 또는 캐시 정리

이걸 inference worker에 섞으면 역할이 흐려집니다.

## 7) 적용 순서

지금은 한 번에 다 만들기보다 아래 순서가 가장 안전합니다.

### 1단계

`AnalysisJobService` 추가
→ 중복된 enqueue/claim/done/fail 로직 통합

### 2단계

`PreprocessWorker` 추가
→ `run_worker.py --type preprocess` 연결

### 3단계

`collector_worker.py` 공통 persist 함수에서 normalize job enqueue 연결
→ 실시간/백필 모두 동일 정책 적용

### 4단계

기존 `InferenceWorker`를 `analysis_jobs` 소비형으로 리팩토링
→ `run_worker.py --type inference` 연결

### 5단계

`scheduler_worker.py` 추가
→ stuck job 회복, window aggregation, maintenance 분리

이 순서가 현재 Phase 2의 안정성을 가장 덜 건드리고, Phase 3 전처리로 가장 부드럽게 넘어갑니다. 현재 문서도 백필/실시간 수집은 이미 안정화됐고, 다음 단계가 전처리와 LLM 분석이라고 분명히 적고 있습니다.

## 8) 최종 판단

제안 중에서 가장 좋은 부분은 두 가지입니다.

하나는 **`AnalysisJobService`를 중심으로 작업 상태 전이를 단일화하려는 점**이고,
다른 하나는 **`run_worker.py`를 분석 계열 워커의 공용 진입점으로 만들려는 점**입니다.

다만 선은 분명히 그어야 합니다.
`run_worker.py`는 **collector까지 통합하는 만능 실행기**가 아니라,
**preprocess / inference / scheduler 계열의 공용 실행기**여야 합니다.
`run_collector.py`는 계속 분리하는 편이 현재 구조와 더 잘 맞습니다.
