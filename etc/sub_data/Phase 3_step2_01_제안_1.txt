좋습니다. 이번에는 **실제 패치에 바로 옮기기 쉬운 diff 스타일 초안**으로 적겠습니다.

현재 구조상 `collector_worker.py`는 실시간 신규/편집 메시지를 DB에 적재하는 저장 루프를 담당하고, `startup_tasks.py`는 `persist_func`를 주입받아 백필을 오케스트레이션하도록 설계돼 있습니다. 또 Phase 2의 핵심 강점은 `begin_nested()` savepoint, 100건 단위 batch commit, cursor-based resume, 편집 메시지 반영 정합성입니다. 따라서 이번 패치는 **이벤트 핸들러가 아니라 공통 persist 지점**에 넣어야 하고, **enqueue는 기존 트랜잭션 안**에 태워야 합니다.   

아래 diff는 실제 코드가 없어서 **함수명은 예시**입니다. 다만 패치 위치와 책임 분리는 현재 문서 구조와 맞춘 형태입니다.

## 1) `collector_worker.py` 패치 핵심

핵심 원칙은 네 가지입니다.

* 신규/편집/백필이 **공통으로 타는 persist 함수** 하나를 기준으로 패치
* `message upsert → version 생성 → normalize job enqueue`를 **같은 세션/같은 savepoint**에서 처리
* `AnalysisJobService.enqueue_job()`은 **commit 하지 않음**
* idempotency key는 반드시 **`message_id` + `version_id` + version tag** 기준으로 구성

이렇게 해야 현재 백필 엔진의 batch commit/savepoint/resume 구조를 유지할 수 있습니다.

### D. `create_message_version()` 추가

편집 메시지 반영을 위해 `version` 계보를 남겨야 합니다. Phase 2 검증 항목에도 원본 편집 메시지 실시간 반영이 포함돼 있습니다. 

```diff
diff --git a/src/workers/collector_worker.py b/src/workers/collector_worker.py
index 4444444..5555555 100644
--- a/src/workers/collector_worker.py
+++ b/src/workers/collector_worker.py
@@
+async def create_message_version(session: AsyncSession, message: Message, tg_msg) -> MessageVersion:
+    current = await session.scalar(
+        select(MessageVersion).where(
+            MessageVersion.message_id == message.id,
+            MessageVersion.is_current.is_(True),
+        )
+    )
+
+    new_text = tg_msg.message or ""
+    new_hash = _sha256(new_text)
+    new_edit_date = getattr(tg_msg, "edit_date", None)
+
+    # 실질 변화 없으면 기존 current 재사용
+    if current and current.raw_hash == new_hash:
+        return current
+
+    if current:
+        current.is_current = False
+        next_version_no = current.version_no + 1
+    else:
+        next_version_no = 1
+
+    version = MessageVersion(
+        message_id=message.id,
+        version_no=next_version_no,
+        message_text=new_text,
+        raw_hash=new_hash,
+        edit_date_utc=new_edit_date,
+        is_current=True,
+    )
+    session.add(version)
+    await session.flush()
+    return version
+
+
+def _sha256(value: str) -> str:
+    return sha256(value.encode("utf-8")).hexdigest()
```

## 4) `AnalysisJobService`에서 꼭 지켜야 할 조건

이건 매우 중요합니다.

### `enqueue_job()` 내부에서 commit 금지

현재 백필 엔진은 100건 단위 batch commit을 통해 FloodWait나 중간 실패에도 이전 작업분을 보존하도록 설계돼 있습니다. `enqueue_job()`가 내부에서 commit을 해버리면 이 구조가 깨집니다.  

즉, 아래처럼 flush까지만 해야 합니다.

```python
job = AnalysisJob(...)
session.add(job)
await session.flush()
return job
```

### 중복이면 기존 job 반환

resume/replay/backfill 재실행을 안전하게 처리하려면 필요합니다.

```python
existing = await session.scalar(
    select(AnalysisJob).where(
        AnalysisJob.job_type == job_type,
        AnalysisJob.idempotency_key == idempotency_key,
    )
)
if existing:
    return existing
```

이건 큰 테스트는 아니지만, 단일 진입점 구조가 깨지지 않았는지 빠르게 검증하는 용도로 좋습니다.

```python
def test_run_worker_parser_preprocess():
    parser = build_parser()
    args = parser.parse_args(["--type", "preprocess"])
    assert args.type == "preprocess"

def test_run_worker_parser_inference():
    parser = build_parser()
    args = parser.parse_args(["--type", "inference", "--job-types", "single_message_summary"])
    assert args.type == "inference"
    assert args.job_types == ["single_message_summary"]
```

## 7) 가장 먼저 깨질 가능성이 높은 부분

이번 패치에서 가장 위험한 부분은 두 개입니다.

하나는 **`enqueue_job()` 안에서 commit이 들어가는 경우**입니다. 이러면 현재 문서의 batch commit/savepoint 설계가 무너질 가능성이 큽니다. 

다른 하나는 **persist 지점이 아니라 이벤트 핸들러에 enqueue를 넣는 경우**입니다. 그러면 백필과 실시간 처리 정책이 달라지고, `startup_tasks.py`가 왜 `persist_func`를 주입받는지의 설계 의미가 사라집니다.

## 8) 최종 권장 순서

실행 순서는 이렇게 가는 것이 가장 안전합니다.

1. `collector_worker.py` 공통 persist 함수 패치
2. `AnalysisJobService` idempotent enqueue 확정
3. `tests/services/test_analysis_job_service.py` 추가
4. `tests/workers/test_collector_persist.py` 추가
5. `PreprocessWorker` 연결 테스트 추가
6. 마지막에 `InferenceWorker` 연결

이 순서가 현재 Phase 2의 강점인 안정성과 정합성을 보존하면서 Phase 3로 넘어가는 가장 낮은 리스크 경로입니다. 문서상 현재는 수집 인프라가 이미 안정적이고, 다음 단계가 전처리와 분석으로 명확히 정리돼 있습니다.
