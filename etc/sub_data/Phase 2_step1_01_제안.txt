다만, STEP 1을 더 단단하게 하려면 아래처럼 잡는 것이 좋습니다.

## 왜 STEP 1부터 가야 하는가

STEP 2 백필, STEP 3 실시간 리스너는 결국 둘 다 같은 질문으로 수렴합니다.

* 이 메시지를 어떤 형태로 저장할 것인가
* 이미 저장된 메시지면 어떻게 갱신할 것인가
* 어느 채널이 수집 대상인가
* 마지막 체크포인트는 어디까지인가

이 기준이 먼저 없으면
백필은 백필대로 저장하고, 실시간은 실시간대로 저장해서
나중에 중복 처리, edited 반영, checkpoint 갱신 기준이 갈라집니다.

즉 STEP 1은 단순 선행작업이 아니라 **Phase 2의 저장 규약 자체**입니다.

# STEP 1 상세 설계안

## 1. 핵심 원칙

STEP 1에서는 아래 4개를 먼저 고정해야 합니다.

### A. 저장 경로 단일화

백필이든 실시간이든 무조건 같은 함수로 저장

예:

* `upsert_message(...)`
* `update_channel_checkpoint(...)`

### B. 상태 변경 주체 분리

* API: 상태만 바꿈
* Worker: 상태를 읽고 실제 수집/백필 수행

이 원칙이 중요합니다.
특히 API 서버와 수집 워커가 분리되어 있다면, **토글 API가 직접 Telethon을 호출해서 백필까지 해버리면 구조가 꼬입니다.**

### C. 채널 상태와 동기화 상태를 혼동하지 않기

이 부분이 가장 중요합니다.

채널은 사실 두 종류의 상태가 있습니다.

1. **동기화 상태**

   * discovered
   * synced
   * missing
   * inaccessible

2. **수집 상태**

   * active
   * paused
   * error

이걸 하나의 status 컬럼에 다 넣으면 나중에 헷갈립니다.
이미 상태 전용 테이블이 있다면, 최소한 내부적으로는 두 개념을 분리해서 다루는 것이 좋습니다.

### D. 체크포인트 갱신 규칙 고정

`last_collected_message_id`는 아무 때나 바꾸면 안 됩니다.

권장:

* 메시지 저장 성공 후 갱신
* 더 큰 message_id일 때만 갱신
* 백필/실시간 모두 동일 규칙 적용

## 2. STEP 1의 실제 구현 대상

### 2-1. 공통 메시지 저장 함수

가장 먼저 만들어야 할 함수입니다.

권장 책임:

* 메시지 DTO 입력
* `(telegram_channel_id, tg_message_id)` 기준 조회
* 없으면 insert
* 있으면 update
* `edited_at`, `is_deleted`, `views`, `forwards`, `raw_payload_json` 반영
* 저장 성공 여부 반환

권장 함수 느낌:

```python
def upsert_message(db: Session, payload: TelegramMessagePayload) -> UpsertResult:
    ...
```

반환값 예시:

* `created`
* `updated`
* `unchanged`
* `failed`

이렇게 두면 백필/실시간 로직에서 후속 처리하기 쉽습니다.

---

### 2-2. Telethon 메시지 → 내부 DTO 변환기

저장 함수가 Telethon 객체를 직접 받지 않게 하는 것이 좋습니다.

즉 중간에 변환 계층을 둡니다.

예:

```python
@dataclass
class TelegramMessagePayload:
    telegram_channel_id: int
    tg_message_id: int
    sender_id: int | None
    message_text_raw: str | None
    reply_to_message_id: int | None
    views: int | None
    forwards: int | None
    grouped_id: int | None
    edited_at: datetime | None
    is_deleted: bool
    message_date: datetime
    collected_at: datetime
    raw_payload_json: str
```

이렇게 하면 나중에 Telethon이 바뀌거나 테스트 코드를 만들 때도 편합니다.

---

### 2-3. 채널 동기화 API 고도화

`POST /sync`는 단순히 채널을 가져오는 API가 아니라 **DB 상태를 정규화하는 작업**이어야 합니다.

권장 동작:

1. 현재 참여 중인 채널 목록 조회
2. 신규 채널 insert
3. 기존 채널 title/username/type update
4. 상태 테이블 없으면 생성
5. 현재 참여 중이면 `sync_status = synced`
6. 이전엔 있었지만 지금 없으면 `sync_status = missing`

중요한 점:

* sync는 **수집 ON/OFF를 바꾸는 API가 아님**
* sync는 **참여 채널 정보를 최신화하는 API**

즉 `sync`와 `toggle`은 역할이 완전히 달라야 합니다.

---

### 2-4. 수집 토글 API

이 API는 단순해야 합니다.

예:
`POST /api/telegram/channels/{id}/toggle`

body 예시:

```json
{
  "enabled": true
}
```

권장 동작:

* 상태 테이블에서 `collection_status = active` 또는 `paused` 변경
* `last_error_message`는 필요 시 유지
* API는 여기서 끝
* 실제 백필은 워커가 상태 변화를 보고 수행

여기서 핵심은
**토글 ON = 즉시 백필 실행**이 아니라
**토글 ON = 백필 대상 상태로 전환**
으로 보는 것입니다.

더 정확히 하려면 상태를 이렇게 둘 수도 있습니다.

* `ACTIVE`
* `PAUSED`
* `PENDING_BACKFILL`
* `ERROR`

이 구조면 워커가 `PENDING_BACKFILL` 채널을 발견해서 먼저 백필하고, 성공 시 `ACTIVE`로 바꾸는 흐름이 자연스럽습니다.

이건 꽤 추천합니다.

## 3. STEP 1에서 꼭 넣어야 하는 세부 규칙

### 메시지 Upsert 규칙

* 유니크 키: `(telegram_channel_id, tg_message_id)`
* 신규 메시지면 insert
* 기존 메시지면 update
* `edited_at`이 더 최신이면 본문/메타데이터 갱신
* `raw_payload_json`은 최신값으로 덮어쓰기
* 삭제 이벤트가 오면 `is_deleted = True`

### 체크포인트 규칙

* 저장 성공한 메시지 기준으로만 갱신
* 현재 값보다 큰 `tg_message_id`일 때만 갱신
* 실패한 메시지는 체크포인트 갱신 금지

### 상태 전이 규칙

추천 흐름:

* `discovered`
* `synced`
* `pending_backfill`
* `active`
* `paused`
* `error`

주의:

* `error`가 나더라도 채널 자체를 disabled 처리하지 말고 원인 추적 가능하게 유지

## 4. STEP 1 구현 순서

이 순서가 가장 안정적입니다.

### 1단계

`TelegramMessagePayload` 같은 내부 DTO 정의

### 2단계

`telethon message -> payload` 변환 함수 구현

### 3단계

`upsert_message()` 구현

### 4단계

`update_channel_checkpoint()` 구현

### 5단계

`POST /sync` 고도화

### 6단계

`POST /toggle` 구현

즉, API보다 먼저 **저장 규약**을 만드는 것이 맞습니다.
API는 그 다음입니다.

---

## 5. STEP 1 완료 기준

이 단계는 아래가 되면 끝입니다.

1. 텔레그램 메시지 하나를 내부 payload로 변환할 수 있다.
2. 같은 메시지를 두 번 넣어도 DB에 1건만 유지된다.
3. 편집 메시지를 넣으면 기존 레코드가 갱신된다.
4. 채널 sync 시 채널/상태 테이블이 정상 갱신된다.
5. 채널 toggle 시 수집 상태가 정확히 바뀐다.
6. 저장 성공 후 체크포인트가 갱신된다.

## 6. 제가 추가로 제안하는 작은 보강

### A. `PENDING_BACKFILL` 상태를 넣는 것

이건 꽤 유용합니다.

이유:

* 토글 ON 직후 바로 ACTIVE로 두면 “백필 전인데 활성 상태”가 됩니다.
* 그래서 워커가 먼저 백필을 끝내고 ACTIVE로 바꾸는 게 더 깔끔합니다.

### B. `upsert_message()`는 DB 모델 직접 반환보다 결과 객체 반환

예:

* `created=True`
* `updated=False`
* `checkpoint_candidate=12345`

이런 식이면 워커가 후속 처리하기 쉽습니다.

### C. 저장 실패 로그와 채널 상태 갱신 분리

메시지 하나 실패했다고 채널을 곧바로 `ERROR`로 바꾸지 말고,
연속 실패나 연결 실패 같은 경우만 채널 상태를 `ERROR`로 두는 편이 좋습니다.
