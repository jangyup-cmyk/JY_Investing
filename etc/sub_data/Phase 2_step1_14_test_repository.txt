지금은 **먼저 `test_repository.py`를 repository 계약에 맞게 미세 조정하는 쪽**이 더 맞습니다.

이유는 단순합니다.
`repository.py`의 실패 계약이 바뀌었기 때문입니다.

예전 계약:

* 알 수 없는 예외도 내부에서 잡음
* `UpsertResult(action="failed")` 반환

지금 계약:

* 알 수 없는 예외는 **raise**
* `IntegrityError`만 특수 처리
* `failed`는 사실상 “일반 실패 결과”가 아니라 예외 흐름으로 이동

그래서 먼저 해야 할 일은 **테스트가 옛 계약을 붙잡고 있는지 정리하는 것**입니다.

## 지금 기준으로 가장 먼저 깨질 테스트

가장 먼저 깨질 가능성이 높은 건 이전에 추가했던 이 계열입니다.

### 1) 실패 시 `action == "failed"` 기대 테스트

이건 이제 repository 테스트에 두면 안 맞습니다.

기존 의도:

* repository가 실패를 결과 객체로 반환한다

현재 실제 계약:

* repository는 일반 예외를 **raise** 한다

즉 이 테스트는 이렇게 바꿔야 합니다.

```python
@pytest.mark.asyncio
async def test_upsert_raises_on_unexpected_db_error(db_session, monkeypatch):
    payload = TelegramMessagePayload(
        telegram_channel_id=1,
        tg_message_id=999,
        sender_id=None,
        message_text_raw="Should Fail",
        reply_to_message_id=None,
        views=None,
        forwards=None,
        grouped_id=None,
        edited_at=None,
        is_deleted=False,
        posted_at=datetime.now(timezone.utc),
        collected_at=datetime.now(timezone.utc),
        raw_payload_json='{"text":"Should Fail"}',
    )

    async def broken_execute(*args, **kwargs):
        raise RuntimeError("forced db failure")

    monkeypatch.setattr(db_session, "execute", broken_execute)

    with pytest.raises(RuntimeError, match="forced db failure"):
        await upsert_telegram_message(db_session, payload)
```

즉 repository 테스트는 이제
**“failed 반환”** 이 아니라
**“예외를 제대로 올린다”** 를 검증해야 합니다.

---

## 그대로 유지해도 되는 테스트

### 2) `created / unchanged / updated` 테스트

이건 그대로 살아남을 가능성이 높습니다.

* 신규 저장
* 동일 메시지 재입력
* edited_at 반영
* is_deleted 반영

이 부분은 지금 repository 구조와 여전히 잘 맞습니다.

---

## 지금 구현 덕분에 살아날 가능성이 커진 테스트

### 3) 메타데이터 보강 테스트

이건 이전엔 실패 가능성이 높았지만, 지금은 통과 가능성이 높아졌습니다.

즉 이 테스트는 남겨두는 게 좋습니다.

* 최초 저장: `sender_id=None`, `reply_to_message_id=None`, `grouped_id=None`
* 재입력: 값 채워짐
* 기대: `updated`

이건 지금 repository의 개선점을 가장 잘 보여주는 테스트입니다.

---

## repository 테스트에서 빼야 더 맞는 테스트

### 4) “실패 시 checkpoint 미갱신” 테스트

이건 **repository 단독 테스트보다 worker/wrapper 테스트로 옮기는 게 더 정확합니다.**

왜냐하면 지금 checkpoint는 repository가 아니라

* `persist_message_payload()`
* worker wrapper
  에서 올리기 때문입니다.

즉 올바른 질문은 이제 이겁니다.

* repository가 예외를 raise 했을 때
* worker가 rollback 하고
* checkpoint를 건드리지 않는가

이건 `test_repository.py`보다
`test_collector_worker.py`나 wrapper 테스트가 더 맞습니다.

# 제가 권하는 실제 순서

## 1단계

`test_repository.py`에서 실패 계약 관련 테스트를 수정

* `action == "failed"` 기대 제거
* `pytest.raises(...)`로 변경

## 2단계

메타데이터 보강 테스트 유지

* 이건 현재 개선의 핵심 검증 포인트

## 3단계

checkpoint 미갱신 테스트는 worker 테스트로 이동 검토

# 한 줄로 정리

지금은
**“어떤 테스트가 깨질지 먼저 보는 것”보다, repository의 새 계약을 기준으로 `test_repository.py`를 먼저 맞추는 것**이 더 좋습니다.

가장 먼저 바꿔야 하는 테스트는 이것입니다.

* **`failed` 결과를 기대하던 테스트 → `raises` 테스트로 변경**

그 다음에 돌려보면, 실제로 남는 실패는 거의 다 **진짜 구현 문제**일 가능성이 높습니다.

좋습니다. 현재 `repository.py` 계약에 맞추면 `test_repository.py`는 아래처럼 정리하는 게 가장 깔끔합니다.

핵심 변경점은 두 가지입니다.

* **일반 예외는 `action="failed"`를 기대하지 않고 `raise`를 기대**
* **`updated` 결과뿐 아니라 실제 DB row 값까지 검증**

아래는 **통째로 교체 가능한 버전**입니다.

```python
import pytest
import pytest_asyncio
from datetime import datetime, timezone, timedelta

from sqlalchemy import select
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker

from src.infra.db.models import (
    Base,
    TelegramChannel,
    TelegramMessage,
    TelegramChannelCollectionState,
)
from src.infra.telegram.schemas import TelegramMessagePayload
from src.infra.db.repository import upsert_telegram_message, update_channel_checkpoint


@pytest_asyncio.fixture
async def db_session():
    engine = create_async_engine("sqlite+aiosqlite:///:memory:", echo=False)
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)

    Session = async_sessionmaker(engine, expire_on_commit=False)

    async with Session() as session:
        # 테스트용 임의 채널 생성
        ch = TelegramChannel(
            telegram_account_id=1,
            tg_channel_id="1001",
            title="Test Channel",
            username="test",
            channel_type="CHANNEL",
            status="SYNCED",
            is_collecting=True,
            last_synced_at=datetime.now(timezone.utc),
        )
        session.add(ch)
        await session.commit()
        yield session

    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.drop_all)


@pytest.mark.asyncio
async def test_upsert_new_message_and_unchanged(db_session):
    payload = TelegramMessagePayload(
        telegram_channel_id=1,
        tg_message_id=10,
        sender_id=None,
        message_text_raw="First Msg",
        reply_to_message_id=None,
        views=100,
        forwards=0,
        grouped_id=None,
        edited_at=None,
        is_deleted=False,
        posted_at=datetime.now(timezone.utc),
        collected_at=datetime.now(timezone.utc),
        raw_payload_json={"test": 1},
    )

    result1 = await upsert_telegram_message(db_session, payload)
    assert result1.action == "created"
    assert result1.checkpoint_candidate == 10
    await db_session.commit()

    result2 = await upsert_telegram_message(db_session, payload)
    assert result2.action == "unchanged"
    await db_session.commit()

    rows = (
        await db_session.execute(
            select(TelegramMessage).where(
                TelegramMessage.telegram_channel_id == 1,
                TelegramMessage.tg_message_id == 10,
            )
        )
    ).scalars().all()

    assert len(rows) == 1
    assert rows[0].message_text_raw == "First Msg"
    assert rows[0].views == 100
    assert rows[0].forwards == 0


@pytest.mark.asyncio
async def test_upsert_edited_message_persists_updated_fields(db_session):
```

추가로 한 가지 짚으면, 현재 이 테스트 파일은 `raw_payload_json`을 **dict**로 넣고 있습니다.
지금 `repository.py`는 이걸 그대로 받아 저장하므로 테스트와 맞습니다. 다만 나중에 `translators.py`를 고쳐서 `raw_payload_json`을 진짜 JSON 문자열로 통일하면, 이 테스트도 문자열 기준으로 같이 바꾸는 게 좋습니다.

다음으로는 이 테스트 파일 기준으로 **지금 당장 가장 먼저 실패할 가능성이 있는 테스트가 무엇인지** 점검하는 게 자연스럽습니다.
