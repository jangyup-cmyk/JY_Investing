3) 5개 이상 AI 에이전트의 권장 역할 구조

이 프로젝트는 “에이전트 수”보다 역할이 겹치지 않도록 설계하는 것이 중요합니다.
추천 구조는 7개입니다.

### 1. News/Event Agent
* 텔레그램 메시지에서 뉴스성 이벤트 추출
* “무슨 일이 발생했는가” 요약

### 2. Theme Discovery Agent
* 이벤트를 산업/정책/공급망/수급 테마로 연결
* 단일 종목 뉴스가 아니라 “확장 가능한 테마”인지 판단

### 3. Credibility Agent
* 출처 신뢰도, 반복 언급 수, 과장 여부 점검
* 루머/광고/낚시성 메시지 필터링

### 4. Market Impact Agent
* 해당 이슈가 실제 시장에서 얼마나 강하게 반응할지 평가
* 단기 급등형인지, 중기 확산형인지 구분

### 5. Trader Agent
* 매매 관점으로 진입 타이밍, 추격 위험, 눌림 가능성 검토
* 실제 주문 가능한 시그널 형태로 변환

### 6. Risk Agent
* 변동성, 중복 테마, 포지션 과밀, 손절 기준 평가
* “사도 되는가”보다 “얼마나 위험한가”를 담당

### 7. Debate Moderator / Final Judge
* 각 에이전트 의견 충돌 시 토론 구조를 정리
* 최종 점수와 결론 생성


4) 멀티 에이전트 토론 흐름

### 1단계: 개별 분석
각 에이전트가 같은 원문 메시지를 독립적으로 해석

### 2단계: 주장 제출
각자 아래 형식으로 결과 제출

* 핵심 주장
* 근거
* 신뢰도 점수
* 반론 포인트

### 3단계: 상호 반박
* Theme Agent ↔ Credibility Agent
* Market Impact Agent ↔ Risk Agent
* Trader Agent ↔ Risk Agent

### 4단계: 중재
Moderator가 중복 주장 정리, 약한 주장 제거

### 5단계: 최종 판정
Final Judge가 아래 결과 생성

* 테마명
* 관련 종목 후보
* 신뢰도 점수
* 매매 가능 여부
* 이유
* 보류/관찰/즉시진입 분류

## 1) 화면의 역할

이 화면은 단순 관리페이지가 아니라 아래 4가지를 동시에 처리해야 합니다.

즉, 이 화면의 이름은 그냥 “채널관리”보다 **“텔레그램 수집 관리”** 또는 **“수집 채널 관리”**가 더 맞습니다.

* 동기화 완료 후 시스템은 로그를 남기고 화면에 결과를 표시해야 한다.
* 로그 예시는 다음과 같다.

# 1) 먼저, 3개 문서가 각각 무엇인지

## 1. WEB 화면용 상세 기능명세서

이건 “웹 화면에서 무엇이 보여야 하고, 버튼을 누르면 무슨 일이 일어나야 하는지”를 문서로 적는 것입니다.

예를 들어 당신이 말한:

* 텔레그램 채널 업데이트 버튼
* 내 텔레그램 채널 리스트 가져오기
* 로그 기록 및 표시

이런 요구사항을 그냥 아이디어 수준으로 두지 않고,
**버튼 이름 / 입력값 / 처리로직 / 성공 시 화면 표시 / 실패 시 메시지**까지 적는 문서가 기능명세서입니다.

즉,
**프론트엔드와 백엔드가 공통으로 보는 “화면 동작 계약서”**라고 보면 됩니다.

---

## 2. 백엔드 API 명세

이건 웹 화면의 버튼과 표가 실제 서버와 어떻게 통신할지 정의하는 문서입니다.

예를 들어 웹에서 **“텔레그램 채널 업데이트”** 버튼을 누르면,
서버에 어떤 주소로 요청을 보내고, 서버가 어떤 형식의 결과를 돌려줄지 정해야 합니다.

예:

* 요청 URL
* 요청 방식 `GET / POST`
* 요청 body
* 응답 JSON 구조
* 성공/실패 코드
* 에러 메시지 형식

즉,
**프론트 화면과 서버 사이의 통신 규칙서**입니다.

* 채널명
* 채널 ID
* 공개/비공개 여부
* 수집 활성 여부
* 마지막 동기화 시각
* 신규 채널인지 여부
* 로그 메시지

이걸 어떤 테이블에, 어떤 컬럼명으로, 어떤 타입으로 저장할지를 먼저 정해야 이후 코드가 꼬이지 않습니다.

즉,
**데이터 저장 구조 설계서**입니다.

# 2) 당신 프로젝트 기준으로 각 문서가 왜 필요한가

이 프로젝트는 일반 웹사이트가 아니라 아래처럼 흐릅니다.

**텔레그램 계정 → 채널 목록 동기화 → 수집 대상 선택 → 메시지 수집 → 정제(B) → AI 분석 → 자동매매**

여기서 A 수집 계층은 생각보다 복잡합니다.
그 이유는 단순히 메시지를 긁어오는 게 아니라, **“어떤 채널을 수집할지 운영자가 통제”**해야 하기 때문입니다.

그래서 A 단계에서만 해도 이미 3가지가 필요합니다.

### 화면

사용자가 무엇을 보고 누를지

### API

눌렀을 때 서버가 무엇을 할지

### DB

그 결과를 어디에 저장할지

# 5) DB 테이블 초안은 어떤 수준까지 정하나

초안은 “완벽한 최종 구조”가 아니라 **지금 단계에서 필요한 핵심 테이블을 먼저 정하는 것**입니다.

당신 프로젝트의 A 수집 계층에서는 최소 아래 테이블이 필요합니다.

---

## 1. `telegram_channels`

채널/그룹 마스터 정보

### 주요 컬럼

* `id`
* `tg_channel_id`
* `title`
* `username`
* `channel_type`
* `is_public`
* `is_collecting`
* `status`
* `relevance_score`
* `last_message_at`
* `last_synced_at`
* `created_at`
* `updated_at`

### 역할

채널 리스트 화면의 기본 데이터

---

## 2. `telegram_channel_sync_logs`

채널 업데이트 실행 이력

### 주요 컬럼

* `id`
* `sync_started_at`
* `sync_finished_at`
* `total_found`
* `new_added`
* `updated_count`
* `inactive_count`
* `status`
* `message`

### 역할

“5개의 신규채널을 추가하였습니다” 같은 로그 이력 저장

---

## 3. `system_logs`

운영 로그

### 주요 컬럼

* `id`
* `module_name`
* `log_level`
* `message`
* `created_at`

### 역할

화면 하단 로그 패널 표시

---

## 4. `telegram_messages`

수집된 원문 메시지 저장

이 테이블이 중요합니다.
왜냐하면 바로 여기서 A와 B가 연결되기 때문입니다.

### 주요 컬럼

* `id`
* `channel_id`
* `tg_message_id`
* `message_text_raw`
* `message_text_normalized`
* `message_hash`
* `posted_at`
* `collected_at`
* `has_media`
* `is_processed`
* `processing_status`
* `created_at`

여기서

* `message_text_raw` 는 A 수집 결과
* `message_text_normalized`, `message_hash`, `is_processed` 는 B 정제 계층으로 넘어가기 위한 준비값

입니다.

즉, 지금 DB 초안을 잡을 때부터 B를 생각해서 컬럼을 일부 넣어두는 것이 좋습니다.

* 가장 최근 성공한 채널 동기화 완료 시각

## 4.4 버튼 기능명세

## 4.4.1 텔레그램 채널 업데이트

### 목적

내 텔레그램 계정의 현재 채널/그룹 목록을 조회하고 DB와 비교하여 신규 추가, 정보 갱신, 상태 변경을 반영한다.

### 입력값

없음

### 처리 흐름

1. 텔레그램 세션 상태 확인
2. 내 계정의 채널/그룹 목록 조회
3. DB 기존 채널 목록 조회
4. 신규 채널 식별 후 저장
5. 기존 채널명, username, 공개여부, 유형 변경사항 반영
6. 기존 존재하나 조회 결과에서 빠진 채널은 `REMOVED` 처리
7. 동기화 로그 저장
8. 화면 요약 카드 및 테이블 갱신
9. 운영 로그 저장

### 성공 메시지 예시

`5개의 신규채널을 추가하였습니다. 현재 정보를 수집하고 있는 채널은 20개입니다.`

### 실패 메시지 예시

* `텔레그램 세션이 만료되어 채널 목록을 불러오지 못했습니다.`
* `채널 동기화 중 일부 채널의 접근 권한이 없어 부분 성공으로 처리되었습니다.`

### 로그 레벨

* 시작: INFO
* 완료: SUCCESS
* 부분 성공: WARNING
* 실패: ERROR

---

## 4.4.2 신규 채널만 추가

### 목적

전체 채널 상태를 다시 계산하지 않고, 새로 확인된 채널만 후보 목록에 추가한다.

### 입력값

없음

### 처리 흐름

1. 텔레그램 세션 상태 확인
2. 현재 채널/그룹 목록 조회
3. DB에 없는 채널만 저장
4. 로그 저장
5. 화면 갱신

### 성공 메시지 예시

`신규 채널 3개를 추가했습니다.`

---

## 4.4.3 선택 채널 수집 시작

### 목적

선택한 채널들을 실제 메시지 수집 대상에 포함한다.

### 입력값

### 처리 흐름

2. 각 채널의 `is_collecting = true`, `status = ACTIVE` 반영
3. 로그 저장
4. 테이블 갱신

### 성공 메시지 예시

`선택한 4개 채널의 수집을 활성화했습니다.`

# 6. DB 테이블 초안

아래는 **초기 운영에 필요한 최소 핵심 테이블**입니다.

---

## 6.1 `telegram_accounts`

### 목적

운영 텔레그램 계정 상태 관리

### 컬럼 초안

```sql
telegram_accounts
- id                    BIGINT PK
- account_name          VARCHAR(100) NOT NULL
- phone_number          VARCHAR(30) NULL
- session_name          VARCHAR(200) NOT NULL
- status                VARCHAR(20) NOT NULL   -- CONNECTED / EXPIRED / DISCONNECTED / ERROR
- last_connected_at     DATETIME NULL
- last_checked_at       DATETIME NULL
- created_at            DATETIME NOT NULL
- updated_at            DATETIME NOT NULL
```

### 비고

지금은 1계정 운영이지만, 다계정 확장 대비용입니다.

---

## 6.2 `telegram_channels`

### 목적

채널/그룹 마스터 정보 저장

### 컬럼 초안

```sql
telegram_channels
- id                    BIGINT PK
- telegram_account_id   BIGINT NOT NULL FK -> telegram_accounts.id
- tg_channel_id         VARCHAR(50) NOT NULL
- access_hash           VARCHAR(100) NULL
- title                 VARCHAR(255) NOT NULL
- username              VARCHAR(255) NULL
- channel_type          VARCHAR(20) NOT NULL   -- CHANNEL / GROUP / SUPERGROUP
- is_public             BOOLEAN NOT NULL DEFAULT FALSE
- is_collecting         BOOLEAN NOT NULL DEFAULT FALSE
- status                VARCHAR(20) NOT NULL   -- ACTIVE / INACTIVE / ERROR / REMOVED
- note                  VARCHAR(500) NULL
- last_message_at       DATETIME NULL
- last_synced_at        DATETIME NULL
- created_at            DATETIME NOT NULL
- updated_at            DATETIME NOT NULL
```

### 제약/인덱스

```sql
UNIQUE (telegram_account_id, tg_channel_id)
INDEX idx_telegram_channels_status (status)
INDEX idx_telegram_channels_collecting (is_collecting)
INDEX idx_telegram_channels_last_synced (last_synced_at)
INDEX idx_telegram_channels_title (title)
INDEX idx_telegram_channels_username (username)
```

---

## 6.3 `telegram_channel_sync_logs`

### 목적

채널 동기화 이력 저장

### 컬럼 초안

```sql
telegram_channel_sync_logs
- id                    BIGINT PK
- telegram_account_id   BIGINT NOT NULL FK -> telegram_accounts.id
- sync_type             VARCHAR(20) NOT NULL   -- FULL / NEW_ONLY
- status                VARCHAR(20) NOT NULL   -- SUCCESS / PARTIAL_SUCCESS / FAILED
- total_found           INT NOT NULL DEFAULT 0
- new_added             INT NOT NULL DEFAULT 0
- updated_count         INT NOT NULL DEFAULT 0
- removed_count         INT NOT NULL DEFAULT 0
- error_count           INT NOT NULL DEFAULT 0
- message               VARCHAR(1000) NULL
- started_at            DATETIME NOT NULL
- finished_at           DATETIME NULL
- created_at            DATETIME NOT NULL
```

### 인덱스

```sql
INDEX idx_sync_logs_status (status)
INDEX idx_sync_logs_started_at (started_at)
```

---

## 6.4 `system_logs`

### 목적

운영 로그 공통 저장

### 컬럼 초안

```sql
system_logs
- id                    BIGINT PK
- module_name           VARCHAR(100) NOT NULL
- log_level             VARCHAR(20) NOT NULL   -- INFO / SUCCESS / WARNING / ERROR
- related_entity_type   VARCHAR(50) NULL       -- CHANNEL / MESSAGE / ACCOUNT / SYNC
- related_entity_id     BIGINT NULL
- message               VARCHAR(2000) NOT NULL
- detail_json           JSON NULL
- created_at            DATETIME NOT NULL
```

### 인덱스

```sql
INDEX idx_system_logs_module (module_name)
INDEX idx_system_logs_level (log_level)
INDEX idx_system_logs_created_at (created_at)
```

## 6.5 `telegram_messages`

### 목적

수집된 원문 메시지 저장
A 수집 계층과 B 정제 계층을 연결하는 핵심 테이블

### 컬럼 초안

```sql
telegram_messages
- id                    BIGINT PK
- telegram_channel_id   BIGINT NOT NULL FK -> telegram_channels.id
- tg_message_id         BIGINT NOT NULL
- sender_name           VARCHAR(255) NULL
- message_text_raw      TEXT NULL
- message_text_normalized TEXT NULL
- message_hash          VARCHAR(128) NULL
- has_media             BOOLEAN NOT NULL DEFAULT FALSE
- media_type            VARCHAR(50) NULL
- posted_at             DATETIME NOT NULL
- collected_at          DATETIME NOT NULL
- is_processed          BOOLEAN NOT NULL DEFAULT FALSE
- processing_status     VARCHAR(20) NOT NULL DEFAULT 'PENDING'  -- PENDING / PROCESSING / DONE / FAILED
- created_at            DATETIME NOT NULL
- updated_at            DATETIME NOT NULL
```

### 제약/인덱스

```sql
UNIQUE (telegram_channel_id, tg_message_id)
INDEX idx_telegram_messages_posted_at (posted_at)
INDEX idx_telegram_messages_collected_at (collected_at)
INDEX idx_telegram_messages_is_processed (is_processed)
INDEX idx_telegram_messages_processing_status (processing_status)
INDEX idx_telegram_messages_hash (message_hash)
```

### 비고

여기서 아래 필드는 B 정제 계층을 염두에 둔 연결용입니다.

* `message_text_normalized`
* `message_hash`
* `is_processed`
* `processing_status`

즉, 지금은 A 문서지만 B 진입을 위해 미리 자리만 잡아둡니다.

## 6.6 `telegram_channel_daily_stats`

- stat_date             DATE NOT NULL
- message_count         INT NOT NULL DEFAULT 0

### 제약/인덱스

# 11. 왜 Phase 1을 먼저 해야 하나

당신이 먼저 Phase 2 요구를 말한 건 자연스러운 일입니다.
왜냐하면 프로젝트의 눈에 보이는 핵심이 텔레그램 수집이기 때문입니다.

하지만 실제 개발은 반대로 가야 합니다.

텔레그램 수집부터 바로 붙이면 아래 문제가 생깁니다.

* 세션 오류가 나도 원인을 모름
* DB 저장이 꼬여도 추적이 안 됨
* 로그가 없어서 장애 분석이 안 됨
* 환경변수와 키 관리가 엉킴
* 웹 화면/API 응답 형식이 일관되지 않음

즉, **Phase 1은 느려 보이지만 전체 개발 속도를 올리는 단계**입니다.

# 12. 당신 프로젝트에 맞는 권장 진행 방식

지금 상태에서는 이렇게 가는 것이 가장 좋습니다.

## 지금 바로 할 일

### 1단계

Phase 1 기준으로

* 폴더 구조
* requirements
* .env
* settings
* logger
* db session
* app 실행 뼈대

를 먼저 만든다.

### 2단계

그다음에 우리가 이미 정리한

* 채널 동기화 화면 명세
* API 명세
* DB 테이블 초안

을 기준으로 Phase 2 코딩에 들어간다.

즉, 지금까지 만든 수집 명세는 버리는 것이 아니라,
**Phase 1 완료 후 바로 Phase 2 구현에 투입할 설계 자산**입니다.
