# 1단계: 루트 + 설정 + 공통 모듈

이 단계 목표는:

* `requirements.txt`
* `.env.example`
* `config/settings.py`
* `src/common/*`

까지 먼저 만드는 것입니다.

## Cursor 프롬프트 1

```text
D:\telegram_trading 프로젝트의 Phase 1 작업을 진행한다.

지금은 1단계만 구현한다.
목표는 루트 기본 파일 + 설정 파일 + 공통 모듈 생성이다.

생성/수정 대상 파일:
- requirements.txt
- .env.example
- config/settings.py
- src/common/logger.py
- src/common/exceptions.py
- src/common/responses.py

환경:
- Windows
- Python 3.11.9 64bit
- .venv
- VS Code

구조 전제:
- 웹 서버와 Collector Worker는 분리 예정
- 하지만 지금은 공통 기반만 먼저 만든다
- 모든 환경변수는 settings 객체를 통해 읽게 할 것
- logger는 웹/워커가 같이 쓸 수 있어야 함
- 응답 형식과 예외 형식은 통일할 것

구현 조건:
1. requirements.txt에는 아래 패키지를 포함
   - fastapi
   - uvicorn[standard]
   - sqlalchemy
   - alembic
   - pydantic-settings
   - python-dotenv
   - psycopg[binary]
   - telethon
   - httpx
   - pytest

2. .env.example에는 아래 범주의 변수 포함
   - APP
   - DB
   - LOG
   - TELEGRAM
   - COLLECTOR
   - BROKER

3. config/settings.py는 아래 구조 권장
   - AppSettings
   - DBSettings
   - LogSettings
   - TelegramSettings
   - CollectorSettings
   - BrokerSettings
   - Settings
   - get_settings() 또는 singleton settings

4. src/common/logger.py
   - 콘솔 + 파일 로그
   - logs 디렉토리 자동 생성
   - app.log / collector.log / error.log
   - get_logger(name, log_type="app") 형태 제공

5. src/common/exceptions.py
   - AppException
   - DatabaseException
   - ExternalAPIException
   - ValidationAppException
   - NotFoundException

6. src/common/responses.py
   - success_response()
   - error_response()

출력 순서:
1. 최종 파일 트리(이번 단계 범위만)
2. 각 파일 역할 1~2줄
3. 각 파일 전체 코드
4. 마지막에 점검 방법 정리

중요:
- 실제 실행 가능한 최소 코드여야 함
- placeholder만 남기지 말 것
- 과설계 금지
- 한국어로 답변
```

```text
D:\telegram_trading 프로젝트의 Phase 1 작업을 이어서 진행한다.

출력 순서:
1. 파일별 역할
```

## Cursor 프롬프트 3

```text
D:\telegram_trading 프로젝트의 Phase 1 작업을 이어서 진행한다.

지금은 3단계만 구현한다.
목표는 최소 실행 가능한 웹 서버 뼈대를 만드는 것이다.

생성/수정 대상 파일:
- src/web/app.py
- src/web/routes/health.py
- run_web.py

이미 존재한다고 가정해야 하는 파일:

- src/common/exceptions.py

구현 조건:

   - 공통 예외 핸들러 등록

2. 파일별 역할
3. 각 파일 전체 코드
4. Windows 기준 실행 명령어
5. /health, /ready 점검 방법

중요:

- 한국어로 답변
```

# 4단계: Collector Worker 실행 뼈대

이 단계 목표는:

## Cursor 프롬프트 4

```text
D:\telegram_trading 프로젝트의 Phase 1 작업을 이어서 진행한다.

지금은 4단계만 구현한다.
목표는 최소 실행 가능한 Collector Worker 뼈대를 만드는 것이다.

생성/수정 대상 파일:
- src/workers/startup_tasks.py
- src/workers/collector_worker.py
- run_collector.py

이미 존재한다고 가정해야 하는 파일:
- config/settings.py
- src/common/logger.py
- src/common/exceptions.py
- src/infra/db/session.py
- src/infra/db/models.py

구현 조건:
1. startup_tasks.py
   - run_startup_tasks() 함수
   - runtime settings 로드용 placeholder 함수
   - startup summary 로그 함수
   - future hooks:
     - channel sync
     - restore last selected channels
     - auto start collection
   - 지금 단계에서는 구조만 실행되게

2. collector_worker.py
   - CollectorWorker 클래스 또는 실행 함수
   - settings 로드
   - logger 로드
   - DB 연결 점검
   - startup tasks 호출
   - 단순 loop 또는 graceful idle 구조
   - 아직 Telethon 실제 수집 로직은 넣지 말 것

3. run_collector.py
   - worker 실행 진입점
   - 시작 로그
   - 종료 로그
   - KeyboardInterrupt 처리

중요:
- python run_collector.py 로 실행 가능한 최소 코드
- placeholder만 두지 말고 실제 시작/점검/로그 흐름은 동작해야 함
- 웹 서버와 공통 settings/logger/db 구조를 공유해야 함

출력 순서:
1. 파일별 역할
2. 각 파일 전체 코드
3. 실행 방법
4. 예상 점검 포인트

한국어로 답변
```

# 최종 정리

지금 가장 실용적인 방법은:

* **파일을 한 번에 다 만들지 말고**
* **공통 → DB → 웹 → 워커 → 전체점검**
  순서로 Cursor에 나눠서 요청하는 것입니다.

이 방식이 import 오류, 설정 누락, 실행 실패를 가장 잘 줄여줍니다.
