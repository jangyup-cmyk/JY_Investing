네. 이 부분은 **공식 문서에서 확인되는 사실**과 **실무적으로 권하는 정책**을 분리해서 가져가면 됩니다.

공식적으로 확인되는 핵심은 세 가지입니다.
첫째, Telegram 서버는 `FLOOD_WAIT_X` 계열 오류로 “X초 기다린 뒤 다시 시도하라”는 신호를 줍니다. 둘째, Telethon v1.42 기준으로는 `flood_sleep_threshold` 이하의 짧은 대기시간은 라이브러리가 자동으로 sleep할 수 있고, 그보다 크면 `FloodWaitError`를 raise합니다. 셋째, Telethon 문서는 `FloodWaitError`를 직접 잡아 `e.seconds`를 보고 처리하는 예시를 제공합니다. ([텔레그램][1])

추가로 버전 차이는 꼭 염두에 두셔야 합니다.
지금 stable 문서 기준(Telethon 1.42) 예시는 `FloodWaitError.seconds`를 쓰지만, v2 문서/마이그레이션 계열에서는 이런 오류값이 `.seconds`가 아니라 공통적으로 `.value`로 바뀌었다고 안내합니다. 그래서 프로젝트가 v1.42 계열이면 `e.seconds`, v2 계열이면 `e.value`를 기준으로 읽는 식으로 분기하거나, 버전을 고정해 두는 편이 안전합니다. ([docs.telethon.dev][2])

그리고 **공식 문서에서 “최대 재시도 횟수는 몇 번”, “몇 초 이상이면 무조건 ERROR 처리” 같은 정책 숫자를 정해주지는 않습니다.** Telegram 쪽 공식 문서는 `FLOOD_WAIT_X`가 오면 X초를 기다려야 한다는 점을 설명하고, Telethon 쪽은 자동 sleep 임계값과 예외 처리 방법을 설명할 뿐입니다. 제가 확인한 공식 자료 범위에서는 “정답 정책 숫자”는 없었습니다. ([텔레그램][1])

그래서 실무 정책은 아래처럼 잡는 것을 권합니다. 이 부분은 제가 문서 내용을 바탕으로 드리는 **구현 가이드**입니다.

**1. 짧은 FloodWait는 Telethon 자동 sleep에 맡기기**
`flood_sleep_threshold`를 예를 들어 `20~30초` 정도로 두면, 아주 짧은 wait는 라이브러리가 알아서 처리합니다. 문서상 threshold 이하에서는 Telethon이 자동 sleep하고, 초과하면 예외를 올립니다. 백필처럼 반복 호출이 많은 코드에서는 이 자동 처리 구간을 너무 낮게 잡지 않는 편이 좋습니다. ([docs.telethon.dev][2])

**2. 긴 FloodWait는 `iter_channel_messages()`에서 직접 재개(resume) 처리**
공식 의미상 `FLOOD_WAIT_X`는 “지금은 너무 빨랐으니 X초 후 다시 해라”이므로, 백필에서는 `last_message_id` 또는 `offset_id/max_id` 기준으로 이어받는 방식이 맞습니다. 이때 sleep 후 처음부터 다시 긁는 것보다, **마지막으로 성공적으로 적재한 메시지 ID 이후부터 재개**하는 쪽이 훨씬 안전합니다. `FloodWait`는 Telegram이 준 정상적인 제어 신호에 가깝고, Telethon도 이런 오류를 별도 클래스로 다루도록 문서화하고 있습니다. ([텔레그램][1])

**3. 재시도 횟수보다 “누적 대기시간 상한”을 두기**
제가 권하는 기준은 “최대 N회”보다 **채널 1개 백필 작업당 누적 대기시간 상한**을 두는 것입니다. 예를 들어:

* `wait <= 60초`: sleep 후 재개
* `60초 < wait <= 300초`: 1~2회까지만 허용
* `wait > 300초`: 즉시 `ERROR` 또는 `PENDING_BACKFILL` 유지 후 다음 사이클로 넘김

이유는 Telegram 공식 신호가 “기다려라”이지 “몇 번까지 재시도해라”는 아니기 때문입니다. 백필 작업은 실시간 이벤트 수집보다 우선순위가 낮으므로, 너무 긴 wait는 워커 전체를 붙잡지 않게 하는 편이 운영상 낫습니다. 이 숫자 자체는 공식값이 아니라 운영 정책값입니다. Telegram/Telethon 문서가 제공하는 건 wait seconds와 자동 sleep threshold뿐입니다. ([텔레그램][1])

**4. 아주 긴 FloodWait는 `ERROR`보다 `PENDING_BACKFILL 재시도 예정`으로 두는 것도 좋음**
이건 운영 철학 차이인데, 제 추천은 **“긴 FloodWait는 실패라기보다 지연”**으로 보는 것입니다. Telegram 공식 의미도 “지금은 기다려라”이지 “잘못된 요청이라 영구 실패”가 아닙니다. 그래서:

* 짧은/중간 wait: sleep 후 resume
* 매우 긴 wait: 이번 실행에서는 중단하고 `last_error_message` 기록, 상태는 `PENDING_BACKFILL` 또는 별도 `RATE_LIMITED`
  처럼 두는 게 더 설명력이 좋습니다. `ERROR`는 권한 문제나 엔티티 손상 같은 진짜 장애와 구분하는 편이 운영상 유리합니다. ([텔레그램][1])

**5. 백필 전용이면 takeout session도 검토할 가치가 있음**
Telethon changelog에는 takeout session(데이터 export 세션)에서 **일부 요청은 flood limit가 더 낮다**고 적혀 있습니다. 이건 “백필에 더 유리하다”는 뜻이 아니라, 오히려 flood 한도가 더 빡빡할 수 있다는 의미로 읽는 게 안전합니다. 그래서 takeout을 쓰더라도 flood wait 방어 로직은 그대로 필요합니다. 백필에서 takeout을 고려하더라도, “FloodWait가 줄어들 것”이라고 기대하고 설계하면 안 됩니다. ([docs.telethon.dev][3])

실무적으로는 아래 정책이 가장 무난합니다.

* Telethon `flood_sleep_threshold = 20~30초`
* `iter_channel_messages()` 내부에서 `FloodWait` catch
* `wait_seconds <= 60`이면 `await sleep(wait_seconds + 1)` 후 같은 채널, 같은 offset에서 resume
* `60 < wait_seconds <= 300`이면 채널당 1~2회만 재시도
* `> 300초`면 이번 사이클은 중단
* 상태는 `ERROR`보다 `PENDING_BACKFILL` 유지 + `last_error_message="FloodWait 421s"` 기록 권장

이건 공식 수치가 아니라, 공식 문서가 제공하는 신호값(`FLOOD_WAIT_X`, Telethon threshold 동작)을 바탕으로 한 운영 정책입니다. ([텔레그램][1])

마지막으로 구현 팁 하나만 드리면, **resume 기준은 “마지막으로 API에서 받아온 메시지 ID”보다 “마지막으로 DB commit까지 끝난 메시지 ID”**가 더 안전합니다. 그래야 FloodWait 직전 메시지가 처리만 되고 커밋이 안 된 경우를 피할 수 있습니다. 이건 공식 문서라기보다 수집기 무결성 측면의 구현 원칙입니다.

원하시면 다음 답변에서 바로
**`client.py`의 `iter_channel_messages()`에 FloodWait 재개 로직을 넣은 코드 초안**을 작성해드리겠습니다.

좋습니다. 아래처럼 가면 됩니다.

공식 기준으로는 `FLOOD_WAIT_X`가 오면 **X초를 기다린 뒤 다시 시도**해야 하고, Telethon 1.42는 `flood_sleep_threshold` 이하의 짧은 대기는 자동 sleep, 그보다 크면 `FloodWaitError`를 raise합니다. Telethon v1 계열은 보통 `e.seconds`, v2 계열은 `e.value`로 대기 시간을 읽습니다. 또 Telethon 문서는 **정확한 제한 수치가 공개돼 있지 않다**고 안내합니다. ([텔레그램][1])

그래서 실무 정책은 이렇게 두는 걸 권합니다.

* `flood_sleep_threshold`: 20~30초
* `wait_seconds <= 60`: 자동 재개
* `60 < wait_seconds <= 300`: 1~2회 정도만 재시도
* `wait_seconds > 300`: 이번 백필은 중단하고 상위에서 `PENDING_BACKFILL` 유지 또는 `ERROR` 처리
* **재개 기준은 마지막으로 성공적으로 yield한 메시지 ID 이후**로 잡기

위 숫자는 공식값이 아니라 운영 정책값입니다. 공식 문서는 wait seconds와 auto-sleep threshold만 설명하고, “몇 번까지 재시도” 같은 숫자는 정해주지 않습니다. ([Telethon Documentation][2])

## `client.py` 코드 초안

아래는 `iter_channel_messages()`에 FloodWait 방어와 resume 로직을 넣은 예시입니다.

```python
from __future__ import annotations

import asyncio
from datetime import datetime, timezone, timedelta
from typing import AsyncIterator, Optional

from telethon import TelegramClient, events, errors

from src.common.logger import logger
from config.settings import settings


class TelegramCollector:
    def __init__(self):
        self.client = TelegramClient(
            settings.TELEGRAM_SESSION_NAME,
            settings.TELEGRAM_API_ID,
            settings.TELEGRAM_API_HASH,
            flood_sleep_threshold=30,   # 짧은 wait는 Telethon 자동 처리
            request_retries=3,          # 무한 재시도는 권장하지 않음
            connection_retries=5,
            retry_delay=1,
            auto_reconnect=True,
        )

    async def connect(self, interactive: bool = False):
        await self.client.connect()
        if not await self.client.is_user_authorized():
            if not interactive:
                raise RuntimeError("텔레그램 인증 세션이 없습니다.")
            raise RuntimeError("interactive 인증 로직이 아직 연결되지 않았습니다.")

    async def disconnect(self):
        await self.client.disconnect()

    async def run_until_disconnected(self):
        await self.client.run_until_disconnected()

    def register_new_message_handler(self, handler):
        self.client.add_event_handler(handler, events.NewMessage)

    def register_edited_message_handler(self, handler):
        self.client.add_event_handler(handler, events.MessageEdited)

    @staticmethod
    def _get_flood_wait_seconds(exc: Exception) -> int:
        # Telethon v1: .seconds / v2: .value
        seconds = getattr(exc, "seconds", None)
        if seconds is not None:
            return int(seconds)

        value = getattr(exc, "value", None)
        if value is not None:
            return int(value)

        return 0

    async def iter_channel_messages(
        self,
        entity,
        *,
        limit: int = 1000,
        days: int = 3,
        max_retry_count: int = 2,
        max_single_wait_seconds: int = 300,
        max_total_wait_seconds: int = 600,
    ) -> AsyncIterator:
        """
        최근 N일 또는 최대 N건까지 메시지를 순회합니다.
        FloodWait 발생 시 마지막으로 성공적으로 읽은 메시지 ID 기준으로 재개합니다.
        """
        cutoff = datetime.now(timezone.utc) - timedelta(days=days)

        yielded_count = 0
        total_wait_seconds = 0
        retry_count = 0

        # Telethon iter_messages는 기본적으로 최신 -> 과거 순서로 읽습니다.
        # 마지막으로 성공 처리한 메시지 ID보다 "더 과거" 쪽만 이어 읽기 위해 offset_id를 사용합니다.
        last_success_message_id: Optional[int] = None

        while yielded_count < limit:
            try:
                current_batch_count = 0

                async for message in self.client.iter_messages(
                    entity,
                    limit=(limit - yielded_count),
                    offset_id=last_success_message_id or 0,
                ):
                    msg_date = getattr(message, "date", None)
                    if msg_date is not None:
                        if msg_date.tzinfo is None:
                            msg_date = msg_date.replace(tzinfo=timezone.utc)
                        else:
                            msg_date = msg_date.astimezone(timezone.utc)

                        if msg_date < cutoff:
                            return

                    yield message

                    yielded_count += 1
                    current_batch_count += 1
                    last_success_message_id = message.id

                    if yielded_count >= limit:
                        return

                # 더 이상 메시지가 없으면 종료
                if current_batch_count == 0:
                    return

                # 정상 진행이 있었으면 retry 카운터는 리셋
                retry_count = 0

            except errors.FloodWaitError as e:
                wait_seconds = self._get_flood_wait_seconds(e)
                total_wait_seconds += wait_seconds
                retry_count += 1

                logger.warning(
                    "FloodWait 감지: entity=%s wait=%ss retry=%s last_success_message_id=%s",
                    entity,
                    wait_seconds,
                    retry_count,
                    last_success_message_id,
                )

                if wait_seconds > max_single_wait_seconds:
                    raise RuntimeError(
                        f"FloodWait too long: {wait_seconds}s (single wait limit exceeded)"
                    ) from e

                if total_wait_seconds > max_total_wait_seconds:
                    raise RuntimeError(
                        f"FloodWait total too long: {total_wait_seconds}s (total wait limit exceeded)"
                    ) from e

                if retry_count > max_retry_count:
                    raise RuntimeError(
                        f"FloodWait retry exceeded: retry_count={retry_count}"
                    ) from e

                await asyncio.sleep(wait_seconds + 1)
                continue
```

## 이 구현에서 중요한 점

`flood_sleep_threshold=30`을 주면, 그 이하의 짧은 wait는 Telethon이 자동으로 처리하고, 더 긴 wait만 여기 `except errors.FloodWaitError`로 올라옵니다. Telethon 문서상 기본 threshold는 60초이며, threshold 이하에서는 자동 sleep, 초과하면 예외를 올립니다. `request_retries`도 존재하지만 무한 재시도는 권장되지 않는다고 문서에 적혀 있습니다. ([Telethon Documentation][2])

`last_success_message_id`를 resume 기준으로 쓴 이유는, Telegram이 준 wait 뒤에 **이미 성공적으로 처리된 마지막 지점 이후부터 다시 읽기** 위해서입니다. 공식 문서가 resume 설계 자체를 강제하진 않지만, `FLOOD_WAIT_X`의 의미가 “X초 뒤 재시도”이므로, 같은 구간을 처음부터 다시 긁기보다 마지막 성공 지점 이후로 재개하는 편이 중복과 누락을 줄이기 좋습니다. `FLOOD_WAIT_X` 자체의 의미는 Telegram 공식 에러 문서에 나와 있습니다. ([텔레그램][1])

## 상위 레벨 정책 추천

이 iterator가 `RuntimeError("FloodWait too long...")`를 올리면, `startup_tasks.py` 쪽에서는 **즉시 `ERROR`로 단정하기보다** 아래처럼 처리하는 걸 권합니다.

* `wait <= 300초`: 여기서 이미 재시도
* `wait > 300초` 또는 누적 대기시간 초과:
  `last_error_message` 기록
  상태는 `PENDING_BACKFILL` 유지 또는 별도 `RATE_LIMITED` 고려
  다음 사이클에 다시 시도

이유는 Telegram 공식 의미가 “지금은 기다려라”이지 “영구 실패”가 아니기 때문입니다. 또 Telethon 문서도 정확한 제한 수치는 알 수 없다고 안내합니다. 

바로 이어서 할 가장 자연스러운 작업은 **`startup_tasks.py`에서 FloodWait 장기 대기 시 `ERROR` 대신 `PENDING_BACKFILL` 유지로 처리하는 상태 전이 코드**를 붙이는 것입니다.

좋습니다.
이 단계에서는 **문자열로 `"FloodWait too long"`를 파싱하지 말고, 전용 예외로 상태 전이를 분기**하는 쪽이 훨씬 안전합니다.

즉 정책은 이렇게 잡는 게 좋습니다.

* 짧은 FloodWait: `client.py` 내부에서 sleep 후 resume
* 너무 긴 FloodWait / 누적 대기 초과 / 재시도 초과:
  `BackfillDeferredError` 같은 **전용 예외**를 올림
* `startup_tasks.py`는 이 예외를 잡아서
  **`ERROR`가 아니라 `PENDING_BACKFILL` 유지** + `last_error_message`만 남김

이렇게 하면 “장애”와 “지연”이 분리됩니다.

```python
from config.settings import settings

            raise RuntimeError("interactive 인증 로직이 아직 연결되지 않았습니다.")

    async def disconnect(self):
        await self.client.disconnect()

    async def run_until_disconnected(self):
        await self.client.run_until_disconnected()

    def register_new_message_handler(self, handler):
        self.client.add_event_handler(handler, events.NewMessage)

    def register_edited_message_handler(self, handler):
        self.client.add_event_handler(handler, events.MessageEdited)

    @staticmethod
    def _get_flood_wait_seconds(exc: Exception) -> int:
        seconds = getattr(exc, "seconds", None)
        if seconds is not None:
            return int(seconds)

        value = getattr(exc, "value", None)
        if value is not None:
            return int(value)

        return 0

    async def iter_channel_messages(
        self,
        entity,
        *,
        limit: int = 1000,
        days: int = 3,
        max_retry_count: int = 2,
        max_single_wait_seconds: int = 300,
        max_total_wait_seconds: int = 600,
    ) -> AsyncIterator:
        """
        최근 N일 또는 최대 N건까지 메시지를 순회합니다.
        FloodWait 발생 시 마지막 성공 메시지 ID 기준으로 resume 합니다.
        너무 긴 대기/누적 대기/재시도 초과는 BackfillDeferredError로 상위에 위임합니다.
        """
        cutoff = datetime.now(timezone.utc) - timedelta(days=days)

        yielded_count = 0
        total_wait_seconds = 0
        retry_count = 0
        last_success_message_id: Optional[int] = None

        while yielded_count < limit:
            try:
                current_batch_count = 0

                async for message in self.client.iter_messages(
                    entity,
                    limit=(limit - yielded_count),
                    offset_id=last_success_message_id or 0,
                ):
                    msg_date = getattr(message, "date", None)
                    if msg_date is not None:
                        if msg_date.tzinfo is None:
                            msg_date = msg_date.replace(tzinfo=timezone.utc)
                        else:
                            msg_date = msg_date.astimezone(timezone.utc)

                        if msg_date < cutoff:
                            return

                    yield message

                    yielded_count += 1
                    current_batch_count += 1
                    last_success_message_id = message.id

                    if yielded_count >= limit:
                        return

                if current_batch_count == 0:
                    return

                retry_count = 0

            except errors.FloodWaitError as e:
                wait_seconds = self._get_flood_wait_seconds(e)
                total_wait_seconds += wait_seconds
                retry_count += 1

                logger.warning(
                    "FloodWait 감지 entity=%s wait=%ss retry=%s total_wait=%s last_success_message_id=%s",
                    entity,
                    wait_seconds,
                    retry_count,
                    total_wait_seconds,
                    last_success_message_id,
                )

                if wait_seconds > max_single_wait_seconds:
                    raise BackfillDeferredError(
                        reason="single_flood_wait_too_long",
                        wait_seconds=wait_seconds,
                        retry_count=retry_count,
                        total_wait_seconds=total_wait_seconds,
                    ) from e

                if total_wait_seconds > max_total_wait_seconds:
                    raise BackfillDeferredError(
                        reason="total_flood_wait_too_long",
                        wait_seconds=wait_seconds,
                        retry_count=retry_count,
                        total_wait_seconds=total_wait_seconds,
                    ) from e

                if retry_count > max_retry_count:
                    raise BackfillDeferredError(
                        reason="flood_wait_retry_exceeded",
                        wait_seconds=wait_seconds,
                        retry_count=retry_count,
                        total_wait_seconds=total_wait_seconds,
                    ) from e

                await asyncio.sleep(wait_seconds + 1)
                continue
```

```python
async def backfill_missed_messages(collector=None, persist_func=None) -> None:
    logger.info("PENDING_BACKFILL 상태의 채널 스캔 시작...")

    async with get_session() as db:
        pending_channels = await find_channels_pending_backfill(db)

        if not pending_channels:
            logger.info("현재 백필이 필요한 채널이 없습니다.")
            return

        logger.info(f"총 {len(pending_channels)}개의 채널이 백필 대기 중입니다.")

        if collector is None or persist_func is None:
            logger.info("collector/persist_func 미주입 상태이므로 실제 백필은 건너뜁니다.")
            return

        for channel, state in pending_channels:
            logger.info(f"백필 대상 채널: [{channel.title}] ({channel.tg_channel_id})")

            try:
                processed_count = await perform_backfill_for_channel(
                    db=db,
                    collector=collector,
                    channel=channel,
                    persist_func=persist_func,
                    days=BACKFILL_DAYS,
                    limit=BACKFILL_LIMIT,
                )

                await mark_backfill_completed(db, channel.id)
                await db.commit()

                logger.info(
                    f"백필 완료 처리: [{channel.title}] ({channel.tg_channel_id}) "
                    f"messages={processed_count}"
                )

            except BackfillDeferredError as e:
                await db.rollback()

                # FloodWait 장기 대기 → 장애가 아니라 지연으로 간주
                await mark_backfill_deferred(db, channel.id, str(e))
                await db.commit()

                logger.warning(

                )

            except Exception as e:
                await db.rollback()

                await mark_backfill_failed(db, channel.id, str(e))
                await db.commit()

                logger.error(
                    f"백필 실패 처리: [{channel.title}] ({channel.tg_channel_id}) error={e}"
                )
```

```python
async def mark_backfill_deferred(db, channel_id: int, reason: str) -> None:
    """
    FloodWait 등으로 이번 사이클에서 백필을 연기할 때 사용.
    상태는 PENDING_BACKFILL 유지, 에러 메시지만 기록.
    """
    stmt = select(TelegramChannelCollectionState).where(
        TelegramChannelCollectionState.telegram_channel_id == channel_id
    )
    result = await db.execute(stmt)
    state = result.scalar_one_or_none()

    now = datetime.now(timezone.utc)

    if state is None:
        state = TelegramChannelCollectionState(
            telegram_channel_id=channel_id,
            status=_status_value(ChannelCollectionStatus.PENDING_BACKFILL),
            last_error_message=reason,
            last_error_at=now,
        )
        db.add(state)
    else:
        state.status = _status_value(ChannelCollectionStatus.PENDING_BACKFILL)
        state.last_error_message = reason
        state.last_error_at = now
```

# 3) 이 정책의 의미

이제 상태 전이는 이렇게 됩니다.

* 정상 백필 완료 → `ACTIVE`
* 권한 문제/파싱 오류/예상 못한 예외 → `ERROR`
* FloodWait 장기 대기 → `PENDING_BACKFILL 유지`

이게 좋은 이유는, FloodWait는 “실패”보다 **“이번 사이클에서 미뤄야 하는 작업”**에 가깝기 때문입니다.
