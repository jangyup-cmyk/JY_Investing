# Antigravity용 수정 프롬프트

## 목적

* **Phase 1 기반 구축**을 설계하되
* **WEB/API 서버와 Collector Worker 분리 구조**를 반영
* 과설계 없이 초기 현실형 구조로 설계안 받기

```text
프로젝트명: 텔레그램 테마연동 자동매매 프로그램

프로젝트 목표:
텔레그램 정보를 수집하여 AI 에이전트가 분석하고,
5개 이상의 AI 에이전트가 상호 토론을 통해 신뢰성 있는 테마를 발굴한 뒤,
발굴된 테마를 자동매매에 활용하는 시스템을 구축하려고 한다.

현재 내가 진행하려는 범위는 전체 프로젝트가 아니라
"Phase 1. 기반 구축"만이다.

개발 환경:
- Python 3.11.9 64bit
- 가상환경: .venv
- IDE: VS Code
- 루트폴더: D:\telegram_trading

중요한 시스템 전제:
- 텔레그램 연동은 Bot API가 아니라 사용자 계정 세션 방식 예정
- 향후 Telethon 기반 수집 예정
- 증권사 API는 한국투자증권 API 예정
- GUI는 WEB 버전 + APP 버전 예정
- 백엔드는 FastAPI 중심 예정
- DB는 PostgreSQL 중심으로 검토 중

중요한 규모 전제:
- 현재 채널 수는 약 150개 수준이며 앞으로 계속 증가 가능
- 공시채널 1개는 하루 약 15,000건 수준
- 나머지 다수 채널은 대체로 하루 500건 이하 수준
- 따라서 초기부터 대규모 분산 시스템까지는 아니지만,
  웹 서버와 텔레그램 수집 워커는 분리하는 구조가 적절하다고 판단하고 있음

매우 중요:
지금은 절대 Phase 2(텔레그램 수집 상세 구현)로 넘어가지 말고,
오직 "Phase 1. 기반 구축"만 설계해줘.

이번 Phase 1의 핵심 목표:
1. WEB/API 서버와 Collector Worker를 분리 가능한 구조로 잡기
2. requirements.txt
3. .env / .env.example
4. 설정 로더
5. 로깅
6. DB 연결
7. 공통 예외 처리
8. 최소 실행 가능한 FastAPI 앱 뼈대
9. 최소 실행 가능한 Collector Worker 뼈대
10. health check / ready check
11. 이후 Phase 2 텔레그램 수집으로 자연스럽게 넘어갈 수 있는 기초 구조

내가 원하는 설계 방향:
- 설계자 관점으로 답변할 것
- 추상론보다 실제 프로젝트에 바로 반영 가능한 수준으로 구체화할 것
- 기능 분리, 파일 분리, 책임 분리를 명확히 할 것
- 웹 서버와 워커가 같은 공통 모듈(settings/logger/db/exceptions)을 공유하는 구조로 설계할 것
- 아직 이 단계에서는 텔레그램 상세 수집 로직 구현은 하지 말 것
- 그러나 run_web.py, run_collector.py 수준의 실행 진입점은 고려할 것
- 과도한 마이크로서비스 구조, Kafka, 큐 시스템 같은 과설계는 피할 것
```
