좋습니다. 이어서 **`tests/workers/test_inference_worker.py` 전체 초안**을 드리겠습니다.

이 파일의 목적은 아래를 검증하는 것입니다.

* `InferenceWorker.run_once()`가 `single_message_summary` job을 정상 소비하는지
* `NormalizedMessage + MessageFeature`를 읽어서 `AnalysisResult`를 생성하는지
* job이 `DONE`으로 끝나는지
* 입력 누락/LLM 오류 시 `FAILED` 또는 `DEAD_LETTER`로 전이되는지
* `WINDOW_THEME_ANALYSIS` placeholder 경로도 최소한 깨지지 않는지

아래 코드는 **앞서 제안한 `InferenceWorker` 골격 기준**입니다.
실제 프로젝트에 맞게 import 경로, enum/string 비교, fixture 공용화 정도만 조정하시면 됩니다.

async def count_rows(async_session: AsyncSession, model) -> int:
    return int(await async_session.scalar(select(func.count()).select_from(model)) or 0)


async def get_job(async_session: AsyncSession, job_id: int) -> AnalysisJob | None:
    return await async_session.get(AnalysisJob, job_id)


async def get_latest_result(async_session: AsyncSession, job_id: int) -> AnalysisResult | None:
    return await async_session.scalar(
        select(AnalysisResult)
        .where(AnalysisResult.job_id == job_id)
        .order_by(AnalysisResult.id.desc())
    )

        select(AnalysisJob)
        .where(AnalysisJob.job_type == job_type)
        .order_by(AnalysisJob.id.asc())
    )
    return list(result)


async def seed_single_message_summary_job(

    *,
    message: Message,
    message_version: MessageVersion,
    normalized_message: NormalizedMessage,
    message_feature: MessageFeature,
    max_attempts: int = 3,
) -> AnalysisJob:
    job = await AnalysisJobService.enqueue_job(
        async_session,
        job_type=JobType.SINGLE_MESSAGE_SUMMARY,
        target_type=JobTargetType.MESSAGE,
        target_id=message.id,
        idempotency_key=f"single_message_summary:message:{message.id}:version:{message_version.id}:sv1",

            "message_id": message.id,
            "version_id": message_version.id,
            "normalized_message_id": normalized_message.id,
            "feature_id": message_feature.id,
        },
        priority=200,
        max_attempts=max_attempts,
    )
    await async_session.commit()
    return job

### 이 파일에서 실제 프로젝트에 맞춰 조정할 부분

가장 가능성이 높은 수정 포인트는 아래입니다.

`Channel`, `Message` 생성 시 필수 컬럼이 실제 모델과 다를 수 있습니다.
`JobType`, `JobStatus`, `JobTargetType`, `ResultType`가 enum이 아니라 문자열이면 비교식을 문자열 기준으로 바꿔야 합니다.
프로젝트에 공용 `conftest.py` fixture가 있다면 이 파일의 DB fixture는 제거하고 공용 fixture를 쓰는 편이 더 좋습니다.
현재 `InferenceWorker`가 `AnalysisResult` 저장 후 같은 세션에서 `mark_done()`까지 호출하는 구조라면 테스트는 그대로 맞지만, 세션 분리 방식이 실제 구현과 다르면 일부 commit 타이밍만 맞춰주면 됩니다.

### 이 테스트 파일이 검증하는 핵심

이 파일이 통과하면 최소한 아래는 확인됩니다.

* `InferenceWorker`가 idle 상태에서 `False`를 반환한다
* `single_message_summary` job을 소비해 `AnalysisResult`를 생성한다
* 입력이 없으면 `INFERENCE_INPUT_NOT_FOUND`로 실패 처리된다
* LLM 오류가 나면 재시도 가능 job은 `PENDING`, 소진된 job은 `DEAD_LETTER`가 된다
* `WINDOW_THEME_ANALYSIS` placeholder 경로가 깨지지 않는다
* `job_types` 필터가 실제로 동작한다

즉, 지금까지 정리한
**collector → preprocess → inference**
파이프라인의 테스트 골격이 한 세트로 갖춰지게 됩니다.
