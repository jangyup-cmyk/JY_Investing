실제 코드 기준으로 매핑은 거의 맞았습니다. 다만 **지금 상태로는 5군데를 꼭 손봐야** 합니다.

가장 좋은 소식부터 말하면, 이름 치환 자체는 잘 됐습니다.
실제 모델명은 `TelegramChannel`, `TelegramMessage`, `MessageVersion`, `AnalysisJob`, `AnalysisResult`이고, 원문 필드는 `message_text_raw`, 편집 시각은 `edited_at`, 메시지 FK는 `telegram_channel_id`입니다. 세션 쪽도 `session_scope()`와 `AsyncSessionLocal()` alias가 이미 준비돼 있고, `startup_tasks.py`는 `persist_func`를 주입받아 백필에서 공통 persist 함수를 호출하는 구조라서, 제가 제안했던 아키텍처 방향과 정확히 맞습니다. `run_collector.py`도 `init_db() → CollectorWorker.start() → close_db()` 흐름으로 깔끔합니다.    

그런데 아래 5개는 실제로 수정이 필요합니다.

첫째, **`Base`가 두 개일 가능성**이 있습니다.
`session.py`는 `src.infra.db.base`에서 `Base`를 가져오는데, `models.py`는 파일 안에서 다시 `class Base(DeclarativeBase)`를 정의하고 있습니다. 이게 실제로 서로 다른 객체면 `init_db()`에서 `create_all()`을 돌려도 모델 테이블이 안 생기거나 일부만 생길 수 있습니다. 이건 가장 먼저 통일해야 합니다. `models.py`에서 로컬 `Base`를 제거하고 `from src.infra.db.base import Base`로 맞추는 쪽이 안전합니다.  

둘째, **`AnalysisJobService.enqueue_job()` 시그니처를 실제 모델에 맞춰야 합니다.**
지금 `collector_worker.py`는 `enqueue_job()` 호출 시 `telegram_message_id=message.id`, `message_version_id=version.id`를 넘기고 있습니다. 그런데 제가 앞서 제안한 서비스 초안은 이 두 인자를 받지 않았습니다. 실제 `AnalysisJob` 모델에는 `telegram_message_id`, `message_version_id`, `job_type`, `target_type`, `target_id`, `status`, `priority`, `attempt_count`, `max_attempts`, `payload_json` 등이 있으니, 서비스 메서드는 이 컬럼들을 그대로 반영해야 합니다. 

셋째, **enum 저장 방식이 실제 모델과 다릅니다.**
`models.py`에서 `JobType`, `JobStatus`, `JobTargetType`, `ResultType`는 Python Enum으로 선언돼 있지만, 실제 DB 컬럼은 대부분 `String(...)`입니다. 즉 서비스/워커에서 `JobType.NORMALIZE` 같은 Enum 객체를 그대로 넣기보다 `.value`를 넣는 편이 안전합니다. 현재 모델 값은 전부 대문자 문자열(`"NORMALIZE"`, `"PENDING"`, `"MESSAGE"`)이므로, 서비스 내부에서 저장/비교 시 `.value`를 쓰는 쪽으로 통일하는 걸 권합니다. 그렇지 않으면 SQLite 바인딩이나 조건 비교에서 미묘한 문제가 날 수 있습니다. 

넷째, **`TelegramMessage.normalized` 관계는 현재 잘못된 가능성이 큽니다.**
지금 `NormalizedTelegramMessage`는 `(telegram_message_id, message_version_id, cleaning_version)` 유니크 키를 가지고 있어서, **메시지 1개에 여러 정규화 결과가 생길 수 있는 구조**입니다. 그런데 `TelegramMessage` 쪽 관계는 `normalized: ... uselist=False`로 돼 있습니다. 이건 설계와 충돌합니다. 여기서는 one-to-one이 아니라 one-to-many가 맞습니다. 관계명을 `normalized_messages`로 바꾸고 `uselist=False`를 제거하는 게 맞습니다. 

다섯째, **`TelegramMessageFeature`에 `normalized_message` relationship이 빠져 있습니다.**
`NormalizedTelegramMessage`는 `features = relationship("TelegramMessageFeature", back_populates="normalized_message")`를 선언하고 있는데, `TelegramMessageFeature` 클래스에는 현재 `normalized_message = relationship(...)`가 없습니다. 이 상태면 mapper 설정 시 `back_populates` 매칭 오류가 날 수 있습니다. `TelegramMessageFeature`에 아래를 추가해야 합니다.

```python
normalized_message: Mapped["NormalizedTelegramMessage | None"] = relationship(
    "NormalizedTelegramMessage",
    back_populates="features",
)
message_version: Mapped["MessageVersion | None"] = relationship("MessageVersion")
```

이건 전처리 워커가 `normalized_message_id`를 중심으로 동작하기 때문에 실제로 중요합니다. 

그리고 한 가지 운영상 중요한 버그도 있습니다.
현재 `collector_worker.py`는 `persist_message(..., is_backfill=True)`일 때 `priority=1`, 실시간일 때 `priority=100`으로 job을 넣고 있습니다. 그런데 제가 제안한 `claim_next_job()`은 **priority 오름차순**으로 가져오므로, 이대로면 **백필이 실시간보다 먼저 처리**됩니다. 의도대로라면 실시간을 더 높게 해야 하므로 아래처럼 뒤집는 게 맞습니다.

```python
priority=100 if is_backfill else 1
```

또는 서비스에서 내림차순으로 가져가도록 바꾸는 방법도 있지만, 현재 설계 기준으로는 숫자를 뒤집는 편이 덜 위험합니다. 

정리하면, 지금 코드베이스에서 **그대로 유지해도 되는 파일**은 `startup_tasks.py`와 `run_collector.py`입니다. `startup_tasks.py`는 네트워크 I/O 동안 DB 세션을 잡지 않도록 버퍼 + `session_scope()`를 잘 쓰고 있고, `persist_func` 주입 구조도 이미 이상적입니다. `run_collector.py`도 실행 진입점으로 충분히 깔끔합니다.  

반대로 **즉시 수정해야 하는 파일**은 `models.py`, `analysis_job_service.py`, 그리고 priority 줄 1개가 있는 `collector_worker.py`입니다. `collector_worker.py` 자체는 공통 persist 지점, version 생성, normalize enqueue 구조가 이미 잘 들어가 있습니다. 이건 아주 좋은 상태입니다. 

바로 적용할 수정 순서는 이렇습니다.

1. `models.py`

* `Base` 단일화
* `TelegramMessage.normalized`를 list 관계로 수정
* `TelegramMessageFeature.normalized_message` 관계 추가

2. `analysis_job_service.py`

* `telegram_message_id`, `message_version_id` 인자 추가
* `job_type`, `target_type`, `status`를 `.value` 기준으로 저장/비교

3. `collector_worker.py`

* `priority=100 if is_backfill else 1`로 수정

4. 그 다음 테스트 실행

```bash
python -m pytest tests/services/test_analysis_job_service.py -vv
python -m pytest tests/workers/test_collector_persist.py -vv
python -m pytest tests/workers/test_preprocess_worker.py -vv
python -m pytest tests/workers/test_inference_worker.py -vv
```
