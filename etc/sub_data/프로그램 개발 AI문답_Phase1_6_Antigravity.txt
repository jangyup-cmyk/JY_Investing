## Antigravity용 프롬프트

```text
프로젝트명: 텔레그램 테마연동 자동매매 프로그램

프로젝트 목표:
텔레그램 정보를 수집하여 AI 에이전트가 분석하고,
5개 이상의 AI 에이전트가 상호 토론을 통해 신뢰성 있는 테마를 발굴한 뒤,
발굴된 테마를 자동매매에 활용하는 시스템을 구축하려고 한다.

현재 내가 진행하려는 범위는 전체 프로젝트가 아니라
"Phase 1. 기반 구축"만이다.

개발 환경:
- Python 3.11.9 64bit
- 가상환경: .venv
- IDE: VS Code
- 루트폴더: D:\telegram_trading

향후 예정 사항:
- 텔레그램 수집: Telethon 기반 사용자 계정 세션 방식
- 증권사 API: 한국투자증권 API
- GUI: WEB 버전 + APP 버전
- 백엔드는 추후 FastAPI 중심으로 운영 예정
- DB는 PostgreSQL 중심으로 가는 방향을 우선 검토하고 싶음

중요:
지금은 절대 Phase 2(텔레그램 수집 상세 구현)로 넘어가지 말고,
오직 "Phase 1. 기반 구축"만 설계해줘.

Phase 1에서 내가 구축하려는 범위:
1. .venv
2. requirements.txt
3. .env / .env.example
4. 설정 로더
5. 로깅
6. DB 연결
7. 공통 예외 처리
8. 최소 실행 가능한 웹 앱 뼈대
9. health check
10. 이후 Phase 2로 자연스럽게 넘어갈 수 있는 기초 구조

내가 원하는 너의 역할:
- 설계자 관점으로 답변할 것
- 추상론 말고, 실제 바로 프로젝트에 반영 가능한 수준으로 구체화할 것
- 기능 쪼개기, 파일 분리, 책임 분리, 초기 구조 설계 중심으로 제안할 것
- 아직 이 단계에서는 과도한 구현 코드보다 "안전한 구조"를 우선할 것
- 단, 설명만 하지 말고 마지막에는 파일별 초안 코드 뼈대까지 제시할 것

반드시 아래 형식으로 답변해줘.

[출력 형식]

1. Phase 1의 목표 요약
- 이 단계가 왜 필요한지
- 완료되면 무엇이 가능한지
- 아직 하지 말아야 할 것은 무엇인지

2. Phase 1 범위 확정
- 포함 범위
- 제외 범위
- 향후 Phase 2와의 연결 지점

3. 기술 스택 권장안
- 웹 프레임워크
- 설정 관리 방식
- 로깅 방식
- DB/ORM/마이그레이션 방식
- 테스트 도구
- 패키지 목록(requirements.txt 초안)
- 각 패키지를 넣는 이유를 짧게 설명

4. 초기 폴더 구조 제안
- D:\telegram_trading 기준으로
- 폴더 트리를 보기 좋게 제시
- "지금 꼭 필요한 폴더"와 "나중에 추가할 폴더"를 구분

5. 환경변수 설계
- .env.example 초안 작성
- 변수명, 용도, 예시값을 표 형태로 정리
- 지금 필요한 변수와 Phase 2/자동매매 대비용 예약 변수를 구분

6. 설정 로더 설계
- settings.py를 어떤 구조로 둘지
- App / DB / Log / Telegram / Broker 설정을 어떻게 나눌지
- 코드에서 환경변수를 직접 읽지 않고 settings 객체를 통해 읽게 해야 하는 이유 설명

7. 로깅 설계
- 콘솔 로그 + 파일 로그 구조
- 로그 포맷 예시
- logs 폴더 구조
- 모듈명, 레벨, 메시지 형식 표준
- 나중에 system_logs DB 테이블과 연결하기 쉬운 구조로 설계

8. DB 연결 설계
- PostgreSQL 기준 권장안
- SQLAlchemy + Alembic 기준으로
- base.py / session.py / models.py 역할 구분
- 초기에는 어떤 최소 모델부터 두는 것이 좋은지
- 추후 telegram_channels, telegram_messages로 확장될 수 있게 설명

9. 공통 응답 / 공통 예외 처리 설계
- API 성공 응답 표준 JSON
- API 실패 응답 표준 JSON
- AppException, DatabaseException, ExternalAPIException 등 기본 예외 구조 제안
- FastAPI 예외 핸들러 설계 방향 제시

10. 최소 실행 가능한 웹 앱 설계
- app.py 역할
- run_web.py 역할
- /health, /ready 엔드포인트 목적
- startup 시 무엇을 체크할지
- 지금 단계에서 UI 없이도 확인 가능한 최소 동작 정의

11. 추천 개발 순서
- 실제 작업 순서를 Step 1, Step 2, Step 3... 형태로 제시
- 각 단계의 산출물도 같이 제시

12. 완료 기준(체크리스트)
- 무엇이 되면 Phase 1 완료로 볼 수 있는지
- 체크박스 형태로 제시

13. 리스크 및 의사결정 포인트
- Windows 환경, .venv, VS Code, PostgreSQL, Telethon 예정 구조에서
  미리 조심해야 할 점을 정리
- 지금 확정해야 할 것과 나중으로 미뤄도 되는 것을 구분

14. 파일별 초안 코드 뼈대
아래 파일들에 대해 "실제 바로 생성 가능한 최소 골격 코드"를 제시해줘.
단, 지나치게 긴 완성형 코드 말고, 구조가 명확한 실무형 초안으로 작성해줘.

대상 파일:
- requirements.txt
- .env.example
- config/settings.py
- src/common/logger.py
- src/common/exceptions.py
- src/common/responses.py
- src/infra/db/base.py
- src/infra/db/session.py
- src/infra/db/models.py
- src/web/app.py
- src/web/routes/health.py
- run_web.py

15. 최종 정리
- 지금 이 설계대로 가면 Phase 2에서 어떤 점이 쉬워지는지
- 다음 단계에서 어떤 파일부터 구현하면 되는지 우선순위 정리

응답 조건:
- 한국어로 작성
- 표, 코드블록, 트리 구조를 적극 활용
- 설명은 실무적으로 구체적으로 작성
- "바로 이 프로젝트에 적용 가능한 수준"으로 작성
- 불필요한 일반론은 줄이고, 이 프로젝트에 맞는 결정 중심으로 작성
- 설계와 코드 뼈대를 구분해서 보여줘
- 마지막에는 "권장 기본안"을 한 번 더 요약해줘
```

### 2차 후속 프롬프트

```text
방금 제시한 Phase 1 설계를 기준으로,
이제 파일별 초안 코드만 다시 정리해줘.

조건:
- Python 3.11.9
- FastAPI + SQLAlchemy + Alembic + pydantic-settings 기준
- Windows + .venv + VS Code 환경
- 지나치게 많은 기능을 넣지 말고, 최소 실행 가능한 구조만 작성
- 각 파일마다 "이 파일의 역할"을 먼저 2~3줄로 설명한 뒤 코드 제시
- 코드에는 필요한 주석만 넣고, 과도한 설명은 줄여줘

대상 파일:
- requirements.txt
- .env.example
- config/settings.py
- src/common/logger.py
- src/common/exceptions.py
- src/common/responses.py
- src/infra/db/base.py
- src/infra/db/session.py
- src/infra/db/models.py
- src/web/app.py
- src/web/routes/health.py
- run_web.py
```

### 3차 후속 프롬프트

```text
방금 작성한 Phase 1 초안 코드 기준으로,

원하는 출력:
1. 파일 생성 순서
2. 각 파일 생성 목적
3. 생성 후 바로 실행해서 확인할 명령어
4. 예상되는 오류와 점검 포인트
5. Phase 1 완료 확인 방법
```
