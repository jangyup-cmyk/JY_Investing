# 3. 왜 이 migration 이 지금 구조와 잘 맞는가

핵심은 세 가지입니다.

## 첫째, idempotency 보장

아래 3개 unique 제약이 매우 중요합니다.

```python
uq_messages_channel_msg
uq_analysis_jobs_message_job_type
uq_preprocessed_messages_message_id
uq_message_summaries_message_id
```

이게 있어야

* 같은 텔레그램 메시지 중복 저장
* 같은 job 중복 생성
* 전처리 결과 중복 insert
* summary 결과 중복 insert

를 DB 레벨에서 마지막으로 막을 수 있습니다.

---

## 둘째, run_once() + scheduler 흐름에 맞음

`analysis_jobs` 에 아래 필드가 있어야 합니다.

* `attempt_count`
* `max_attempts`
* `last_error`
* `started_at`
* `finished_at`
* `heartbeat_at`

그래야 앞에서 정리한

* `claim_next_job()`
* `complete_job()`
* `fail_job()`
* `requeue_stuck_jobs()`
* `mark_dead_jobs()`

가 전부 자연스럽게 맞아떨어집니다.

---

## 셋째, 조회 성능 최소 보장

worker 가 가장 많이 하게 될 조회는 사실상 이겁니다.

```sql
WHERE job_type = ?
  AND status IN ('PENDING', 'RETRY_PENDING')
ORDER BY created_at
```

그래서 이 인덱스가 중요합니다.

```python
ix_analysis_jobs_job_type_status_created_at
```

이 인덱스 하나만 있어도 큐 처리 성능이 훨씬 안정됩니다.

# 5. 지금 코드와 migration 사이에서 가장 자주 어긋나는 부분

실제로는 여기서 자주 깨집니다.

## 1. `payload` 타입

SQLite에서는 `JSON` 이 사실상 text처럼 동작할 수 있고, PostgreSQL에서는 진짜 JSON/JSONB 입니다.

초기에는 `sa.JSON()` 으로 가도 충분합니다.
나중에 PostgreSQL 최적화할 때만 `JSONB` 로 옮겨도 됩니다.

---

## 2. `telegram_message_id` 타입

텔레그램 메시지 ID 는 보통 `Integer` 로도 되지만, 여유 있게 `BigInteger` 가 더 안전합니다.

---

## 3. `created_at`, `updated_at` 자동 처리

초기에는 서비스 레이어에서 직접 넣는 편이 단순합니다.

예:

* `persist_message()` 에서 `created_at`
* `upsert_preprocessed_message()` 에서 `created_at`, `updated_at`
* `upsert_summary()` 에서 `created_at`, `updated_at`

즉, 처음부터 DB default 에 너무 많이 맡기지 않는 편이 디버깅이 쉽습니다.

---

# 6. 이미 테이블이 있는데 migration 을 맞춰야 하는 경우

만약 지금 테이블이 일부라도 이미 존재한다면, 새로 create 하는 대신 **alter migration** 으로 가야 합니다.

그 경우 가장 우선순위는 아래입니다.

## analysis_jobs 에 추가

* `payload`
* `attempt_count`
* `max_attempts`
* `last_error`
* `started_at`
* `finished_at`
* `heartbeat_at`

## unique constraint 추가

* `messages(channel_id, telegram_message_id)`
* `analysis_jobs(message_id, job_type)`

## 결과 테이블 unique 추가

* `preprocessed_messages.message_id`
* `message_summaries.message_id`

즉, 실제 상황이 “빈 DB”냐 “기존 DB 보정”이냐에 따라 migration 전략이 달라집니다.

# 7. 기존 DB 보정용 alter migration 예시

혹시 이미 `analysis_jobs` 테이블이 있고 컬럼만 부족한 상황이면 아래 같은 방식으로 갑니다.

```python
def upgrade():
    op.add_column("analysis_jobs", sa.Column("payload", sa.JSON(), nullable=True))
    op.add_column("analysis_jobs", sa.Column("attempt_count", sa.Integer(), nullable=False, server_default="0"))
    op.add_column("analysis_jobs", sa.Column("max_attempts", sa.Integer(), nullable=False, server_default="3"))
    op.add_column("analysis_jobs", sa.Column("last_error", sa.Text(), nullable=True))
    op.add_column("analysis_jobs", sa.Column("started_at", sa.DateTime(), nullable=True))
    op.add_column("analysis_jobs", sa.Column("finished_at", sa.DateTime(), nullable=True))
    op.add_column("analysis_jobs", sa.Column("heartbeat_at", sa.DateTime(), nullable=True))

    op.create_unique_constraint(
        "uq_analysis_jobs_message_job_type",
        "analysis_jobs",
        ["message_id", "job_type"],
    )

    op.create_index(
        "ix_analysis_jobs_job_type_status_created_at",
        "analysis_jobs",
        ["job_type", "status", "created_at"],
    )
```

이건 **신규 생성형 migration** 이 아니라 **보정형 migration** 입니다.

# 8. 지금 단계에서 가장 실전적인 체크리스트

Alembic 초안 반영 전에 이 5개만 먼저 보면 됩니다.

## 체크 1

현재 `AnalysisJob` 모델에 아래 필드가 실제로 있는가

* `payload`
* `attempt_count`
* `max_attempts`
* `last_error`
* `started_at`
* `finished_at`
* `heartbeat_at`

## 체크 2

`MessageService.persist_message()` 가 `messages` + `analysis_jobs` 를 한 트랜잭션으로 저장하는가

## 체크 3

`PreprocessedMessage`, `MessageSummary` 가 `message_id unique` 구조인가

## 체크 4

현재 DB 가 비어 있는가, 아니면 기존 데이터가 있는가

## 체크 5

운영 DB 가 SQLite 인가 PostgreSQL 인가

운영 DB 가 PostgreSQL 이면 이후 `claim_next_job()` 의 `SKIP LOCKED` 와도 훨씬 잘 맞습니다.

# 10. 한 줄 요약

지금 필요한 Alembic 초안의 핵심은
**`analysis_jobs` 를 단순 작업 테이블이 아니라 재시도/복구 가능한 큐 테이블로 완성하고, `message_id` 기반 unique 제약으로 파이프라인의 idempotency 를 DB 레벨에서 고정하는 것**입니다.

다음 답변에서는 이어서 바로
**현재 네 프로젝트가 “기존 테이블 보정형”인지 “신규 생성형”인지에 따라 나눈 Alembic 적용 순서**까지 정리해드리겠습니다.
