좋습니다. 그럼 바로 **`persist_message()` 필드 정합성 점검 절차**를 실전 기준으로 정리하겠습니다.

## 1. 점검 대상

한 번에 아래 3곳만 펼쳐 놓고 비교하면 됩니다.

1. `MessageIngestService.persist_message()`
2. `TelegramMessage` 모델
3. `MessageVersion` 모델

핵심은 단순합니다.

* `persist_message()`에서 생성하는 **키워드 인자 이름**
* 모델 선언부의 **ORM 컬럼 속성명**

이 둘이 **완전히 동일**해야 합니다.

---

## 2. 먼저 봐야 하는 코드 형태

보통 `persist_message()` 안에는 이런 부분이 있습니다.

```python
message = TelegramMessage(
    ...필드들...
)

version = MessageVersion(
    ...필드들...
)
```

여기서 점검 대상은 **왼쪽 이름**입니다.

예를 들어:

```python
TelegramMessage(
    channel_id=channel.id,
    telegram_message_id=tg_message.id,
    raw_text=text,
    posted_at=tg_message.date,
)
```

이 경우 점검 대상은 아래 4개입니다.

* `channel_id`
* `telegram_message_id`
* `raw_text`
* `posted_at`

오른쪽 값보다 **왼쪽 이름이 모델과 같은지**를 먼저 봐야 합니다.

---

## 3. `TelegramMessage`에서 바로 대조할 항목

아래 항목은 거의 반드시 확인해야 합니다.

### 식별자 계열

* `id`
* `channel_id`
* `telegram_message_id` 또는 `message_id`

여기서 가장 흔한 실수는:

* 서비스는 `message_id=` 사용
* 모델은 `telegram_message_id` 로 선언

이 경우 바로 불일치입니다.

---

### 본문 계열

* `raw_text`
* `text`
* `content`
* `caption`

이 중 모델이 무엇을 쓰는지 확인해야 합니다.

가장 흔한 실수는:

* 서비스는 `text=...`
* 모델은 `raw_text` 만 존재

---

### 시간 계열

* `posted_at`
* `message_date`
* `collected_at`
* `edit_date`
* `last_edited_at`

여기서 자주 섞입니다.

예:

* 텔레그램 원본 작성 시각 → `posted_at`
* 우리 시스템 수집 시각 → `collected_at`

이 둘을 하나로 합쳐 넣으면 나중에 정렬·재수집·버전 비교가 꼬입니다.

---

### 부가 상태 계열

* `views`
* `forwards`
* `grouped_id`
* `reply_to_message_id`
* `sender_id`
* `has_media`
* `is_deleted`

모델에 없는 필드를 서비스가 넣고 있거나,
반대로 모델 필수값인데 서비스에서 빠뜨리는 경우가 많습니다.

---

## 4. `MessageVersion`에서 바로 대조할 항목

여기는 특히 더 중요합니다.

### FK 연결 필드

가장 먼저 확인할 것:

* `message_id`
* `telegram_message_id`
* `telegram_message_pk`

셋 중 실제 모델이 무엇을 쓰는지 정확히 봐야 합니다.

가장 흔한 오류는 다음 두 가지입니다.

1. **필드명 자체가 틀림**

   * 모델: `telegram_message_id`
   * 서비스: `message_id=message.id`

2. **이름은 맞는데 값 의미가 틀림**

   * 모델의 FK는 내부 PK(`TelegramMessage.id`)를 기대
   * 서비스는 외부 텔레그램 ID(`tg_message.id`)를 넣음

이건 실행은 될 수도 있지만 **데이터 연결이 틀어지는 위험한 오류**입니다.

---

### 버전 정보 계열

* `version_no`
* `version`
* `revision_no`

서비스와 모델이 같은 이름을 쓰는지 봐야 합니다.

---

### 원문 스냅샷 계열

* `raw_text`
* `text`
* `content`
* `caption`
* `metadata_json`

여기도 `TelegramMessage`와 똑같이 이름 불일치가 자주 납니다.

---

## 5. 가장 실전적인 점검 순서

아래 순서대로 하면 됩니다.

### 1단계

`TelegramMessage` 모델에서 컬럼명만 뽑습니다.

예:

```python
id
channel_id
telegram_message_id
raw_text
posted_at
collected_at
edit_date
views
forwards
```

### 2단계

`persist_message()`의 `TelegramMessage(...)` 생성부를 한 줄씩 대조합니다.

예:

* `channel_id` → 체크 완료
* `message_id` → 수정 필요
* `text` → 수정 필요
* `date` → 수정 필요

### 3단계

`MessageVersion`도 똑같이 합니다.

예:

* `telegram_message_id` → 확인 필요
* `version` → 수정 필요
* `text` → 수정 필요

### 4단계

`flush()` 위치를 확인합니다.

`MessageVersion`가 `TelegramMessage.id`를 참조한다면 보통 이 흐름이어야 안전합니다.

```python
session.add(message)
session.flush()

version = MessageVersion(
    telegram_message_id=message.id,
    ...
)
session.add(version)
```

`flush()` 없이 `message.id`를 바로 쓰면 FK 값이 비정상일 수 있습니다.

---

## 6. 체크 결과는 이렇게 정리하면 됩니다

아래 형식으로 채우면 가장 빠릅니다.

### TelegramMessage 매핑 점검표

| persist_message 인자 | TelegramMessage 필드    | 상태    | 비고             |
| ------------------ | --------------------- | ----- | -------------- |
| `channel_id`       | `channel_id`          | 체크 완료 | 정상             |
| `message_id`       | `telegram_message_id` | 수정 필요 | 이름 불일치         |
| `text`             | `raw_text`            | 수정 필요 | 본문 필드명 불일치     |
| `date`             | `posted_at`           | 수정 필요 | 의미는 맞으나 이름 불일치 |

### MessageVersion 매핑 점검표

| persist_message 인자      | MessageVersion 필드     | 상태    | 비고          |
| ----------------------- | --------------------- | ----- | ----------- |
| `message_id=message.id` | `telegram_message_id` | 확인 필요 | FK 의미 확인 필요 |
| `version=1`             | `version_no`          | 수정 필요 | 이름 불일치      |
| `text`                  | `raw_text`            | 수정 필요 | 본문 필드명 불일치  |

---

## 7. 이 단계에서 꼭 같이 봐야 하는 부수 점검

필드명만 맞아도 끝은 아닙니다. 아래도 함께 봐야 합니다.

### 중복 메시지 조회 조건

보통 아래 2개 조합이 핵심입니다.

* `channel_id`
* `telegram_message_id`

조회 조건이 이 둘과 다르면 중복 적재 위험이 있습니다.

---

### 수정 메시지(edit) 처리

수정된 메시지가 다시 들어왔을 때:

* `TelegramMessage`는 최신 상태 갱신
* `MessageVersion`는 새 버전 추가

이 정책이 코드에 반영돼 있는지 확인해야 합니다.

---

### nullable / default

모델이 `nullable=False`인데 서비스가 값을 안 넣으면 저장 실패합니다.

특히 아래 필드가 위험합니다.

* `channel_id`
* `telegram_message_id`
* `created_at`
* `collected_at`
* 버전 FK

---

## 8. 바로 패치가 필요한 전형적인 예

예를 들어 현재 코드가 이렇다면:

```python
message = TelegramMessage(
    message_id=tg_message.id,
    text=text,
    date=tg_message.date,
)
```

모델이 아래라면:

```python
telegram_message_id = Column(...)
raw_text = Column(...)
posted_at = Column(...)
```

패치는 이렇게 가야 합니다.

```python
message = TelegramMessage(
    telegram_message_id=tg_message.id,
    raw_text=text,
    posted_at=tg_message.date,
)
```

---

## 9. 지금 기준으로 가장 자연스러운 다음 행동

이제 바로 해야 할 일은 아래 하나입니다.

**`persist_message()`의 `TelegramMessage(...)`, `MessageVersion(...)` 생성부를 표로 옮겨서 `체크 완료 / 수정 필요 / 확인 필요`로 채우는 것**입니다.

코드를 붙여 주시면 다음 답변에서 제가 바로:

* `TelegramMessage` 매핑표
* `MessageVersion` 매핑표
* 수정이 필요한 필드명
* FK 연결 오류 여부
* `flush()` 필요 여부

까지 한 번에 정리해드리겠습니다.
