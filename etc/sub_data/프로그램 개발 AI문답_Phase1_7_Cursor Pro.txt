## Cursor Pro용 Phase 1 구현 메인 프롬프트

```text
프로젝트명: 텔레그램 테마연동 자동매매 프로그램

현재 작업 범위:
전체 프로젝트가 아니라 오직 "Phase 1. 기반 구축"만 구현한다.

개발 환경:
- OS: Windows
- Python 3.11.9 64bit
- 가상환경: .venv
- IDE: VS Code
- 루트폴더: D:\telegram_trading

프로젝트 목표 요약:
텔레그램 정보를 수집하여 AI 에이전트가 분석하고,
5개 이상의 AI 에이전트가 상호 토론을 통해 신뢰성 있는 테마를 발굴한 뒤,
발굴된 테마를 자동매매에 활용하는 시스템을 만들 예정이다.
하지만 지금은 절대 텔레그램 수집 상세 구현이나 자동매매 구현으로 넘어가지 말고,
기반 구축만 진행한다.

Phase 1 구현 범위:
1. requirements.txt 작성
2. .env.example 작성
3. config/settings.py 구현
4. src/common/logger.py 구현
5. src/common/exceptions.py 구현
6. src/common/responses.py 구현
7. src/infra/db/base.py 구현
8. src/infra/db/session.py 구현
9. src/infra/db/models.py 구현
10. src/web/app.py 구현
11. src/web/routes/health.py 구현
12. run_web.py 구현
13. 최소 실행 가능한 FastAPI 앱 상태까지 맞추기
14. /health, /ready 엔드포인트 동작 가능하게 하기

기술 기준:
- FastAPI
- Uvicorn
- SQLAlchemy
- Alembic은 지금 즉시 깊게 구현하지 않아도 되지만, 나중에 붙이기 쉽게 구조를 잡을 것
- pydantic-settings 기반 설정 로더
- python-dotenv
- PostgreSQL 기준 DB 연결 구조
- pytest 기본 준비 가능 구조
- Telethon, 한국투자증권 API는 "향후 확장 예정" 수준으로만 반영하고 지금 구현하지 말 것

중요 제약:
- Phase 2(텔레그램 수집)로 넘어가지 말 것
- 불필요한 과설계 금지
- 그러나 임시방편 코드도 금지
- "지금 당장 실행 가능한 최소 골격"을 목표로 할 것
- 파일 간 책임을 명확히 분리할 것
- 모든 환경변수는 settings 객체를 통해 읽게 할 것
- API 응답 형식은 성공/실패 공통 포맷으로 통일할 것
- 예외 처리도 공통 구조로 설계할 것
- 로그는 콘솔 + 파일 모두 남기게 할 것
- health check에서 DB 연결 상태를 확인할 수 있게 할 것
- 코드 주석은 필요한 수준만 넣고 과도한 설명은 줄일 것
- import 경로가 꼬이지 않게 실제 실행 가능한 구조로 맞출 것

원하는 결과:
- 위 파일들을 실제 생성/수정 가능한 수준의 코드로 작성
- 각 파일은 서로 연결되어 바로 실행 가능해야 함
- run_web.py 실행 시 FastAPI 앱이 뜨고 /health, /ready 응답 가능해야 함
- DB 접속 실패 시에도 예외 형식이 일관되게 유지되어야 함

반드시 아래 순서로 작업해줘:

1. 먼저 최종 파일 트리를 제시
2. 그 다음 각 파일의 역할을 1~3줄로 설명
3. 그 다음 각 파일의 전체 초안 코드를 제시
4. 마지막에 실행 순서를 Windows 기준 명령어로 제시
5. 마지막에 예상 점검 포인트를 체크리스트로 제시

필수 파일 목록:
- requirements.txt
- .env.example
- config/settings.py
- src/common/logger.py
- src/common/exceptions.py
- src/common/responses.py
- src/infra/db/base.py

- src/web/app.py
- src/web/routes/health.py
- run_web.py

반드시 지켜야 할 구현 방향:

[1] requirements.txt
최소 패키지만 넣되, 아래 범주는 반영
- fastapi
- uvicorn[standard]
- sqlalchemy
- alembic
- pydantic-settings
- python-dotenv
- psycopg[binary]
- telethon
- httpx
- pytest

[2] .env.example
아래 성격의 환경변수 포함
- 앱 설정
- DB 설정
- 로그 설정
- 텔레그램 예약 설정
- 한국투자증권 예약 설정

[3] settings.py
아래 구조를 권장
- AppSettings
- DBSettings
- LogSettings
- TelegramSettings
- BrokerSettings
- Settings
그리고 get_settings() 또는 settings singleton 형태로 어디서든 쉽게 불러오게 할 것

[4] logger.py
- 콘솔 + 파일 핸들러
- logs 디렉토리 자동 생성 고려
- module_name 기준 로그 사용이 편한 형태
- 너무 복잡한 커스텀 로거 금지

[5] exceptions.py
최소 아래 예외 구조 포함
- AppException
- DatabaseException
- ExternalAPIException

각 예외는 status_code, message, error_code를 가질 수 있게 설계

[6] responses.py
성공/실패 JSON 응답 helper 작성
예:
{
  "success": true,
  "message": "...",
  "data": ...
}
{
  "success": false,
  "message": "...",
  "error_code": "...",
  "data": null
}

[7] DB 관련
- base.py: DeclarativeBase 또는 Base 정의
- session.py: DATABASE_URL 생성, engine, SessionLocal, get_db()
- models.py: 최소 예시 모델 1개 이상
  권장: SystemLog 또는 HealthCheckLog 수준
- 향후 telegram_channels, telegram_messages 확장 가능하도록 작성

[8] app.py
- FastAPI 앱 생성
- 앱 제목/버전 설정
- 예외 핸들러 등록
- health 라우터 등록
- startup 로그 남기기
- 너무 많은 라우터 미리 만들지 말 것

[9] health.py
최소 2개 엔드포인트:
- GET /health
- GET /ready
둘 다 공통 응답 포맷 사용
/ready 에서는 DB 연결 확인 포함

[10] run_web.py
- uvicorn 실행 진입점
- host, port는 settings 기반
- reload는 환경변수 debug 값 고려 가능

출력 형식:
- 한국어
- 코드블록 적극 사용
- 각 파일은 "파일 경로"를 제목처럼 먼저 쓰고 그 아래 전체 코드 제시
- 추상 설명보다 실행 가능한 결과물 중심
- 마지막에는 "생성 후 실행 순서"와 "오류 점검 포인트"를 정리
```

## Cursor에 추가로 붙이면 좋은 보조 지시문

메인 프롬프트 뒤에 아래를 덧붙이면 결과가 더 안정적입니다.

```text
추가 지시:
- Windows 경로 기준으로 생각할 것
- 상대 import / 절대 import 충돌이 없게 맞출 것
- Python 3.11 문법 기준으로 작성할 것
- pydantic v2 계열 기준으로 작성할 것
- FastAPI 최신 일반 구조를 따르되, 과도한 의존성 주입은 피할 것
- 코드가 길어져도 생략하지 말고 각 파일을 끝까지 작성할 것
- placeholder만 남기지 말고 실제 실행 가능한 최소 코드로 작성할 것
- 단, 아직 사용하지 않는 기능을 억지로 구현하지는 말 것
```

## Cursor에게 “파일 직접 생성 방식”으로 시킬 때 쓰는 버전

Cursor에서 코드 편집을 바로 유도하려면 아래처럼 더 직접적으로 써도 좋습니다.

```text
지금부터 D:\telegram_trading 프로젝트의 Phase 1 파일들을 실제로 생성/수정하는 작업을 진행해줘.

대상 파일:
- requirements.txt
- .env.example
- config/settings.py
- src/common/logger.py
- src/common/exceptions.py
- src/common/responses.py
- src/infra/db/base.py
- src/infra/db/session.py
- src/infra/db/models.py
- src/web/app.py
- src/web/routes/health.py
- run_web.py

작업 방식:
1. 먼저 필요한 디렉토리 구조를 맞춘다
2. 각 파일에 최소 실행 가능한 코드를 작성한다
3. 파일 간 import가 실제로 맞물리게 수정한다
4. FastAPI 앱이 실행되도록 만든다
5. /health, /ready 응답이 가능하도록 만든다

중요:
- 지금은 텔레그램 수집 기능 구현 금지
- 지금은 한국투자증권 API 구현 금지
- 설정/로깅/DB/예외/헬스체크까지만
- placeholder성 TODO만 남기지 말고 실행 가능한 초안 작성
- 코드 수정 후 마지막에 어떤 파일을 만들었는지 요약
- 마지막에 실행 명령어도 제시
```

## 가장 실전적인 사용 순서

추천은 아래 순서입니다.

1. **Antigravity**에 Phase 1 설계 프롬프트 입력
2. 설계 결과 확인
3. **Cursor Pro**에 지금 드린 구현 프롬프트 입력
4. 생성된 코드 실행
5. 오류가 나면 “정밀 점검 프롬프트”로 바로 수정

이 순서가 가장 안정적입니다.
