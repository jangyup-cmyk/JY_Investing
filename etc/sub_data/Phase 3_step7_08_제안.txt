좋습니다. 이어서 바로 **`retry 후 성공` 통합 테스트 1개**를 붙일 수 있는 형태로 드리겠습니다.

이 테스트의 목적은 단순합니다.

* 첫 번째 inference 는 실패
* job 은 `RETRY_PENDING`
* 두 번째 inference 는 성공
* 최종 결과 row 는 **1건만 존재**
* `attempt_count == 2`

---

# 1) 테스트에 사용할 flaky client

첫 호출만 실패하고, 두 번째 호출부터 성공하는 mock client 입니다.

```python
class FlakySummaryClient:
    def __init__(self):
        self.call_count = 0

    def summarize(self, text: str) -> str:
        self.call_count += 1
        if self.call_count == 1:
            raise RuntimeError("temporary timeout")
        return f"[SUMMARY] {text[:100]}"
```

# 3) `test_pipeline_flow.py` 에 추가할 테스트

```python
def test_pipeline_inference_retry_then_success(
    db_session_factory,
    message_service,
    preprocess_worker,
    flaky_inference_worker,
):
    # 1. 메시지 저장
    persisted = message_service.persist_message(
        channel_id=1,
        telegram_message_id=1003,
        text="LG에너지솔루션 신규 수주 관련 속보",
        sent_at=datetime(2026, 3, 22, 9, 20, 0),
        raw_payload={
            "id": 1003,
            "text": "LG에너지솔루션 신규 수주 관련 속보",
        },
    )

    # 2. preprocess 정상 처리
    processed_pre = preprocess_worker.run_once()
    assert processed_pre == 1

    # 3. 첫 번째 inference 실패
    processed_inf_1 = flaky_inference_worker.run_once()
    assert processed_inf_1 == 0

    with db_session_factory() as session:
        summary_job = get_job(session, persisted.id, "SINGLE_MESSAGE_SUMMARY")
        assert summary_job.status == JobStatus.RETRY_PENDING
        assert summary_job.attempt_count == 1
        assert summary_job.last_error is not None
        assert "temporary timeout" in summary_job.last_error

        result_count = (
            session.query(MessageSummary)
            .filter(MessageSummary.message_id == persisted.id)
            .count()
        )
        assert result_count == 0

    # 4. 두 번째 inference 성공
    processed_inf_2 = flaky_inference_worker.run_once()
    assert processed_inf_2 == 1

    # 5. 최종 상태 검증
    with db_session_factory() as session:
        summary_job = get_job(session, persisted.id, "SINGLE_MESSAGE_SUMMARY")
        assert summary_job.status == JobStatus.DONE
        assert summary_job.attempt_count == 2
        assert summary_job.started_at is not None
        assert summary_job.finished_at is not None
        assert summary_job.started_at <= summary_job.finished_at
        assert summary_job.last_error in (None, "")

        result = (
            session.query(MessageSummary)
            .filter(MessageSummary.message_id == persisted.id)
            .one()
        )
        assert result.message_id == persisted.id
        assert result.summary_text is not None
        assert result.summary_text != ""
        assert "[SUMMARY]" in result.summary_text

        # 결과 row 중복 생성 방지
        result_count = (
            session.query(MessageSummary)
            .filter(MessageSummary.message_id == persisted.id)
            .count()
        )
        assert result_count == 1
```

# 4) 이 테스트가 고정하는 핵심 계약

이 테스트가 들어가면 아래 동작이 강제로 고정됩니다.

* `RETRY_PENDING` job 은 다시 claim 가능해야 함
* 재시도 시 `attempt_count` 가 증가해야 함
* 성공하면 최종 상태가 `DONE` 이어야 함
* 실패 흔적 `last_error` 는 성공 시 정리되는 편이 자연스러움
* summary 결과는 재시도 후에도 **1건만 존재**해야 함

즉, `upsert_summary()` 가 제대로 동작하는지도 같이 검증됩니다.

# 5) 여기서 바로 깨질 수 있는 지점

이 테스트가 실패하면 보통 원인은 아래 셋 중 하나입니다.

첫째, `claim_next_job()` 가 `RETRY_PENDING` 을 다시 가져오지 못하는 경우입니다.

현재 조건이 반드시 아래처럼 되어 있어야 합니다.

```python
AnalysisJob.status.in_([JobStatus.PENDING, JobStatus.RETRY_PENDING])
```

둘째, `fail_job()` 이 실패 시 `RETRY_PENDING` 대신 `FAILED` 로만 남기는 경우입니다.

그럴 경우 테스트 기대값을 바꾸기보다, 지금 구조에서는 `RETRY_PENDING` 으로 가는 쪽이 운영상 더 좋습니다.

셋째, `upsert_summary()` 가 아니라 매번 insert 를 해서 결과 row 가 2건 생기는 경우입니다.

이 경우 summary 저장 로직을 반드시 upsert 로 바꿔야 합니다.
