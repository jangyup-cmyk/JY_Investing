## 이제 바로 들어가야 할 Phase 2 구현 순서

지금부터는 순서가 중요합니다.
추천 순서는 아래와 같습니다.

### 1) 채널 동기화부터 완성

먼저 해야 할 것은 **텔레그램 참여 채널 → DB 반영**입니다.

구현 목표:

* 현재 참여 중인 채널/그룹 목록 조회
* 신규 채널 insert
* 기존 채널 정보 update
* 더 이상 참여하지 않는 채널 상태 갱신
* collection state 기본값 생성

이 단계가 먼저 끝나야 이후 토글과 수집 대상 선정이 가능합니다.

---

### 2) 채널 수집 토글 구현

다음은 **어떤 채널을 실제로 수집할지 제어하는 기능**입니다.

구현 목표:

* API에서 채널 활성/비활성 전환
* `TelegramChannelCollectionState.status` 반영
* 워커가 ACTIVE 채널만 수집 대상으로 인식

여기서 중요한 건 “API가 상태를 바꾸고, 워커가 그 상태를 읽어 반영”하는 구조입니다.
토글 API가 직접 수집 로직까지 건드리면 나중에 꼬이기 쉽습니다.

---

### 3) 백필 함수 구현

그 다음이 **최근 3일 또는 최근 1,000건 백필**입니다.

구현 목표:

* 채널 활성화 시 1회 백필
* 워커 시작 시 누락 보정 백필
* 마지막 체크포인트 기반 이어받기
* 저장은 무조건 upsert

여기서 핵심은 백필과 실시간 저장이 **동일한 저장 함수**를 쓰도록 만드는 것입니다.
적재 경로가 두 개로 갈라지면 중복 처리 기준이 흔들립니다.

---

### 4) 실시간 이벤트 리스너 연결

그 다음에 실시간 수집을 붙이면 됩니다.

구현 목표:

* 새 메시지 이벤트 수신
* 편집 메시지 이벤트 수신
* ACTIVE 채널 여부 검사
* 공통 persistence 함수 호출
* 채널 체크포인트 갱신

이 단계에서 중요한 것은 “수집 성공”보다 **예외가 나도 워커 전체가 죽지 않는 구조**입니다.

---

### 5) 운영 로그 정리

마지막으로 로그를 붙이세요.

최소 로그 항목:

* 워커 시작/종료
* 텔레그램 연결 성공/실패
* 채널 동기화 결과
* 백필 시작/종료
* 메시지 적재 성공/실패 수
* 채널별 에러 메시지
* reconnect 발생 기록

이 로그가 있어야 나중에 “왜 안 들어왔는지”를 찾을 수 있습니다.

---

## 지금부터 특히 조심할 4가지

### 1. `edited_at` 처리와 upsert 충돌

편집 메시지가 들어왔을 때 기존 데이터가 덮어써져야 하지만,
이미 저장된 원문 보존 정책과 충돌하지 않게 해야 합니다.

권장 방식:

* 기본 행은 update
* 필요하면 원문 변경 이력은 별도 로그 테이블로 나중에 확장

초기에는 이력 테이블까지는 없어도 됩니다.

### 2. 삭제 메시지 처리

텔레그램 삭제 이벤트를 어느 수준까지 받을지 미리 정해야 합니다.

초기 권장:

* hard delete 금지
* `is_deleted = true`
* 가능하면 `deleted_at`도 추후 고려

### 3. `reply_to_message_id` 참조 무결성

답장 대상 메시지가 아직 DB에 없는 경우가 있습니다.
그래서 외래키를 너무 강하게 걸면 수집이 꼬일 수 있습니다.

초기에는:

* 숫자만 저장
* 나중에 참조 연결은 후처리
  가 더 안전합니다.

### 4. `raw_payload_json` 크기

원본 보존은 좋지만, 매번 전체 JSON 직렬화가 과도하면 성능 저하가 날 수 있습니다.
그래도 현재 단계에서는 성능보다 보존성이 우선이므로 지금 선택은 맞습니다.
다만 나중에 대량 수집 단계에서는 압축/필드 축소 여부를 다시 점검하면 됩니다.

---

## 현 시점 판단

지금은 더 설계를 늘리기보다, 바로 아래 단계로 가는 것이 맞습니다.

**다음 구현 대상 1순위는**
`채널 동기화 API + 채널 상태 반영 + 공통 메시지 persistence 함수`
입니다.

이 3개가 먼저 되면 그 다음 백필과 실시간 이벤트 연결은 훨씬 깔끔하게 붙습니다.
