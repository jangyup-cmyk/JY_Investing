**STEP 1 구현 명세서**로 정리

이번 단계의 목적은 단순히 API 2개를 만드는 것이 아니라,
**Phase 2 전체가 공통으로 따를 저장 규약과 상태 전이 규약을 먼저 고정하는 것**입니다.

# STEP 1 구현 명세서

## 범위

1. 공통 메시지 Payload 규격 정의
2. 공통 메시지 Upsert 저장 함수 구현
3. 채널 체크포인트 갱신 함수 구현
4. 채널 동기화 API 고도화
5. 채널 수집 토글 API 구현
6. 상태 전이 규칙 및 예외 처리 규칙 확정

---

# 1. STEP 1의 최종 목표

이 단계가 끝나면 아래가 가능해야 합니다.

* Telethon 메시지를 내부 표준 Payload로 변환할 수 있다.
* 같은 메시지를 여러 번 넣어도 DB에는 1건만 유지된다.
* 수정 메시지가 들어오면 기존 레코드가 갱신된다.
* 채널 sync 시 채널/상태 테이블이 정규화된다.
* 채널 toggle 시 워커가 해석 가능한 상태로 바뀐다.
* 메시지 저장 성공 후 체크포인트가 일관되게 갱신된다.

즉, STEP 2 백필과 STEP 3 실시간 리스너는 이 단계 결과물 위에만 올라가면 됩니다.

---

# 2. 핵심 설계 원칙

## 원칙 1. 저장 경로는 무조건 하나

백필과 실시간 수집은 모두 같은 함수로 저장합니다.

* `map_telethon_message_to_payload()`
* `upsert_message()`
* `update_channel_checkpoint()`

적재 경로가 두 갈래로 나뉘면 중복 처리 기준과 수정 반영 기준이 갈라집니다.

## 원칙 2. API와 워커의 책임 분리

* API는 상태만 바꿉니다.
* 워커는 상태를 읽고 실제 수집/백필을 수행합니다.

즉 `/toggle` API는 Telethon을 직접 호출하지 않습니다.

## 원칙 3. 채널 존재 상태와 수집 상태는 구분

한 채널은 “내 계정에서 현재 접근 가능한지”와 “수집 대상으로 켜져 있는지”가 다릅니다.

그래서 내부적으로 최소한 아래 두 축으로 생각해야 합니다.

* 동기화 상태: `DISCOVERED`, `SYNCED`, `MISSING`, `INACCESSIBLE`
* 수집 상태: `PENDING_BACKFILL`, `ACTIVE`, `PAUSED`, `ERROR`

테이블 컬럼이 하나여도, 로직은 이 구분을 유지해야 합니다.

## 원칙 4. 체크포인트는 저장 성공 후에만 갱신

메시지 저장이 실패했는데 `last_collected_message_id`가 올라가면 누락이 생깁니다.

# 4. 데이터 계약 정의

## 4-1. 내부 Payload 객체

Telethon 객체를 DB 저장 함수가 직접 받지 않도록 중간 표준 객체를 둡니다.

```python
from dataclasses import dataclass
from datetime import datetime
from typing import Optional

@dataclass
class TelegramMessagePayload:
    telegram_channel_id: int
    tg_message_id: int
    sender_id: Optional[int]
    message_text_raw: Optional[str]
    reply_to_message_id: Optional[int]
    views: Optional[int]
    forwards: Optional[int]
    grouped_id: Optional[int]
    edited_at: Optional[datetime]
    is_deleted: bool
    message_date: datetime
    collected_at: datetime
    raw_payload_json: str
```

## 4-2. Upsert 결과 객체

저장 함수는 DB 모델 대신 결과 객체를 반환하는 것이 좋습니다.

```python
from dataclasses import dataclass
from typing import Literal, Optional

@dataclass
class UpsertMessageResult:
    action: Literal["created", "updated", "unchanged", "failed"]
    telegram_channel_id: int
    tg_message_id: int
    checkpoint_candidate: Optional[int]
    error_message: Optional[str] = None
```

이렇게 하면 워커가 후속 처리하기 쉬워집니다.

# 5. 상태 정의

## 5-1. 채널 동기화 상태

```python
class ChannelSyncStatus(str, Enum):
    DISCOVERED = "DISCOVERED"
    SYNCED = "SYNCED"
    MISSING = "MISSING"
    INACCESSIBLE = "INACCESSIBLE"
```

## 5-2. 채널 수집 상태

```python
class ChannelCollectionStatus(str, Enum):
    PENDING_BACKFILL = "PENDING_BACKFILL"
    ACTIVE = "ACTIVE"
    PAUSED = "PAUSED"
    ERROR = "ERROR"
```

핵심은 `PENDING_BACKFILL`입니다.
토글 ON 직후 곧바로 ACTIVE로 두지 말고, 워커가 백필을 끝낸 뒤 ACTIVE로 올리는 흐름이 가장 안전합니다.

# 6. 함수 단위 구현 명세

## 6-1. `map_telethon_message_to_payload(message, telegram_channel_id)`

### 목적

Telethon 메시지 객체를 내부 표준 Payload로 변환

### 입력

* `message`: Telethon message object
* `telegram_channel_id`: 내부 채널 식별용 채널 ID

### 출력

* `TelegramMessagePayload`

### 처리 규칙

* 본문 없는 메시지는 `message_text_raw = None`
* `reply_to_message_id`가 없으면 `None`
* `views`, `forwards`, `grouped_id`, `sender_id`도 없으면 `None`
* `raw_payload_json`은 가능한 한 원본 속성을 손실 없이 JSON 직렬화
* `collected_at`은 현재 시각
* `message_date`는 텔레그램 원본 작성 시각
* 삭제 이벤트용 메시지라면 `is_deleted = True`

### 예외 처리

* 직렬화 실패 시 전체 수집 중단 금지
* 직렬화 실패 원인을 로그에 남기고, 가능한 최소 필드만으로 fallback JSON 생성

---

## 6-2. `upsert_message(db, payload)`

### 목적

중복 없는 단일 저장 경로 제공

### 입력

* DB Session
* `TelegramMessagePayload`

### 출력

* `UpsertMessageResult`

### 조회 기준

* `(telegram_channel_id, tg_message_id)` 복합 유니크 키

### 동작 규칙

#### 신규 메시지

* 레코드가 없으면 insert
* 결과: `created`

#### 기존 메시지

* 아래 필드 update 후보

  * `message_text_raw`
  * `reply_to_message_id`
  * `views`
  * `forwards`
  * `grouped_id`
  * `sender_id`
  * `edited_at`
  * `is_deleted`
  * `raw_payload_json`
  * `message_date`
  * `collected_at`

* 결과 판단

  * 값 변경이 있으면 `updated`
  * 변경이 없으면 `unchanged`

### 수정 메시지 규칙

* `edited_at`이 새 값으로 들어오면 갱신
* `edited_at`이 비어 있어도 본문/메타데이터 차이가 있으면 update 가능
* `raw_payload_json`은 최신본으로 덮어쓰기

### 삭제 메시지 규칙

* hard delete 금지
* `is_deleted = True`
* 초기 단계에서는 `deleted_at` 없이 시작 가능

### 실패 규칙

* 예외 발생 시 rollback
* 결과: `failed`
* `checkpoint_candidate = None`

---

## 6-3. `update_channel_checkpoint(db, telegram_channel_id, candidate_message_id, collected_at)`

### 목적

마지막 성공 수집 지점을 안전하게 갱신

### 입력

* DB Session
* 채널 ID
* 후보 메시지 ID
* 수집 시각

### 출력

* 없음 또는 bool

### 규칙

* 상태 테이블 레코드가 없으면 생성
* 현재 `last_collected_message_id`보다 후보가 클 때만 갱신
* `last_collected_at`도 같이 갱신
* 저장 실패 메시지에는 호출하지 않음

### 금지 규칙

* `failed` 결과인데 체크포인트 올리기 금지
* 더 작은 message_id로 되돌리기 금지

---

## 6-4. `sync_channels(account)` 또는 `sync_channels_service.run()`

### 목적

현재 내 계정이 접근 가능한 텔레그램 채널 목록을 DB와 정규화

### 처리 순서

1. Telethon으로 참여 채널/그룹 조회
2. 각 채널에 대해

   * 신규면 `TelegramChannel` insert
   * 기존이면 title, username, type 등 update
3. `TelegramChannelCollectionState`가 없으면 생성
4. 현재 조회된 채널은 `sync_status = SYNCED`
5. 이전에 DB엔 있었지만 이번 조회엔 없으면 `sync_status = MISSING`

### 주의

* sync는 수집 활성화를 의미하지 않음
* sync는 “채널 정보를 최신 상태로 맞추는 작업”임

---

## 6-5. `toggle_channel_collection(channel_id, enabled)`

### 목적

사용자가 수집 ON/OFF를 제어

### 입력

* 내부 채널 ID
* `enabled: bool`

### 출력

* 변경된 상태 정보

### 규칙

* `enabled = True`

  * 현재 `PAUSED` 또는 초기 상태면 `PENDING_BACKFILL`
* `enabled = False`

  * `PAUSED`
* `ERROR` 상태 채널도 다시 ON하면 `PENDING_BACKFILL`로 재진입 가능
* 토글 API는 백필을 직접 실행하지 않음

# 7. API 명세

## 7-1. `POST /api/telegram/channels/sync`

### 목적

참여 채널 목록 동기화

### 요청

body 없음 또는 선택 옵션

### 응답 예시

```json
{
  "success": true,
  "new_channels_count": 12,
  "updated_channels_count": 31,
  "missing_channels_count": 2
}
```

### 서버 내부 처리

* Telethon 연결 확인
* sync service 호출
* 결과 요약 반환

---

## 7-2. `POST /api/telegram/channels/{id}/toggle`

### 요청 예시

```json
{
  "enabled": true
}
```

### 응답 예시

```json
{
  "success": true,
  "channel_id": 17,
  "collection_status": "PENDING_BACKFILL"
}
```

### 상태 전이 규칙

* OFF → ON: `PENDING_BACKFILL`
* ON → OFF: `PAUSED`

# 8. 상태 전이표

## 8-1. 동기화 상태

| 현재         | 이벤트           | 다음           |
| ---------- | ------------- | ------------ |
| 없음         | sync에서 신규 발견  | DISCOVERED   |
| DISCOVERED | 채널 정보 정상 저장   | SYNCED       |
| SYNCED     | 다음 sync에서도 존재 | SYNCED       |
| SYNCED     | 다음 sync에서 미발견 | MISSING      |
| MISSING    | 다시 발견         | SYNCED       |
| any        | 접근 실패/권한 문제   | INACCESSIBLE |

## 8-2. 수집 상태

| 현재               | 이벤트           | 다음               |
| ---------------- | ------------- | ---------------- |
| 초기               | toggle ON     | PENDING_BACKFILL |
| PAUSED           | toggle ON     | PENDING_BACKFILL |
| PENDING_BACKFILL | 백필 성공         | ACTIVE           |
| PENDING_BACKFILL | 백필 실패         | ERROR            |
| ACTIVE           | toggle OFF    | PAUSED           |
| ERROR            | toggle ON 재시도 | PENDING_BACKFILL |
| ACTIVE           | 반복적 연결 실패     | ERROR            |

# 9. 트랜잭션 규칙

## 메시지 저장 단위

메시지 1건당 독립 트랜잭션이 가장 안전합니다.

이유:

* 한 메시지 실패가 전체 워커를 멈추지 않음
* 디버깅이 쉬움
* 체크포인트 누락 방지가 명확함

초기 Phase 2에서는 배치 최적화보다 안정성이 우선입니다.

## 추천 흐름

1. Payload 생성
2. `upsert_message()`
3. 결과가 `created/updated/unchanged`이면
4. `update_channel_checkpoint()`
5. commit
6. 실패 시 rollback + 로그

---

# 10. 로그 규칙

STEP 1부터 아래 로그를 남기는 것이 좋습니다.

## sync 로그

* 신규 채널 수
* 갱신 채널 수
* missing 채널 수
* 실패 사유

## toggle 로그

* 채널 ID
* 이전 상태
* 변경 상태
* 요청 시각

## message upsert 로그

* 채널 ID
* 메시지 ID
* 결과: created / updated / unchanged / failed
* 실패 메시지

## checkpoint 로그

* 채널 ID
* 이전 체크포인트
* 새 체크포인트

# 11. 예외 처리 규칙

## 메시지 직렬화 실패

* 전체 워커 중단 금지
* 최소 필드로 fallback 저장 시도
* 실패 로그 기록

## DB IntegrityError

* rollback
* 동일 메시지 경쟁 저장일 가능성 고려
* 재조회 후 `unchanged` 또는 `updated`로 정규화 가능

## Telethon 개별 메시지 파싱 오류

* 해당 메시지만 스킵 또는 failed 처리
* 워커 계속 진행

## 채널 sync 실패

* 전체 프로세스 실패 반환
* 기존 상태는 유지
* `last_error_message` 기록

# 12. 테스트 체크리스트

STEP 1 구현 후 최소 아래 테스트는 통과해야 합니다.

## 저장 함수 테스트

1. 신규 메시지 insert 성공
2. 동일 메시지 재입력 시 중복 생성 없음
3. 편집 메시지 입력 시 update
4. 삭제 메시지 입력 시 `is_deleted=True`
5. `reply_to_message_id`, `views`, `forwards`, `grouped_id` 저장 확인

## 체크포인트 테스트

6. 저장 성공 후 체크포인트 증가
7. 더 작은 메시지 ID가 들어와도 체크포인트 유지
8. 저장 실패 시 체크포인트 미변경

## sync 테스트

9. 신규 채널 발견 시 channel/state 생성
10. 기존 채널 정보 변경 시 update
11. 미발견 채널은 `MISSING`

## toggle 테스트

12. OFF → ON 시 `PENDING_BACKFILL`
13. ON → OFF 시 `PAUSED`
14. ERROR → ON 재시도 시 `PENDING_BACKFILL`

# 결론

지금 바로 착수할 STEP 1의 본질은 이것입니다.

1. **내부 메시지 표준 객체를 만든다**
2. **백필/실시간 공통 Upsert 저장 함수를 만든다**
3. **채널 상태 전이 규칙을 고정한다**
4. **그 위에 sync/toggle API를 얹는다**

이렇게 가면 STEP 2 백필 엔진과 STEP 3 실시간 리스너는 거의 조립 수준으로 붙습니다.

다음으로 이어질 가장 자연스러운 작업은 **STEP 1 파일별 구현 초안**입니다.
