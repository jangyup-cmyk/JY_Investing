종합해서 보면, 현재 프로젝트는 **Phase 2 수집 인프라까지는 방향이 꽤 좋습니다.** 문서상 구조는 `run_web.py`와 `run_collector.py`를 분리한 2-프로세스 모델이고, FastAPI 쪽은 `Depends(get_db)`로 세션 생명주기를 분리했으며, 수집 쪽은 Telethon 기반 실시간 리스너 + 백필 + 커서 기반 resume + 배치 커밋 + savepoint까지 이미 반영돼 있습니다. 또한 `last_backfill_at`와 `last_collected_at`를 분리해 복구 지점을 명확히 했고, 24개 테스트와 7-tier 검증 기준을 통과한 것으로 정리돼 있어, “수집 파이프라인의 기초 체력”은 이미 확보된 상태로 보입니다.      

다만 지금 문서 기준으로 가장 큰 구조 리스크는 **“워커와 API가 서로 다른 방식으로 세션을 쓰는가” 자체보다, 현재 DB가 `aiosqlite`인 상태에서 앞으로 API·수집 워커·전처리 워커·LLM 워커가 동시에 붙을 때 쓰기 경합이 커진다**는 점입니다. SQLite는 여러 읽기 트랜잭션은 가능하지만 **동시 쓰기는 1개만 허용**하고, WAL 모드는 읽기/쓰기 동시성은 높여주지만 여전히 writer는 1개입니다. 따라서 프로세스를 늘릴수록 병목이 “세션 패턴”보다 “백엔드 DB의 write serialization”에서 먼저 드러날 가능성이 큽니다.
[참조사이트] https://www.sqlite.org/lang_transaction.html?utm_source=chatgpt.com

그래서 1번 질문에 대한 제 판단은 이렇습니다. **현재의 이원화된 세션 관리 방식은 원칙적으로 문제 아닙니다.** API는 request-scoped session, 워커는 job-scoped session을 쓰는 것이 맞습니다. 오히려 잘못된 패턴은 “무한 루프 워커가 장시간 하나의 세션을 쥐고 네트워크 I/O와 DB I/O를 함께 끌고 가는 것”입니다. 문서상 워커는 백필/실시간 적재와 체크포인트를 분리하고 있고, DI도 일부 반영돼 있어 방향은 맞습니다. 다만 다음 단계에서는 `get_db()`와 별개로 **공통 `session_scope()` 컨텍스트 매니저를 하나 만들고**, API는 그것을 FastAPI dependency로 감싸고, 워커는 동일한 `session_scope()`를 직접 쓰게 통일하는 편이 좋습니다. 즉 “세션 생성 API는 하나, 사용 컨텍스트만 둘”로 정리하는 게 가장 깔끔합니다.    

실무적으로는 이렇게 정리하시면 됩니다.
API 라우트는 **요청당 1세션**, 워커는 **배치당 1세션** 또는 **메시지 묶음당 1트랜잭션**으로 가고, **Telethon 호출 전후의 대기 구간에서는 세션을 들고 있지 않게** 하세요. 그리고 SQLite를 당분간 유지한다면 최소한 WAL, busy timeout, 짧은 트랜잭션, 큰 배치의 분할 커밋을 기본값으로 가져가야 합니다. 반대로 **Phase 4 직전까지도 SQLite를 유지할 생각이라면**, 큐 도입보다 먼저 PostgreSQL로 올리는 쪽이 구조적으로 더 큰 효과를 냅니다. 큐를 얹어도 최종 sink가 single-writer SQLite면 병목이 그대로 남기 때문입니다.
[참조사이트] https://sqlite.org/wal.html?utm_source=chatgpt.com


2번 질문인 메시지 큐는, **지금 당장 붙일 단계는 아니지만 LLM 추론이 본격적으로 들어가기 직전에는 거의 필수**라고 봅니다. 현재 구조는 Telethon 이벤트를 받아 곧바로 파싱 후 DB upsert 하는 ingestion-first 구조인데, 문서에 적힌 다음 단계가 Phase 3 전처리, Phase 4 LLM 분석, Phase 5 다중 에이전트 토론입니다. 이 세 단계는 수집보다 훨씬 느리고, 실패 재시도와 우선순위 제어가 필요합니다. 따라서 “수집 완료”와 “분석 시작” 사이를 비동기 작업 단위로 끊어야 합니다.   

다만 **Redis Pub/Sub는 이 프로젝트의 핵심 분석 파이프라인용으로는 비추천**입니다. Redis 공식 문서 기준으로 Pub/Sub는 **at-most-once**라서, subscriber가 순간적으로 실패하면 메시지가 그냥 사라질 수 있습니다. 반면 Redis Streams는 메시지가 저장되고 consumer group, pending, ack 기반으로 운영할 수 있어 훨씬 낫습니다. RabbitMQ는 queue, ack, publisher confirms, prefetch, durability 같은 개념이 잘 정리돼 있어서 **“반드시 처리돼야 하는 분석 작업”**에 더 적합합니다. Celery는 브로커가 아니라 **분산 task queue 프레임워크**라서, Redis나 RabbitMQ 같은 브로커를 뒤에 두고 task routing, retry, scheduling을 얹고 싶을 때 선택하는 도구입니다.
[참조사이트]  https://redis.io/docs/latest/develop/pubsub/?utm_source=chatgpt.com
[참조사이트]  https://www.rabbitmq.com/docs/confirms?utm_source=chatgpt.com
[참조사이트]  https://docs.celeryq.dev/en/stable/?utm_source=chatgpt.com


그래서 추천 순서는 이렇습니다.
**지금~Phase 3:** 큐 없이 진행 가능. 대신 DB에 `analysis_jobs` 또는 outbox 테이블을 두고, 수집이 끝난 메시지를 “분석 대기” 상태로 적재하세요.
**Phase 4 초입:** LLM 호출이 붙는 순간부터는 최소한 **DB job queue → 별도 inference worker** 구조로 분리하세요.
**트래픽이 커지거나 분석 지연이 누적될 때:** Redis Streams 또는 RabbitMQ 도입.
**스케줄링/재시도/작업 라우팅까지 통합하고 싶을 때:** Celery 추가.
즉, 당신 프로젝트에서는 **Redis Pub/Sub는 제외**, **Redis Streams 또는 RabbitMQ**, 그리고 **필요 시 Celery**가 맞는 그림입니다. 특히 자동매매까지 가면 “한 번은 꼭 처리되어야 하는 작업”이 많아지므로 RabbitMQ 쪽이 더 보수적이고 안전합니다.
[참조사이트]  https://redis.io/docs/latest/develop/pubsub/?utm_source=chatgpt.com

3번 질문인 LLM 추론 엔진 대비 전처리는, 지금부터 **“원문 보존 + 정규화 + 피처화 + 추론 결과 분리”**로 설계해두는 게 가장 중요합니다. 문서에는 Phase 3에서 텍스트 정제, 스팸 필터링, 티커 추출을 계획하고 있는데, 이걸 단순 전처리 함수 수준으로 끝내지 말고 **데이터 계약(data contract)** 으로 고정해야 합니다. 즉, LLM에 바로 raw text를 던지는 구조가 아니라, LLM이 항상 같은 스키마의 입력을 받게 만들어야 이후 멀티에이전트 단계가 안정됩니다.    

구체적으로는 최소한 아래 5계층으로 분리해 두는 편이 좋습니다.
`raw_message`: 텔레그램 원문, edit 여부, channel id, tg_message_id, timestamp, reply/forward 관계, media 유무.
`normalized_message`: URL/이모지/불필요 공백/광고문구 제거, 종목명·별칭 표준화, 숫자/퍼센트/가격 표현 정규화.
`message_features`: ticker 후보, 기업명 alias 해소, 섹터 사전 매칭, 키워드, 언어, 길이, 중복해시(simhash/minhash), 신뢰도 priors.
`inference_jobs`: 어떤 모델이 어떤 버전의 프롬프트로 어떤 입력을 분석해야 하는지.
`inference_outputs`: 요약, 이벤트 타입, 테마 점수, 근거 span, confidence, 모델 버전.

이렇게 쪼개 두면 나중에 프롬프트를 바꿔도 raw를 다시 수집할 필요가 없고, 정규화 로직만 개선해도 재추론이 쉬워집니다. 특히 편집 메시지를 반영하는 구조가 이미 있으므로, `edit_version`과 `supersedes_message_id` 같은 계보 정보까지 남겨두면 LLM 결과 캐시 무효화가 매우 쉬워집니다.   

LLM 정확도 측면에서는 **형태소 분석보다 먼저 “금융 도메인 표준화”가 우선**입니다. 한국어 주식 텔레그램 메시지는 형태소보다도 축약어, 초성, 종목 별칭, 테마 은어, 재전송/복붙, 링크 포함 광고가 더 큰 노이즈입니다. 그래서 1차 목표는 문법적 분석이 아니라,
“이 메시지가 무엇을 말하는가”보다 먼저
“이 메시지가 실제로 어느 종목/섹터/이벤트와 연결되는가”를 안정적으로 구조화하는 데 둬야 합니다.
즉 `종목명 alias dictionary`, `테마 사전`, `시장 이벤트 taxonomy`, `채널 신뢰도 prior`, `near-duplicate 제거`가 형태소 분석보다 먼저입니다.

멀티에이전트 대비 전처리에서 특히 중요한 것은 **agent가 raw chat을 통째로 읽지 않게 하는 것**입니다. 각 agent에게는 “공유 증거 묶음(evidence bundle)”만 넘기세요. 예를 들면:

* 대표 메시지 3~5개
* 공통 ticker 집합
* 공통 theme 후보
* 시간대별 언급량 변화
* 채널 신뢰도 가중치
* 원문 인용용 message_id 목록

이렇게 하면 에이전트들이 같은 근거 집합 위에서 서로 다른 관점을 내게 할 수 있고, 토큰 비용도 크게 줄어듭니다. 멀티에이전트의 실패 원인은 대개 모델이 멍청해서가 아니라, 각 agent가 서로 다른 원문 조각을 보고 떠드는 데서 생깁니다.

프레임워크는 **초기 Phase 4에서는 LangChain/AutoGen을 바로 넣지 않는 쪽**을 권합니다. LangChain 측 문서상 현재 agent 구현은 LangGraph primitive 위에 올라가 있고, LangGraph는 더 낮은 수준의 커스터마이징을 전제로 합니다. AutoGen은 다중 에이전트와 동적 워크플로우에 초점이 있지만, 그것 자체가 복잡성을 가져옵니다. 지금 프로젝트는 아직 “수집 → 전처리 → 단일 추론 → 집계 → 토론”의 기본 상태기계가 먼저라서, 초반에는 **직접 만든 명시적 상태머신 + 함수형 파이프라인**이 더 안정적입니다. 관측성, 재시도, 분기 제어, 사람 검토 루프가 복잡해질 때 LangGraph나 AutoGen을 검토하는 게 맞습니다. 
[참조 사이트] https://docs.langchain.com/oss/python/langgraph/agentic-rag?utm_source=chatgpt.com

정리하면, 앞으로의 진행 방향은 우선순위가 분명합니다.

첫째, **세션 설계 자체보다 DB 백엔드 리스크를 먼저 해결**하세요. 공통 `session_scope()`를 만들고, 워커 세션을 배치 단위로 짧게 끊고, SQLite를 유지한다면 WAL/timeout/짧은 트랜잭션을 강제하세요. 가능하면 Phase 4 전에 PostgreSQL로 올리는 것이 좋습니다. 
[참조 사이트] https://sqlite.org/wal.html?utm_source=chatgpt.com

둘째, **큐는 지금 당장보다 “LLM이 붙기 직전”이 도입 시점**입니다. 다만 Pub/Sub가 아니라 durable queue 계열로 가야 하고, 단기적으로는 DB job table, 중기적으로는 Redis Streams 또는 RabbitMQ, 장기적으로는 필요 시 Celery를 얹는 순서가 가장 현실적입니다.
[참조 사이트] https://redis.io/docs/latest/develop/pubsub/?utm_source=chatgpt.com

셋째, **Phase 3를 단순 텍스트 정제가 아니라 ‘LLM-ready 데이터 모델링’ 단계로 재정의**하세요. 원문 보존, 정규화, 피처화, 추론 작업, 추론 결과를 분리해두면 Phase 4와 5가 훨씬 안정적으로 올라갑니다.  

제 결론은 한 문장입니다.
**현재 구조는 Phase 2까지는 합격이고, 다음 병목은 세션 패턴이 아니라 SQLite와 동기식 분석 결합입니다.** 따라서 다음 한 수는 “세션 공통화 → LLM-ready 스키마 설계 → 분석 작업 비동기화” 순서로 가는 것이 가장 안전합니다.
