# 요구사항 반영 방향

당신이 추가한 내용은 핵심적으로 4가지입니다.

1. **서버가 연결되면 `텔레그램 채널 업데이트`는 자동 실행**
2. **업데이트가 시작될 때 `신규 채널만 추가`, `선택 채널 수집 시작`의 마지막 설정을 기억해서 함께 실행**
3. **동시에 수동 실행 기능도 유지**
4. **서버를 껐다가 다시 시작할 때, 이전 설정을 그대로 쓰거나 다시 선택할 수 있어야 함**

즉, 이제 수집 계층은
**“수동 버튼형 화면” + “서버 시작 시 자동 복원형 동작”**
두 가지를 모두 가져야 합니다.

# 1) 기능 해석

## 기존 개념

* 사용자가 버튼을 눌러서

  * 채널 업데이트
  * 신규 채널 추가
  * 수집 시작
    을 각각 실행

## 변경 후 개념

* **서버 연결/기동 시 자동 실행**
* 다만 자동 실행 방식은 **이전 작업 설정을 기억**해야 함
* 사용자는 필요하면 화면에서 **수동으로 다시 실행/변경** 가능
* 서버 재시작 후에는 **이전 설정 복원 또는 새로 선택** 가능

즉, 단순 실행이 아니라 **운영 상태 저장**이 필요합니다.

# 2) 핵심 동작 시나리오

## 시나리오 A. 서버 시작 시 자동 실행

1. 서버가 시작됨
2. 저장된 자동실행 설정 조회
3. `텔레그램 채널 업데이트` 자동 실행
4. 저장된 옵션에 따라 아래 동시 수행

   * `신규 채널만 추가`
   * `마지막으로 선택했던 채널 수집 시작`
5. 로그 기록
6. 웹 화면 상태 갱신

---

## 시나리오 B. 사용자가 수동으로 재설정

1. 사용자가 웹 화면에서 채널 목록 확인
2. 수집 대상 채널을 다시 선택
3. 자동실행 옵션을 바꿈
4. 저장
5. 다음 서버 시작부터 새 설정 적용

---

## 시나리오 C. 서버 재시작 후 선택 복원

1. 서버 중지
2. 다시 실행
3. 기존에 저장된 마지막 선택값 로드
4. 사용자에게 두 가지 상태 제공

   * 자동 복원 상태로 실행
   * 사용자가 다시 선택하여 재설정 가능

# 3) 기능 요구사항으로 문서화

아래 문구를 기능명세서에 바로 넣으면 됩니다.

## 자동 실행 요구사항

* 서버가 정상 연결 및 시작되면 `텔레그램 채널 업데이트` 기능은 자동 실행되어야 한다.
* 자동 실행 시 시스템은 저장된 마지막 작업 설정을 조회하여 함께 반영해야 한다.
* 자동 실행 시 아래 기능이 옵션에 따라 함께 수행될 수 있어야 한다.

  * `신규 채널만 추가`
  * `선택 채널 수집 시작`
* 자동 실행 결과는 시스템 로그 및 화면 로그 패널에 기록되어야 한다.

## 수동 실행 요구사항

* 사용자는 웹 화면에서 `텔레그램 채널 업데이트`, `신규 채널만 추가`, `선택 채널 수집 시작`, `선택 채널 수집 중지`를 수동으로 실행할 수 있어야 한다.
* 수동 실행은 자동 실행 설정과 별개로 즉시 반영되어야 한다.

## 마지막 작업 기억 요구사항

* 시스템은 마지막으로 사용자가 설정한 아래 항목을 저장해야 한다.

  * 자동 업데이트 실행 여부
  * 신규 채널만 추가 실행 여부
  * 자동 수집 시작 대상 채널 목록
  * 자동 복원 사용 여부
* 저장된 설정은 서버 재시작 후에도 유지되어야 한다.

## 서버 재시작 후 재설정 요구사항

* 서버 재시작 시 시스템은 마지막 저장 설정을 기준으로 자동 복원할 수 있어야 한다.
* 사용자는 웹 화면에서 기존 설정을 해제하고 다시 선택하여 저장할 수 있어야 한다.

# 5) 채널 선택 기억 로직

`선택 채널 수집 시작`을 자동화하려면 단순히 버튼만 기억하면 안 되고,
**어떤 채널들을 선택했는지**도 저장해야 합니다.

즉, 시스템이 기억해야 하는 것은 두 종류입니다.

## A. 동작 옵션 기억

* 자동 업데이트 여부
* 신규 채널만 추가 여부
* 자동 수집 시작 여부
* 재시작 복원 여부

## B. 대상 기억

* 마지막으로 사용자가 선택한 채널 ID 목록

그래서 다음 서버 시작 시:

* 자동 업데이트 실행
* 신규 채널만 추가 실행
* 마지막 선택 채널들 중 여전히 유효한 채널만 자동 수집 시작

이 흐름으로 가야 합니다.

# 6) 예외 상황 처리

이 기능은 자동화가 들어가므로 예외 처리가 중요합니다.

## 경우 1. 이전에 선택한 채널이 사라진 경우

* 해당 채널은 자동 수집 대상에서 제외
* 로그 기록:

  * `이전 자동 수집 대상 2개 채널은 현재 조회되지 않아 제외되었습니다.`

## 경우 2. 채널은 존재하지만 접근 권한이 없는 경우

* 상태를 `ERROR` 또는 `REMOVED`로 반영
* 자동 수집 시작 제외

## 경우 3. 사용자가 자동 복원을 원하지 않는 경우

* 서버 시작 후 채널 업데이트만 자동 실행
* 수집 대상은 비활성 상태 유지

## 경우 4. 수동 재설정 후 저장

* 이전 자동 설정 덮어쓰기

* `auto_add_new_channels_only`
  업데이트 실행 시 신규 채널만 추가할지 여부

* `auto_start_last_selected_channels`
  마지막 선택 채널 자동 수집 시작 여부

* `auto_restore_last_selection`
  서버 재시작 시 마지막 설정 복원 여부

* `last_selected_channel_ids_json`
  마지막으로 선택한 채널 목록 저장

# 9) 서버 시작 시 내부 처리 로직

이제 백엔드 시작 시 startup 로직은 단순 앱 시작이 아니라 아래 순서가 됩니다.

## startup 처리 흐름

1. 서버 시작
2. DB 연결 확인
3. 텔레그램 런타임 설정 조회
4. `auto_channel_update = true` 이면 채널 업데이트 실행
5. `auto_add_new_channels_only = true` 이면 신규 채널만 추가 적용
6. `auto_start_last_selected_channels = true` 이면 마지막 선택 채널 자동 수집 시작
7. 결과 로그 저장
8. 웹 화면에서 현재 자동실행 상태 표시

# 11) 수동과 자동의 우선순위

이건 미리 정하는 게 좋습니다.

## 권장 우선순위

* **수동 설정 저장이 자동 설정보다 우선**
* 사용자가 수동으로 바꾸고 저장하면, 그 값이 다음 서버 시작 시 기준값이 됨

즉:

* 자동은 “이전 저장값 재사용”
* 수동은 “새 기준값 설정”

이 구조가 가장 자연스럽습니다.

> 이전에 저장된 설정에 따라 신규 채널 추가와 마지막 선택 채널 수집 시작이 함께 수행되며,
> 사용자는 웹 화면에서 이를 수동으로 재설정하고 다음 시작 시 다시 반영되도록 해야 한다.

이 요구는 매우 좋습니다.
왜냐하면 실제 운영에서는 매번 버튼을 누르는 것보다 **자동 복원 + 수동 재설정** 구조가 훨씬 실용적이기 때문입니다.

# 3. 운영 개념

수집 계층은 이제 아래 두 방식으로 동작합니다.

## 수동 실행

* 텔레그램 채널 업데이트
* 신규 채널만 추가
* 선택 채널 수집 시작
* 선택 채널 수집 중지

## 자동 실행

서버 시작 시 저장된 설정을 기준으로 자동 실행

* 채널 업데이트 자동 실행
* 신규 채널만 추가 자동 실행
* 마지막 선택 채널 자동 수집 시작
* 마지막 설정 자동 복원

* 자동복원 ON / OFF

---

## 4.4 자동 실행 설정 영역 기능명세

## 4.4.1 서버 시작 시 텔레그램 채널 업데이트 자동 실행

### 설명

서버가 시작되고 DB 및 텔레그램 세션 상태가 정상일 경우 자동으로 전체 채널 동기화를 수행한다.

### 저장값

`auto_channel_update`

---

## 4.4.2 업데이트 시 신규 채널만 자동 추가

### 설명

자동 실행 시 전체 상태 재계산 대신 신규 채널만 추가하도록 제한할 수 있다.

### 저장값

`auto_add_new_channels_only`

---

## 4.4.3 업데이트 후 마지막 선택 채널 자동 수집 시작

### 설명

자동 동기화 완료 후 이전에 저장된 선택 채널들 중 유효한 채널을 자동으로 수집 활성화한다.

### 저장값

`auto_start_last_selected_channels`

---

## 4.4.4 재시작 시 마지막 설정 자동 복원

### 설명

서버 재시작 시 마지막 자동 설정 및 마지막 선택 채널을 복원한다.

### 저장값

`auto_restore_last_selection`

---

## 4.4.5 적용 버튼

### 설명

현재 자동 실행 설정과 마지막 선택 채널 정보를 저장한다.

### 처리

* 런타임 설정 저장
* 마지막 선택 채널 목록 저장
* 로그 기록
* 화면 갱신

---

## 4.4.6 초기화 버튼

### 설명

자동 실행 설정과 마지막 선택 채널 저장값을 초기화한다.

### 처리

* 자동실행 옵션 false 처리
* 마지막 선택 채널 목록 삭제
* 로그 기록

---

## 4.4.7 마지막 선택 다시 불러오기

### 설명

저장된 마지막 선택 채널 목록을 현재 테이블에 체크 상태로 복원한다.

---

## 4.5 수동 제어 버튼 기능명세

## 4.5.1 텔레그램 채널 업데이트

### 목적

내 계정의 전체 채널/그룹 목록을 조회하여 DB에 반영

### 입력값

없음

### 처리 흐름

1. 텔레그램 세션 상태 확인
2. 현재 채널/그룹 목록 조회
3. 기존 DB 채널 비교
4. 신규 저장, 변경 반영, 누락 채널 REMOVED 처리
5. 동기화 로그 저장
6. 시스템 로그 저장
7. 화면 갱신

### 성공 메시지 예시

`5개의 신규채널을 추가하였습니다. 현재 정보를 수집하고 있는 채널은 20개입니다.`

---

## 4.5.2 신규 채널만 추가

### 목적

현재 목록 중 DB에 없는 채널만 추가

### 입력값

없음

### 성공 메시지 예시

`신규 채널 3개를 추가했습니다.`

---

## 4.5.3 선택 채널 수집 시작

### 목적

선택한 채널들의 수집을 활성화하고, 해당 선택을 마지막 선택값으로 저장 가능하게 한다.

### 입력값

* 채널 ID 배열
* 선택 저장 여부

### 성공 메시지 예시

`선택한 4개 채널의 수집을 활성화했습니다.`

---

## 4.5.4 선택 채널 수집 중지

### 목적

선택한 채널들의 수집을 비활성화

### 입력값

* 채널 ID 배열

### 성공 메시지 예시

`선택한 2개 채널의 수집을 중지했습니다.`

---

## 4.5.5 새로고침

### 목적

요약 카드, 설정, 채널 목록, 로그를 다시 조회

---

## 4.6 자동 실행 동작 명세

## 4.6.1 서버 시작 시 자동 실행 순서

1. 서버 시작
2. DB 연결 확인
3. 텔레그램 런타임 설정 조회
4. `auto_restore_last_selection = true` 인 경우 마지막 설정 로드
5. `auto_channel_update = true` 인 경우 채널 업데이트 실행
6. `auto_add_new_channels_only = true` 인 경우 신규 채널만 추가 방식 적용
7. `auto_start_last_selected_channels = true` 인 경우 마지막 선택 채널 자동 수집 시작
8. 결과 로그 저장
9. 화면 상태 갱신 가능 상태로 유지

---

## 4.6.2 자동 실행 시 예외 처리

### 세션 만료

* 자동 실행 중단
* ERROR 로그 기록
* 화면 상태에 세션 만료 표시

### 이전 선택 채널 일부 누락

* 존재하지 않는 채널 제외
* WARNING 로그 기록

### 채널 접근 권한 없음

* 해당 채널 제외
* 상태를 ERROR 또는 REMOVED 처리

---

* 수집 시작/중지 로그
* 설정 저장/초기화 로그
* 세션/접속 오류 로그

### 구분값

* STARTUP

# 8. 처리 흐름 확정

## 8.1 수동 흐름

```text
1. 사용자가 채널 업데이트 또는 신규 채널 추가 실행
2. 채널 목록 갱신

4. 선택 채널 수집 시작
```

```text
1. 서버 시작
2. DB 연결 확인
3. telegram_runtime_settings 조회
4. auto_restore_last_selection = true 이면 마지막 선택값 로드
5. auto_channel_update = true 이면 자동 채널 업데이트 실행
6. auto_add_new_channels_only = true 이면 신규 채널만 추가 방식 적용
7. auto_start_last_selected_channels = true 이면 마지막 선택 채널 자동 수집 시작
8. 결과 로그 저장
9. telegram_messages 원문 저장 시작
```

# 10. 핵심 정리

이번 v2.0에서 달라진 핵심은 아래입니다.

* 수동 실행 중심에서 **자동 복원형 운영 구조**로 확장
* 서버 시작 시 `텔레그램 채널 업데이트` 자동 실행 가능
* `신규 채널만 추가`, `마지막 선택 채널 자동 수집 시작`을 설정 기반으로 함께 실행
* 마지막 선택값을 저장하고 재시작 시 복원 가능
* 수동 기능은 그대로 유지하며, 사용자가 언제든 재설정 가능

즉, 이제 수집 계층은 단순 버튼형이 아니라
**운영 서버형 수집 관리 시스템**으로 정의된 상태입니다.


---------------------------------------------------------------------------------------------------------
[사용자 요청사항_10]
채널개수가 150개 정도 되고 채널마다 하루 수집되는 메세지는 최소 10개, 최대 5,000개임
---------------------------------------------------------------------------------------------------------


중요한 정보입니다.
이 수치가 들어오면 설계 기준이 꽤 달라집니다.

## 1) 먼저 규모를 해석하면

채널 150개, 채널당 하루 메시지 수가 최소 10개~최대 5,000개라면:

* **최소 총량**: 1,500건/일
* **최대 총량**: 750,000건/일

최대치 기준으로 보면 하루 75만 건입니다.
이건 초당 평균 약 **8.7건/초** 수준인데, 텔레그램은 특정 시간대에 몰리기 때문에 실제 피크는 이보다 훨씬 높게 튈 수 있습니다.

즉, 이 프로젝트는 이제:

* 단순 CRUD 웹앱
* 버튼 눌러 가끔 가져오는 수집기

가 아니라,

* **상시 동작 수집 프로세스**
* **배치/비동기 처리**
* **저장소 확장성 고려**
* **정제 계층 분리**

가 필요한 구조입니다.

---

## 2) 기존 설계에서 가장 크게 바뀌는 판단

이제 가장 중요한 판단은 이것입니다.

> **웹 서버와 수집 워커를 분리해야 합니다.**

예전에는 “서버 시작 시 자동으로 채널 업데이트”를 웹앱 startup에 얹는 것도 가능했지만,
지금 규모에서는 **웹 서버가 직접 수집 루프까지 같이 돌면 위험**합니다.

이유는 간단합니다.

### 문제점

* 웹 서버 재시작 때마다 수집기가 중복 실행될 수 있음
* FastAPI `reload=True` 개발 모드에서 두 번 실행될 수 있음
* 웹 요청 처리와 메시지 수집이 서로 영향을 줌
* 75만 건/일 급에서 DB 세션, 로그, 응답 지연이 꼬일 수 있음

그래서 구조를 이렇게 바꾸는 게 맞습니다.

### 권장 구조

* **WEB/API 서버**

  * 화면
  * 설정 저장
  * 상태 조회
  * 수동 명령 실행
* **Telegram Collector Worker**

  * 채널 업데이트
  * 메시지 수집
  * 원문 저장
* **Preprocessing Worker**

  * 정규화
  * 중복 제거
  * 종목/키워드/이벤트 추출

즉, 자동실행은 유지하되
**“웹앱이 직접 수집을 시작”하는 게 아니라 “수집 워커에게 시작 명령을 내리는 구조”** 로 가야 합니다.

---

## 3) 당신이 앞서 정한 자동실행 요구사항은 유지 가능함

다만 의미를 조금 바꿔야 합니다.

기존 의미:

* 서버 시작 → 자동 채널 업데이트 → 자동 수집 시작

수정 의미:

* **운영 서버(또는 워커 관리자) 시작** → 자동 채널 업데이트 작업 실행 → 저장된 설정대로 수집 워커가 채널 수집 시작

즉, 기능은 유지되지만 실행 주체를 바꾸는 것입니다.

### 바꿔야 하는 표현

기존:

* `서버 시작 시 텔레그램 채널 업데이트 자동 실행`

권장 수정:

* `수집 서비스 시작 시 텔레그램 채널 업데이트 자동 실행`

또는 화면에는 사용자 친화적으로 그대로 두고, 내부 문서에서는:

* **WEB 서버 startup**
* **Collector worker startup**

를 분리해서 적는 것이 좋습니다.

---

## 4) 이 규모에서 A. 수집 계층 설계에 추가해야 할 것

## A-1. 채널 업데이트와 메시지 수집을 분리

`텔레그램 채널 업데이트`는 상대적으로 가벼운 관리 작업입니다.
반면 메시지 수집은 상시 동작입니다.

그래서 분리해야 합니다.

### 채널 업데이트

* 내 계정 채널/그룹 목록 갱신
* 신규 채널 식별
* 채널 상태 동기화

### 메시지 수집

* `is_collecting = true` 채널만 계속 수신
* 원문 메시지 저장
* 통계 반영
* 중복 방지

이 둘을 한 함수/한 작업으로 섞으면 안 됩니다.

---

## A-2. DB 설계는 “쓰기 많은 시스템” 기준으로 가야 함

하루 최대 75만 건이면, `telegram_messages` 테이블은 금방 커집니다.

그래서 지금부터 아래를 고려해야 합니다.

### 꼭 필요한 것

* `UNIQUE (telegram_channel_id, tg_message_id)`
* `posted_at`, `collected_at`, `processing_status` 인덱스
* 채널별 최근 수집 시각 관리
* 일별 통계 테이블 분리

### 권장 추가

* **월 단위 파티셔닝 고려**
* 오래된 원문 메시지 아카이브 정책
* 정제 완료 후 장기 보관 정책 분리

**테이블이 커질 것을 전제로 설계**해야 합니다.

  * 별도 collector metric

으로 처리

## 6) 자동실행 구조도 조금 더 안전하게 바꿔야 함

이전에는 이렇게 생각했죠.

* 서버 연결되면 자동 실행
* 마지막 선택 기억
* 신규 채널만 추가 동시 실행

이건 유지 가능합니다.
다만 이 규모에서는 **자동실행 정책 저장 + 워커 1개만 실행 보장**이 추가되어야 합니다.

### 꼭 필요한 추가 규칙

* 자동실행은 **오직 1개의 collector worker만 수행**
* 웹 서버 여러 인스턴스가 떠도 수집은 한 곳만 실행
* startup 자동실행 전에 **실행 중 락(lock)** 확인

권장 방식:

* `runtime lock`
* `collector heartbeat`
* `last_started_at`, `last_heartbeat_at`

이런 개념을 둬야 중복 수집을 막을 수 있습니다.

## 1) 다시 계산해보면

### 1. WEB/API 서버

역할:

* 채널 관리 화면
* 자동실행 설정 저장
* 수동 명령 실행
* 로그/상태 조회

### 2. Telegram Collector Worker

역할:

* 텔레그램 채널 업데이트
* 마지막 선택 채널 복원
* 메시지 수집
* 원문 저장
* 통계 반영

### 3. PostgreSQL

역할:

* 채널 정보 저장
* 수집 메시지 저장
* 런타임 설정 저장
* 마지막 선택값 저장
* 로그/상태 저장

즉, **웹과 수집은 분리**, 그러나 **복잡한 분산 시스템까지는 가지 않는 구조**가 맞습니다.

---

## 2) 왜 이 구조가 맞는가

현재 규모에서는 Kafka, Redis Queue, 다중 워커 클러스터까지는 과합니다.
하지만 웹 서버 안에서 텔레그램 수집까지 같이 돌리는 것도 좋지 않습니다.

그래서 가장 균형 잡힌 구조가:

* **웹은 관리**
* **워커는 수집**
* **DB는 저장**

입니다.

이렇게 하면:

* 웹 재시작이 수집 중단으로 바로 이어지지 않고
* 수집 오류가 나도 웹 화면은 살아 있고
* 자동실행/재시작 복원도 안정적으로 관리할 수 있습니다

---

## 3) 전체 흐름

```text
[사용자]
   ↓
[WEB/API 서버]
   ├─ 자동실행 설정 저장
   ├─ 채널 목록 조회
   ├─ 수동 실행 명령
   └─ 상태/로그 조회
   ↓
[PostgreSQL]
   ↑
[Telegram Collector Worker]
   ├─ 채널 업데이트
   ├─ 마지막 선택 복원
   ├─ 수집 대상 채널 메시지 수집
   ├─ telegram_messages 저장
   └─ 일별 통계/상태 반영
```

## 4) 수집 계층을 3개 하위 영역으로 나누기

A. 수집 계층을 이제 아래처럼 나누는 것이 좋습니다.

## A-1. 채널 관리

* 내 계정 채널/그룹 목록 동기화
* 신규 채널 추가
* 채널 상태 변경
* 수집 대상 ON/OFF

## A-2. 메시지 수집

* 선택 채널 메시지 실시간/주기 수집
* 원문 저장
* 메시지 중복 방지
* 채널별 최근 수집 상태 기록

## A-3. 운영 상태 관리

* 자동실행 설정
* 마지막 선택 채널 복원
* 워커 상태 표시
* 에러/로그 관리

이렇게 나누면 Phase 2 구현도 훨씬 쉬워집니다.

## 5) 자동실행의 정확한 의미

이제 자동실행은 **웹 서버 startup**이 아니라
**Collector Worker startup** 기준으로 잡는 것이 맞습니다.

### 자동실행 흐름

1. Collector Worker 시작
2. DB 연결 확인
3. 자동실행 설정 조회
4. 채널 업데이트 실행
5. 옵션에 따라 신규 채널만 추가 적용
6. 마지막 선택 채널 복원
7. 자동 수집 시작
8. 상태/로그 저장

사용자 화면에는 편하게
**“서버 시작 시 자동 실행”**
이라고 보여줘도 되지만, 내부 설계는 **수집 워커 시작 시**로 명확히 하는 것이 좋습니다.

## 8) 메시지 저장 전략

현재 규모에서는 `telegram_messages`를 PostgreSQL에 저장해도 충분합니다.
다만 아래 원칙은 반드시 지키는 게 좋습니다.

### 원칙

* 유니크 키로 중복 방지

  * `(telegram_channel_id, tg_message_id)`
* 배치 저장 고려
* 메시지 한 건마다 시스템 로그 남기지 않기
* `posted_at`, `collected_at`, `processing_status` 인덱스 유지

즉,
**메시지는 메시지 테이블에 저장하고, 운영 로그는 운영 로그대로 분리**해야 합니다.

# 5) 왜 `services` 계층을 지금부터 두는가

이건 중요합니다.

초기 프로젝트에서 흔히 하는 실수가:

* route에서 DB 직접 만짐
* worker에서 DB 직접 만짐
* telegram client에서 저장까지 다 처리

하는 방식입니다.

이렇게 되면 곧 꼬입니다.

그래서 지금부터 아래처럼 분리하는 것이 좋습니다.

## route

HTTP 요청/응답만 담당

## service

실제 업무 로직 담당

## infra

외부 연동 담당

* DB
* Telegram
* 나중에 Broker API

이 구조로 가야
웹/API와 Worker가 **같은 서비스 로직을 공유**할 수 있습니다.

# 7) Phase 1 완료 기준도 워커 분리형으로 바뀜

이제 완료 기준은 아래처럼 잡으면 됩니다.

## 웹 서버 기준

* `run_web.py` 실행 가능
* `/health`, `/ready` 정상 응답
* DB 연결 확인 가능

## 워커 기준

* `run_collector.py` 실행 가능
* DB 연결 확인 가능
* 런타임 설정 로드 가능
* 아직 텔레그램 실제 수집은 안 붙어도 됨

# 10) 권장 기본안

최종적으로 지금 프로젝트의 Phase 1 기본안은 아래입니다.

* 웹 서버와 수집 워커는 분리
* PostgreSQL 공용 사용
