좋습니다. 기존 테이블이 이미 있는 상황을 기준으로, **가장 안전한 1차 보정형 migration 1개 + 데이터 정리용 SQL**을 바로 적용 가능한 형태로 정리하겠습니다.

핵심 원칙은 이렇습니다.

* **1차 migration**: 컬럼 보강, 결과 테이블 생성, 기본 인덱스 추가
* **정리 SQL**: null/default 보정, 중복 점검/정리
* **그 다음** unique 제약 강화

즉, **중복 정리 전에 unique constraint를 서두르지 않는 방식**입니다.

## 2) 이 migration의 의도

이 1차 migration은 **안전하게 스키마를 수용 가능한 상태로 맞추는 것**이 목적입니다.

아직 일부를 deliberately 미뤘습니다.

* `messages(channel_id, telegram_message_id)` unique
* `analysis_jobs(message_id, job_type)` unique
* `preprocessed_messages(message_id)` unique
* `message_summaries(message_id)` unique

이 제약은 **중복 데이터 정리 후** 넣는 게 안전합니다.

## 6) 중복 데이터 정리 SQL

중복 제거 전에는 반드시 백업이 있어야 합니다.

### 6-1. messages 중복 제거

기준:

* 같은 `(channel_id, telegram_message_id)` 에서
* `created_at` 최신 row 우선
* 같으면 `id` 큰 row 우선

```sql
WITH ranked AS (
    SELECT
        id,
        ROW_NUMBER() OVER (
            PARTITION BY channel_id, telegram_message_id
            ORDER BY created_at DESC, id DESC
        ) AS rn
    FROM messages
)
DELETE FROM messages
WHERE id IN (
    SELECT id
    FROM ranked
    WHERE rn > 1
);
```

---

### 6-2. analysis_jobs 중복 제거

기준:

* 같은 `(message_id, job_type)` 에서
* `DONE` 우선
* 그다음 `RUNNING`
* 그다음 최신 `created_at`
* 같으면 `id` 큰 row 우선

```sql
WITH ranked AS (
    SELECT
        id,
        ROW_NUMBER() OVER (
            PARTITION BY message_id, job_type
            ORDER BY
                CASE status
                    WHEN 'DONE' THEN 0
                    WHEN 'RUNNING' THEN 1
                    WHEN 'RETRY_PENDING' THEN 2
                    WHEN 'PENDING' THEN 3
                    WHEN 'FAILED' THEN 4
                    WHEN 'DEAD' THEN 5
                    ELSE 9
                END,
                created_at DESC,
                id DESC
        ) AS rn
    FROM analysis_jobs
)
DELETE FROM analysis_jobs
WHERE id IN (
    SELECT id
    FROM ranked
    WHERE rn > 1
);
```

## 8) 정리 후 넣을 unique 제약 강화 migration 초안

이건 2차 migration 입니다. 여기서부터 제약을 강하게 겁니다.

파일 예시:

```text
alembic/versions/20260322_03_add_pipeline_unique_constraints.py
```

```python
"""add unique constraints for pipeline tables

Revision ID: 20260322_03
Revises: 20260322_02
Create Date: 2026-03-22 15:30:00
"""

import sqlalchemy as sa


revision = "20260322_03"
down_revision = "20260322_02"
branch_labels = None
depends_on = None


def upgrade():
    with op.batch_alter_table("messages") as batch_op:
        batch_op.create_unique_constraint(
            "uq_messages_channel_msg",
            ["channel_id", "telegram_message_id"],
        )

    with op.batch_alter_table("analysis_jobs") as batch_op:
        batch_op.create_unique_constraint(
            "uq_analysis_jobs_message_job_type",
            ["message_id", "job_type"],
        )

    with op.batch_alter_table("preprocessed_messages") as batch_op:
        batch_op.create_unique_constraint(
            "uq_preprocessed_messages_message_id",
            ["message_id"],
        )

    with op.batch_alter_table("message_summaries") as batch_op:
        batch_op.create_unique_constraint(
            "uq_message_summaries_message_id",
            ["message_id"],
        )


def downgrade():
    with op.batch_alter_table("message_summaries") as batch_op:
        batch_op.drop_constraint("uq_message_summaries_message_id", type_="unique")

    with op.batch_alter_table("preprocessed_messages") as batch_op:
        batch_op.drop_constraint("uq_preprocessed_messages_message_id", type_="unique")

    with op.batch_alter_table("analysis_jobs") as batch_op:
        batch_op.drop_constraint("uq_analysis_jobs_message_job_type", type_="unique")

    with op.batch_alter_table("messages") as batch_op:
        batch_op.drop_constraint("uq_messages_channel_msg", type_="unique")
```

## 10) 지금 바로 가장 먼저 볼 포인트

이 단계에서 실제로 가장 먼저 확인해야 하는 건 딱 세 가지입니다.

첫째, `analysis_jobs` 에 새 컬럼들이 잘 들어갔는가

* `attempt_count`
* `max_attempts`
* `last_error`
* `started_at`
* `finished_at`
* `heartbeat_at`

둘째, 중복 데이터가 실제로 있는가
특히 `analysis_jobs(message_id, job_type)` 중복이 많이 나오면, 그동안 job 생성이 idempotent 하지 않았다는 뜻입니다.

셋째, unique 제약을 걸기 전에 중복이 완전히 정리됐는가
이걸 확인하지 않고 2차 migration 을 적용하면 바로 실패합니다.

## 11) 한 줄 요약

가장 실전적인 보정 절차는
**1차 migration으로 스키마를 수용 가능한 상태로 만든 뒤, 데이터 정리 SQL로 중복과 null을 정리하고, 마지막 2차 migration으로 unique 제약을 잠그는 것**입니다.

다음 단계는 현재 실제 모델 코드와 이 초안을 1:1로 대조해서, **부족한 필드만 체크리스트로 표시하는 갭 분석표**를 만드는 순서가 가장 자연스럽습니다.
